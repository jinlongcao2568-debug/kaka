## [D2-R-001] 0. 文档任务与裁决范围

本文档不是一般性的“数据结构草表”，也不是前端、接口或规则实现时随手参考的字段备忘录。  
本文档的正式任务是：

1. 冻结  正式对象的**存在性**，即哪些对象属于正式对象，哪些对象不是正式对象；
2. 冻结每个正式对象的**阶段归属**、**上游依赖**、**下游消费方**与**生命周期位置**；
3. 冻结每个正式对象的**最小必有字段**、**字段命名规范**、**状态字段归属**与**引用关系**；
4. 冻结正式对象之间的**消费优先级**，防止页面、接口、导出、销售、交付或脚本生成第二套主判断；
5. 冻结正式对象与公开能力边界之间的对应关系，确保 D13 中的 A/B/C/D 能力都必须落在正式对象体系上；
6. 冻结对象的机器承接要求，确保 `contracts/schemas/*`、`contracts/enums/*` 与本文保持单一口径；
7. 冻结对象进入测试、接口、页面与交付前必须满足的最低结构要求。

本文档必须回答以下问题：

- 哪些对象属于阶段 1-9 的正式对象；
- 哪些对象是统一事实中枢、商业对象、触达对象、交付对象或治理隔离对象；
- 每个对象最少必须带哪些字段，缺什么就不得进入正式主链；
- 哪些对象只允许内部消费，哪些对象可以成为客户版 / 对外 API / 外发包的上游；
- 哪些对象能够被升级为顶层主判断，哪些对象只能作为解释、追溯、补证或治理输入；
- 哪些对象绝对不得被页面、接口、导出、销售或脚本绕开。

## [D2-R-002] 1. 生效规则与优先级

### [D2-R-003] 1.1 生效规则

- 本文是 L0 在“正式对象与字段字典”维度的正式展开文档。
- 本文只允许细化 L0 已冻结的正式对象体系，不允许新增会改变主路线、阶段顺序、公开边界、结果语义或统一事实中枢的新裁决。
- 本文与 L0 冲突时，以 L0 为准。
- 本文与 D13 冲突时，先以 L0 裁决冲突，再以 D13 的能力边界解释对象是否允许进入正式公开主链。
- 页面、接口、脚本、导出、销售话术、临时台账、CSV、Excel、缓存 JSON、测试假对象与开发备注，都不得反向定义正式对象。

### [D2-R-004] 1.2 文档优先级

| 优先级 | 文档或资产 | 裁决范围 |
|---|---|---|
| P1 | L0 权威总文档 | 正式阶段、正式对象存在性、公开边界、统一事实中枢、消费优先级、对外红线 |
| P2 | 本文 D2 | 对象定义、字段规范、阶段归属、上游依赖、下游消费、命名规则 |
| P3 | D3 / D4 / D5 / D6 / D7 / D11 / D13 | 规则、接口、页面、字段策略、交付矩阵、测试、能力边界的对象消费细化 |
| P4 | `contracts/schemas/*` 与 `contracts/enums/*` | 机器可读承接 |
| P5 | 局部设计、临时脚本、样例数据 | 不得突破上位口径 |

### [D2-R-005] 1.3 与 D13 的对齐要求

D13 明确要求，任何正式能力都必须同时说明阶段归属、对象归属、边界归属、结果归属与门禁归属。  
因此，D13 中每一项正式能力都必须落在本文定义的正式对象上，不允许出现“能力有定义、对象无落点”的情况。

本文对 D13 的约束性承接如下：

- A 层能力必须至少映射到一个正式主链对象与一个正式消费对象；
- B 层能力必须至少映射到一个正式核验对象，并显式受治理字段约束；
- C 层能力必须至少映射到 `CLUE` / `OBSERVATION` / `review_request` 的正式对象，不得悬空存在；
- D 层能力不得进入正式公开主链对象；若确需留痕，只能落在治理隔离对象或补证对象上。

## [D2-R-006] 2. 正式对象体系总原则

### [D2-R-007] 2.1 单一正式对象体系原则

 只允许存在一套正式对象体系。  
正式对象必须能够回答：

- 它属于哪一阶段；
- 它从哪些上游正式对象来；
- 它被哪些正式对象消费；
- 它能不能进入正式客户版、正式导出、正式 API 或正式销售对象；
- 它能不能成为顶层主判断的依据。

任何不能回答上述问题的结构，都不得进入正式对象体系。

### [D2-R-008] 2.2 单向消费原则

顶层消费优先级固定为：

`project_fact / legal_action_recommendation / report_record`
→ `rule_gate_decision / evidence_gate_decision / review_request`
→ `field_lineage_record / notice_version_chain / clock_chain_profile`
→ `project_base / public_chain / 核验画像对象`
→ `private_supplement_record / controlled_exception_record / 备注类信息`

因此：

- 页面不得绕开 `project_fact` 从 `rule_hit` 重算主结论；
- 接口不得绕开 `legal_action_recommendation` 从原始公告拼行动建议；
- `saleable_opportunity` 不得绕开阶段 6 直接从单条规则或销售备注生成；
- `contact_target` 不得绕开合法来源与治理状态直接从自然人字段推导生成；
- 交付、治理与商业结果不得反向篡改阶段 3 真相层对象。

### [D2-R-009] 2.3 正式对象的最低成立标准

一个对象只有在同时满足以下条件时，才算正式对象：

1. 已被 L0 或本文明确声明存在；
2. 已明确阶段归属；
3. 已明确最小上游依赖；
4. 已明确最小下游消费方；
5. 已冻结最小必有字段；
6. 已明确不得绕开的硬约束；
7. 已有明确机器承接落点。

少任一项，都只能算临时结构、辅助结构或实现细节，不算正式对象。

## [D2-R-010] 3. 对象分层与生命周期总览

### [D2-R-011] 3.1 正式对象分层

| 层级 | 默认阶段 | 对象族 | 核心职责 |
|---|---|---|---|
| 调度与入口层 | 1 | `task_record`、`execution_context`、`review_lane_profile` | 启动任务、声明策略、冻结窗口与归一路径 |
| 真相层 | 2-3 | `public_chain`、`clock_chain_profile`、`notice_version_chain`、`project_base`、`field_lineage_record`、`bidder_candidate`、`project_manager` | 固化公开载体、结构化关键对象与字段级来源 |
| 核验层 | 4 | `public_attack_surface`、`focus_bidder_verification_profile`、`pseudo_competitor_signal_set`、`evidence_grade_profile`、专题画像对象 | 围绕关键对象执行公开核验、冲突预判与证据等级评估 |
| 规则证据层 | 5 | `rule_hit`、`evidence`、`review_request`、`rule_gate_decision`、`evidence_gate_decision` | 运行规则、固化证据、形成双闸门与复核请求 |
| 统一事实层 | 6 | `project_fact`、`legal_action_recommendation`、`review_queue_profile`、`report_record` | 形成唯一正式统一事实、动作建议、报告与复核编排 |
| 商业对象层 | 7 | `challenger_candidate_profile`、`legal_action_actor_profile`、`procurement_decision_actor_profile`、`sales_lead`、`buyer_fit`、`challenger_buyer_fit`、`offer_recommendation`、`saleable_opportunity`、`account_context` | 把统一事实转成可经营、可判断、可推进的正式商业对象 |
| 触达对象层 | 8 | `contact_target`、`outreach_plan`、`touch_record` | 在合规边界内形成可触达对象、触达动作与触达留痕 |
| 交付与治理回写层 | 9 | `order_record`、`payment_record`、`delivery_record`、`opportunity_outcome_event`、`governance_feedback_event` | 订单、支付、交付、失败原因与治理结果正式回写 |
| 治理隔离层 | 跨阶段 | `private_supplement_record`、`controlled_exception_record`、`model_governance_record` | 隔离非公开补证、受控例外与模型入链治理 |

### [D2-R-012] 3.2 生命周期原则

- 正式对象只能沿阶段方向推进，不得倒置生成。
- 后阶段对象可以回写前阶段的运行状态，但不得反向改写前阶段真相对象的内容事实。
- 同一对象在不同阶段可以被补充，但不得在不同阶段出现并行同名、异义或多套结构。
- 一个对象一旦被认定为正式对象，其字段命名、状态字段、引用关系与上游依赖不得在实现层私自重写。

## [D2-R-013] 4. 建模与命名规则

### [D2-R-014] 4.1 命名规则

| 项 | 规则 |
|---|---|
| 对象名 | 一律使用 `snake_case` |
| 主键字段 | 一律使用 `*_id` |
| 可选主键或可选引用字段 | 仅在确有上位裁决允许时使用 `*_id_optional` |
| 状态字段 | 一律使用 `*_status`、`*_state`、`*_level`、`*_lane`、`*_bucket` |
| 时间字段 | 一律使用 `*_at`、`*_from`、`*_until`、`*_due_at` |
| 布尔字段 | 使用 `is_*`、`has_*`、`requires_*` |
| 对象引用 | 只能引用正式对象 ID，不得引用页面临时 key 或数组索引 |

### [D2-R-015] 4.2 字段归属规则

- 本文只冻结对象最小必有字段与字段命名规则。
- 字段是否允许客户可见、是否要脱敏、是否进入对外 API、是否允许导出，由 D6 和 D7 继续裁决。
- 字段依赖的正式枚举统一进入 `contracts/enums/enum_catalog.json` 承接。
- 未冻结枚举的字段，不得进入正式对象。

### [D2-R-016] 4.3 引用规则

- 对象之间的引用必须以正式对象 ID 进行。
- 顶层对象不得直接引用页面组件状态、脚本状态或人工备注作为正式字段。
- `report_record`、`saleable_opportunity`、`contact_target`、`delivery_record` 等顶层对象，必须能追溯到其最小上游正式对象。
- 任何字段若无法追溯到正式来源对象或来源切片，不得进入顶层对象。

## [D2-R-019] 7. 阶段 1 对象

### [D2-R-077] 17.1 Schema 承接规则

- `contracts/schemas/schema_catalog.json` 是正式对象契约的唯一机器承接目录。

  - `sourceL0Refs`
  - `sourceD2Refs`
- 顶层对象不得在 schema 中偷加本文未声明的核心字段族。

### [D2-R-078] 17.2 Enum 承接规则

- `contracts/enums/enum_catalog.json` 是正式枚举目录。
- 本文出现的状态字段、结果字段、release 字段、审批字段、交付字段都必须在枚举目录中有正式承接。
- 任何对象若依赖未冻结枚举，不得进入正式页面、正式接口、正式规则实现或正式测试断言。

### [D2-R-079] 17.3 双口径阻断规则

以下任一情况都视为对象双口径：

1. 文档对象名与 schema 对象名不一致；
2. 对象阶段归属不一致；
3. 最小必有字段不一致；
4. 硬约束不一致；
5. 下游消费方向不一致。

一旦发生双口径，必须阻断后续实现。
