## [D11-R-001] 0. 文档任务与裁决范围

本文档的任务不是简单列出“跑哪些测试”，而是把  的正式测试口径收敛为一套可执行、可审计、可回归、可阻断发布的统一验收标准。

本文档必须回答以下问题：

1. 什么叫“真正做完”；
2. 哪些对象、规则、gate、页面、接口、治理资产必须进入最小必测面；
3. L0 的关键断言如何在测试资产中落地；
4. 什么是一票失败条件；
5. 哪些变更必须触发回归、治理检查或发布阻断；
6. golden case、fixture、schema 校验、release 检查、治理检查如何分工；
7. 普通 direct-dev 与高风险/live/机器契约窗口分别需要哪些测试资产。

本文档不直接展开以下事项：

- 业务抓取实现、解析实现、规则算法实现的具体代码细节；
- 页面样式、按钮视觉呈现、导出排版细节；
- 商业机会评分公式、触达文案模板、支付与交付流程实现；
- 部署平台、CI/CD 工具、具体监控产品选型。

上述事项分别由 D2、D3、D4、D5、D8、D9、D10、D12 承接，但其完成判定、回归要求与发布阻断必须受本文约束。

## [D11-R-002] 1. 生效规则与对齐原则

### [D11-R-003] 1.1 生效规则

- D11 是  正式测试验收、金标回归与发布前检查的唯一文档口径。
- 任一正式对象、正式规则、正式接口、正式页面、正式导出、正式交付或正式治理资产，只要进入仓库基线，就必须能映射到 D11 的测试入口。
- 测试通过不等于越过上位边界；若实现突破 L0、D13、D6、D7 的硬边界，测试应判失败，而不是用“业务例外”放行。
- 文档、contracts、scripts、fixture、golden、release checklist 必须保持单一口径；任何一处双口径，都视为测试资产不完整。
- D11 只允许细化 L0 与各 D 文档的正式要求，不允许反向改变对象定义、结果语义、公开边界或交付边界。

### [D11-R-004] 1.2 测试层对齐原则

所有正式测试都必须同时对齐以下八个维度：

1. **对象对齐**：测试必须围绕正式对象，而不是围绕页面临时字段或脚本中间变量；
3. **结果对齐**：测试必须验证 `AUTO_HIT / CLUE / OBSERVATION / review_request` 的正式语义；
4. **gate 对齐**：测试必须验证 `rule_gate_decision` 与 `evidence_gate_decision` 缺一不可；
5. **消费对齐**：测试必须验证页面、接口、导出、销售、交付不得绕开 `project_fact` 与正式治理链；
6. **治理对齐**：测试必须验证字段策略、对象级交付矩阵、release gates、审批链与例外治理；
7. **回归对齐**：正式变更必须能映射到回归 manifest；
8. **发布对齐**：通过单测、schema 校验不等于允许发布；发布必须经过 release 检查与治理检查。

### [D11-R-005] 1.3 D11 不得放宽的上位事项

D11 不得放宽以下上位要求：

- `project_fact` 是唯一统一事实中枢；
- `rule_gate_decision` 与 `evidence_gate_decision` 缺一不可；
- `saleable_opportunity` 不得直接消费单条规则、单页公告或销售备注；
- 自然人高限制字段、联系对象释放、外发主证必须经过治理与审批；
- 页面、接口、导出、销售表述不得形成第二套主判断体系。
- 招投标双线业务方向必须进入测试面：`POST_CANDIDATE_EVIDENCE_PACK` 候选后证据包是核心商业主线，`PRE_BID_PREDICTION` 是辅助产品线；投前预测遇到 `05 开标信息`、已过投标截止/开标时间或已出现 `07 中标候选人公示` 必须判为不适用；标准投前预测销售窗口必须为截止/开标前 168 小时以上，72-168 小时只能做限时快筛，少于 72 小时不得作为正常投前预测销售。
- `AnalysisStrategyPlan v1` 必须在下载和解析前决定产品线、流程范围、文件动作、解析深度、规则核验和大模型触发条件；测试不得允许“发现文件就全部深解析”的实现通过。
- 广州 `01-12` 流程样例 URL 只能作为适配器验证样本；测试必须防止它们成为默认生产爬取目标、真实样本数量或下载完整率统计。

## [D11-R-006] 2. 验收总原则

### [D11-R-007] 2.1 验收不是“脚本绿了”

正式验收不是“某几个脚本返回 0”，而是以下事项同时成立：

1. 文档口径稳定；
2. contracts 与文档一致；
3. golden 样例与断言可执行；
4. 关键对象、关键 gate、关键边界已进入必测面；
5. 发布前检查未发现对象越界、字段越界、边界越界、交付越界；
6. 关键消费面未反向重算主判断；
7. 关键治理资产已可追溯；
8. 没有一票失败信号。

### [D11-R-008] 2.2 测试资产分层

正式测试资产分为六层：

1. **Schema / Contract 校验**：对象、枚举、引用关系、必填字段、枚举来源；
2. **Golden Case 校验**：正向、反向、降级、阻断样本；
3. **接口契约校验**：资源路径、operationId、权限矩阵、错误码、封套一致性；
4. **页面 / 导出契约校验**：工作台目录、按钮流、导出模板、保留态阻断；
5. **治理契约校验**：字段策略、交付矩阵、release gate、审批链、例外目录；
6. **Release 检查**：当前变更是否扩大承诺、放宽边界、提升 release level、放宽高限制字段。

### [D11-R-009] 2.3 测试资产的最低要求

- 每一类正式对象至少要有 schema 断言；
- 每一类正式规则至少要有 golden case 与降级 / 阻断样本；
- 每一类正式交付形态至少要有 release 检查；
- 每一类高限制字段或对象至少要有治理阻断样本；
- 每一类正式消费面至少要有“不得绕开 `project_fact` / gate / governance”的断言。

## [D11-R-010] 3. P0-P8 验收门槛

> 注：本文沿用现有测试分层表达，P0-P8 是验收成熟度门槛，不等同于业务阶段 1-9。

| 验收层级 | 正式名称 | 最低通过件 | 一票失败信号 |
|---|---|---|---|
| `P0` | 统一底座成立 | 单一 L0 母版、单一裁决索引面、正式对象与正式枚举无双口径 | 出现第二套对象名、第二套结果语义、第二套边界口径 |
| `P1` | 抓取成立 | `public_chain`、`clock_chain_profile`、`notice_version_chain` 能追溯原始载体 | 原始页面 / 附件不可追溯，时间链或版本链无法固定 |
| `P2` | 解析成立 | `project_base`、`bidder_candidate`、`project_manager`、`field_lineage_record` 稳定形成 | 关键字段无来源切片、无来源版本、无归一规则 ID 却进入正式主链 |
| `P4` | 规则与证据成立 | 至少一类 A 层正式规则与一类 B/C 层降级规则稳定输出；双闸门同步成立 | 只跑规则不跑证据，或缺任一 gate 仍升级 |
| `P5` | 统一事实成立 | `project_fact`、`legal_action_recommendation`、`review_queue_profile`、`report_record` 可稳定形成 | 项目归一未完成、gate 未过、边界未过却直接形成正式主结论 |
| `P6` | 商业识别成立 | `challenger_candidate_profile`、`legal_action_actor_profile`、`procurement_decision_actor_profile`、`saleable_opportunity` 只消费阶段 6 正式对象 | 机会对象由单条规则、单页公告或销售备注直接生成 |
| `P7` | 触达与收入回写成立 | `contact_target`、`touch_record`、`order_record`、`payment_record`、`delivery_record`、`governance_feedback_event` 可回写正式链路 | 联系来源不合规、触达无审批 / 频控 / quiet hours / 审计 |

### [D11-R-011] 3.1 当前验证最低门槛

普通 direct-dev 按影响面运行最小必要验证；高风险/live/机器契约窗口至少必须满足以下验收前提：

- `P0` 至少达到稳定通过；
- `P1-P5` 的 schema、golden 与基本回归入口已存在；
- L0 断言最小集已落库；
- release 检查可阻断对象越界、字段越界、外发越界与公开边界越界。

## [D11-R-017] 5. L0 断言落地

### [D11-R-018] 5.1 断言总原则

L0 断言不是说明文案，而是必须具备可执行入口、失败语义与阻断动作的正式测试资产。

### [D11-R-019] 5.2 核心断言目录

| 断言 ID | 断言内容 | 最低落地资产 | 失败后的正式处理 |
|---|---|---|---|
| `L0-A-002` | `project_fact` 是唯一正式统一事实中枢 | schema 校验 + page/API contract check | 一票失败，阻断主判断消费 |
| `L0-A-003` | 双闸门缺一不可 | gate policy check + golden | 一票失败，阻断规则升级与客户版放行 |
| `L0-A-004` | `saleable_opportunity` 必须消费阶段 6 正式对象 | schema 引用检查 + regression | 一票失败，阻断商业对象放行 |
| `L0-A-005` | 自然人高限制触达必须受审批、频控、quiet hours 与审计约束 | governance checks + release checks | 一票失败，阻断联系对象释放与触达执行 |

### [D11-R-020] 5.3 断言实现要求

- 每条断言必须有唯一断言 ID；
- 每条断言必须说明适用对象、适用交付形态与失败动作；
- 断言失败不得降格为 warning；
- 任一断言只要被正式标记为一票失败，就不得通过页面文案、模板裁剪或例外记录掩盖。

## [D11-R-021] 6. 正式金标样例体系

### [D11-R-022] 6.1 金标样例目标

Golden case 的目标不是展示“最好看的命中结果”，而是覆盖正式结果语义、降级逻辑、阻断逻辑与治理边界。

### [D11-R-023] 6.2 首版金标样例目录

| 样例 ID | 样例目标 | 预期结果 | 必须覆盖的要点 |
|---|---|---|---|
| `GC-CLUE-001` | 项目经理公开冲突线索成立，但仍需人工确认 | `CLUE` + `REVIEW` | B/C 交界、不可越级为 `AUTO_HIT` |
| `GC-OBSERVATION-001` | 评分异常存在信号，但公开颗粒度不足 | `OBSERVATION` + `REVIEW` | C 层线索、颗粒度不足不得升级 |
| `GC-BLOCK-002` | 只存在截图 / OCR / 模型摘要而无原始载体 | `BLOCK` | 主证不可回链阻断 |
| `GC-BLOCK-003` | `project_fact` 未形成却尝试生成正式机会对象 | `BLOCK` | 单向消费原则 |
| `GC-BLOCK-004` | 自然人联系方式未审批却尝试进入触达执行 | `BLOCK` | 高限制字段与联系对象释放阻断 |
| `GC-API-001` | 客户只读接口试图原样返回内部 gate 明细 | `BLOCK` | D4 / D6 / D7 一致性 |
| `GC-EXPORT-001` | 客户版模板尝试导出隔离补证正文 | `BLOCK` | D6 / D7 外发治理一致性 |

### [D11-R-024] 6.3 金标样例编制要求

- 每个样例必须包含：输入对象、适用边界层级、核心前提、预期结果、预期 gate、预期 release 结果、预期阻断原因；
- 金标样例不得只覆盖正向命中，必须覆盖负向、降级、阻断、治理冲突；
- B / C 层样例必须覆盖“允许存在但不得冒充全国默认能力”的情况；
- 样例必须能被 scripts 执行，不得只停留在文档叙述。

## [D11-R-025] 7. 一票失败条件

下列情形任一成立，都必须判定为**一票失败**：

2. 关键字段缺失 `field_lineage_record` 最小必有信息，却被 `project_fact`、`legal_action_recommendation`、客户版报告或对外 API 消费；
3. `rule_gate_decision` 或 `evidence_gate_decision` 任一缺失、未过或未正确消费，却仍升级为正式主结论；
4. `project_fact` 尚未形成，却由规则层、页面层、接口层、销售层直接构造顶层主判断；
5. `saleable_opportunity` 未消费阶段 6 正式对象，而是由单条规则、单页公告、CRM 备注或人工临时结论直接生成；
6. `contact_target` 来源不合规，或触达缺审批链、频控、quiet hours、审计留痕；
7. 正式客户版证据包、法律动作建议或外发主证仅以截图、OCR 文本或模型摘要作为唯一主证；
8. 页面、接口、导出、销售材料与 A/B/C/D 边界矩阵不一致；
9. 对象级交付矩阵允许、字段策略禁止时，系统仍然放行；
10. 文档已改，但 contracts / testing / scripts 未同步。

## [D11-R-026] 8. 变更触发回归矩阵

### [D11-R-027] 8.1 变更总原则

任何影响正式对象、正式枚举、结果语义、公开边界、字段可见性、交付形态、联系对象释放、外发模板、审批链或例外治理的变更，都必须触发对应回归。

### [D11-R-028] 8.2 变更与必跑检查映射

| 变更类型 | 必跑检查 |
|---|---|
| 正式对象、正式枚举、状态机变化 | `validate-contracts` + `run-golden` + `check-release` |
| 规则码、结果语义、证据等级、双闸门逻辑变化 | `run-golden` + gate policy check + regression manifest |
| A/B/C/D 打标变化、公开边界变化 | `run-golden` + `run-governance-contracts` + `check-release` |
| 接口资源、权限矩阵、错误码、返回封套变化 | `scripts/check-state-alignment.ps1` + `python tests/run_tests.py` + `run-golden` |
| 页面工作台、按钮流、导出模板变化 | `python tests/run_tests.py` + `run-golden` + `check-release` |
| 字段策略、脱敏、审批链、客户白名单变化 | `run-governance-contracts` + `check-release` |
| 对象级交付矩阵、release gate、外发模板变化 | `run-governance-contracts` + `check-release` |
| 联系对象释放、频控、quiet hours、触达渠道变化 | `run-governance-contracts` + `run-golden` + `check-release` |
| 例外与治理补证策略变化 | `run-governance-contracts` + release exception check |

### [D11-R-029] 8.3 不允许跳过回归的场景

- 为了赶窗口期而跳过回归；
- 只改“文案”却实际改变了结果语义或 release 语义；
- 只改“模板”却实际扩大了字段可见范围或对象放行范围；
- 只改“脚本”却实际放宽了上位断言或发布门禁。

## [D11-R-030] 9. 页面、接口、导出、治理一致性检查

### [D11-R-031] 9.1 页面一致性检查

页面侧至少要验证：

- 项目事实页是否优先消费 `project_fact`；
- 证据工作台是否把 `field_lineage_record`、gate 对象保留在解释层；
- 保留态页面是否仍被误做成真执行面；
- 导出按钮是否错误绕过审批链与 release level；
- 演示预览版是否冒充正式客户版。

### [D11-R-032] 9.2 接口一致性检查

接口侧至少要验证：

- 顶层结论接口是否直接返回正式对象而不是中间对象拼装；
- 客户只读接口是否误返回内部 reasoning 字段；
- 错误码是否与阻断理由、裁决引用一致；
- RESERVED / BLOCKED 能力是否在接口层被显式阻断。

### [D11-R-033] 9.3 导出与外发一致性检查

导出与外发至少要验证：

- 模板是否只消费白名单对象与白名单字段；
- `EXTERNAL_ACTION_READY` 与证据等级是否匹配；
- 外发主证是否具备固定包、原始载体、时间戳与调阅路径；
- 内部补证、例外正文、自然人高限制字段是否错误进入正式外发包。

### [D11-R-034] 9.4 治理一致性检查

治理侧至少要验证：

- `coverage_registry`、`public_boundary_registry`、`field_policy_dictionary`、`delivery_matrix`、`release_gates` 之间是否互相可回指；
- 例外记录、补证记录、审批链是否完整；
- 当前 release 结果是否与当前策略目录一致；
- 是否存在通过脚本、配置热修、数据库直改绕过治理链的情况。
