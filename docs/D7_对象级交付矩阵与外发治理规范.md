## [D7-R-001] 0. 文档任务

本文件不重新定义对象，不重新定义结果语义，也不重新定义能力边界。  
本文件只回答以下问题：

1. 哪些正式对象可以进入内部研判版、客户交付版、外发异议辅助包、演示预览版、对外 API；
2. 每种交付形态要求的最低 `release_level`、最低证据等级、最低审批链与最低审计留痕是什么；
3. 哪些对象只能进入内部面，哪些对象只能摘要出现，哪些对象一律禁止外发；
4. D6 字段策略与 D7 对象级交付矩阵发生冲突时，以谁为准；
5. 受控例外、私有补证、人工覆盖、治理阻断在交付链路中如何生效；
6. 何种情况下必须一票阻断客户版、外发包、对外 API 或演示版放行。

## [D7-R-002] 1. 生效规则与优先级

### [D7-R-003] 1.1 生效规则

- D7 是  正式交付治理的唯一对象级矩阵口径。
- 任一对象是否允许进入某种交付形态，必须先经过 D7 裁决，再进入 D6 字段策略裁决。
- 页面、接口、导出模板、外发包、销售材料、演示版、对外 API 均视为交付形态消费面，统一受本文件约束。
- D7 只允许细化 L0、D2、D3、D6、D13，不允许放宽其硬边界。
- 任一实现若无法说明其交付形态、release level、审批链、审计链与阻断理由，不得视为正式交付实现。

### [D7-R-004] 1.2 冲突优先级

当同一对象面临多层约束时，优先级固定如下：

1. **L0 硬边界与 D13 公开能力边界**
2. **D3 双闸门与规则优先序**
3. **D7 对象级交付矩阵**
4. **D6 字段策略与审批链**
5. **模板级或页面级展示细化**
6. **局部例外、演示需要、销售诉求**

### [D7-R-005] 1.3 更严格者优先

- D7 允许、D6 禁止时：**禁止**
- D7 摘要化、D6 原值允许时：**摘要化**
- D7 允许、release gate 未通过时：**阻断**
- D7 允许、字段策略允许、但审计链缺失时：**阻断**
- D7 允许、字段策略允许、审批链完成、但对象仍含 D 层输入时：**阻断**

## [D7-R-006] 2. 正式交付形态与 release level

### [D7-R-007] 2.1 正式交付形态

| 交付形态 | 代码 | 正式含义 | 典型消费面 |
|---|---|---|---|
| 内部研判版 | `INTERNAL_REVIEW` | 内部核查、复核、追溯、治理调试与内部简报 | 内部工作台、内部复核简报、治理后台 |
| 客户交付版 | `CLIENT_DELIVERY` | 正式客户可见报告、客户只读视图、客户版摘要导出 | 客户报告、客户只读页、客户交付包 |
| 外发异议辅助包 | `EXTERNAL_ACTION_ASSIST` | 供正式异议、质疑、投诉或相关动作准备使用的外发辅助包 | 外发异议辅助包、正式动作准备材料 |
| 演示预览版 | `DEMO_PREVIEW` | 演示、售前预览、样板查看、受限预览 | 演示页、售前样板、预览摘要 |
| 对外 API | `EXTERNAL_API` | 客户授权的对象投影与摘要接口 | 客户集成、受控查询接口 |

### [D7-R-008] 2.2 正式 release level

| release level | 正式含义 | 最低适用形态 |
|---|---|---|
| `INTERNAL_ONLY` | 仅内部可见，不得客户可见、不得对外 API、不得正式外发 | `INTERNAL_REVIEW`、`DEMO_PREVIEW` |
| `CLIENT_VISIBLE` | 可进入客户版交付与对白名单对象的对外 API | `CLIENT_DELIVERY`、`EXTERNAL_API` |
| `EXTERNAL_ACTION_READY` | 可进入正式外发动作辅助包 | `EXTERNAL_ACTION_ASSIST` |

### [D7-R-009] 2.3 release level 与对象放行的关系

- `INTERNAL_ONLY` 不代表一定允许内部全量显示，仍需受 D6 字段策略与角色权限限制。
- `CLIENT_VISIBLE` 不代表当然可外发；正式外发必须达到 `EXTERNAL_ACTION_READY`。
- 任一对象若其核心证据尚未达到目标场景最低 `external_use_grade`，则不得升级到对应 `release level`。
- release level 只回答“理论上可以进入何种交付形态”，不替代审批链、审计链、模板白名单和字段白名单。

## [D7-R-010] 3. 对象级交付矩阵总原则

### [D7-R-011] 3.1 总原则

- **对象先于字段**：先判断对象能否进入交付形态，再判断字段能否进入。
- **高优先对象优先消费**：`project_fact`、`legal_action_recommendation`、`report_record` 是正式交付主对象；不得低优先对象反向替代。
- **内部解释对象不得冒充正式主对象**：`field_lineage_record`、`rule_gate_decision`、`evidence_gate_decision`、`review_queue_profile` 只能作为解释层、追溯层或治理层。
- **治理隔离对象不得静默泄露**：`private_supplement_record`、`controlled_exception_record` 正文默认不得进入客户版、外发包、演示版或对外 API。
- **商业对象、触达对象、履约对象受批次与治理双约束**：即便对象正式存在，也必须在相应文档与治理链放行后方可进入客户可见或外发形态。

### [D7-R-012] 3.2 禁止事项

- 不得因为页面需要解释更多细节，就把内部解释对象直接透给客户。
- 不得因为销售需要“更有说服力”，把演示预览版冒充正式客户交付版。
- 不得因为对象被允许进入某形态，就默认其全部字段都可原值出现。
- 不得通过截图、附件打包、隐藏字段、tooltip、错误消息、日志、缓存、导出 JSON 等绕开矩阵。
- 不得用局部例外去放宽 D 层禁入边界或双闸门阻断。

## [D7-R-025] 6. 对象级特殊规则

### [D7-R-026] 6.1 `project_fact`

- 是客户交付版、外发异议辅助包、演示预览版与对外 API 的首选顶层对象。
- 对外 API 仅允许投影白名单摘要，不允许输出全部内部聚合字段。
- 若 `project_fact` 所依赖的任一核心输入含 D 层输入、双闸门阻断或未审计人工覆盖，必须阻断升级交付。

### [D7-R-027] 6.2 `legal_action_recommendation`

- 客户交付版默认允许摘要进入。
- 外发异议辅助包允许进入正式对象，但必须达到 `EXTERNAL_ACTION_READY`。
- 若未明确 `window_status`、`applicable_regime` 或 `competent_authority_scope`，不得进入正式外发包。

### [D7-R-028] 6.3 `field_lineage_record`

- 永远不属于客户主对象、外发主对象或对外 API 对象。
- 仅供内部追溯、复核、治理与审计使用。
- 不得以图片、附件截图或引用块形式伪装透出。

### [D7-R-029] 6.4 `rule_gate_decision` / `evidence_gate_decision`

- 永远不作为客户版或外发版主对象原样出现。
- 只允许把结论投影为 `project_fact` 的摘要阻断原因、风险摘要或治理提示。
- gate 明细原文仅限内部工作台。

### [D7-R-030] 6.5 `private_supplement_record` / `controlled_exception_record`

- 正文默认隔离，只有元数据可在内部治理面可见。
- 不得进入客户版、外发包、对外 API、演示版。
- 任何“以附件形式单独发出”的做法都视为违规外发。

### [D7-R-031] 6.6 `contact_target`

- 属于高限制内部触达准备对象，不得进入客户交付版、外发包、演示版或对外 API。
- 即使内部也必须受 D6 高限制字段策略、专项审批、频控、quiet hours 与审计链约束。
- 不得因为已经存在于正式对象体系中，就误认为可以对销售或客户直接开放。

## [D7-R-032] 7. release gate 体系

### [D7-R-033] 7.1 正式 release gate 总表

| release gate | 适用交付形态 | 最低 release level | 最低前提 |
|---|---|---|---|
| `internal_review_release` | `INTERNAL_REVIEW` | `INTERNAL_OPERABLE` | 对象已形成、角色授权成立、审计可留痕 |
| `client_report_release` | `CLIENT_DELIVERY` | `LEADPACK_DELIVERABLE` | 对象矩阵放行、字段策略放行、审批链完成、最低证据等级满足 |
| `external_action_release` | `EXTERNAL_ACTION_ASSIST` | `LEADPACK_DELIVERABLE` | 双闸门通过、主证可回链、窗口说明齐备、审批链完成 |
| `demo_preview_release` | `DEMO_PREVIEW` | `DEV_ALLOWED` | 强免责声明、强脱敏、不可反向导出 |
| `external_api_release` | `EXTERNAL_API`（阻断保留位） | `EXTERNAL_BLOCKED` | 外部平台发布策略默认阻断；仅保留审批/治理映射，不形成对外可放行能力 |
| `high_restriction_contact_release` | 高限制联系对象内部查看 / 释放 | `INTERNAL_OPERABLE` | lawful basis、频控、quiet hours、专项审批与审计同时满足 |

### [D7-R-034] 7.2 gate 成立的最低检查项

任一正式放行至少应检查：

1. 对象是否在 D7 矩阵中被允许进入该交付形态；
2. 对象所需字段是否被 D6 放行；
3. 对象依赖的证据等级是否达到目标场景最低阈值；
4. 对象是否受 D13 边界允许；
5. 双闸门状态是否允许；
6. 审批链是否完成；
7. 审计链是否可追；
8. 是否存在高优先级阻断理由；
9. 是否存在未决人工覆盖或未决例外；
10. 是否满足模板白名单或接口白名单。

## [D7-R-035] 8. 一票阻断规则

以下任一成立时，必须阻断客户版、外发包、对外 API 或演示版放行：

1. 任一核心输入来自 `D_NON_PUBLIC_OR_FORENSIC`；
2. `evidence_gate_decision` 未达到目标交付形态最低证据等级；
3. 正式外发主证仅依赖截图、OCR 文本或模型摘要，而无可回链原始载体；
4. `private_supplement_record` 仍处于隔离或待审状态，却试图影响正式交付；
5. `manual_override_status` 为待审或被更高优先级门禁覆盖；
6. D7 对象级矩阵允许，但 D6 字段策略未允许；
7. B 层或 C 层能力被包装成全国默认稳定能力；
8. 演示版或预览版被用于正式客户交付；
9. 对外 API 试图投影内部解释对象或高限制自然人字段；
10. 例外链路试图放宽 D 层禁入、双闸门或审批链硬门槛。

## [D7-R-036] 9. 审批链、审计链与职责分离

### [D7-R-037] 9.1 审批链要求

| 动作 | 最低审批链 |
|---|---|
| 内部研判简报放行 | `internal_review_release` |
| 客户版报告放行 | `client_report_release` |
| 外发异议辅助包放行 | `external_action_release` |
| 高限制联系对象查看或释放 | `high_restriction_contact_release` |
| 受控例外放行 | `boundary_exception_release` |

### [D7-R-038] 9.2 SoD 原则

- 创建对象的人不得同时是最终放行人；
- 私有补证的提交者不得同时是释放者；
- 例外的申请者不得同时是批准者；
- 高限制字段查看申请人与批准人不得处于同一审批链终点；
- 审批链缺失、审计链缺失、角色混同，均视为阻断。

### [D7-R-039] 9.3 审计留痕最低要求

每一次正式放行至少应留痕：

- 对象 ID
- 交付形态
- release gate ID
- release level
- 模板或接口 operation
- 审批链记录
- 操作人、批准人、时间
- 证据等级快照
- 版本 / 时钟快照
- 放行或阻断理由

## [D7-R-040] 10. 例外治理与补证治理

### [D7-R-041] 10.1 受控例外

- 受控例外只能用于短期止损、兼容过渡、应急阻断排除或治理性暂缓。
- 受控例外只能降低、延后、限制交付，不得扩大交付边界。
- 受控例外不得把 D 层材料提升为正式主证，不得让原本 `BLOCK` 的对象进入客户版或外发包。

### [D7-R-042] 10.2 私有补证

- 私有补证默认只存在于隔离链。
- 私有补证可以影响内部复核或治理判断，但不能未经审批直接进入正式客户版、对外 API 或外发包。
- 私有补证若要影响正式交付，必须具备明确场景、明确合法基础、明确影响范围与完整审批链。

---

## [D7-R-043] 11. 模板、页面、接口的一致性要求

### [D7-R-044] 11.1 页面

- 页面不得自行决定对象级放行。
- 工作台、摘要页、治理后台、演示页都必须读取 D7 矩阵裁决结果。
- 页面不得通过隐藏区块、展开面板、调试开关绕开对象矩阵。

### [D7-R-045] 11.2 导出模板

- 导出模板只能消费 D7 已放行的对象。
- 模板是否可导出，不取决于页面当前可见，而取决于 release gate 是否允许。
- 导出模板不得把内部对象拼成客户版或外发包。

### [D7-R-046] 11.3 对外 API

- API 只允许投影 D7 与 D6 同时放行的对象与字段。
- API 不得因为“调用方便”而暴露内部解释对象。
- 错误响应不得泄露内部对象明细或高限制字段。

## [D7-R-052] 13. 测试与回归要求

### [D7-R-053] 13.1 最小必测项

1. 对象矩阵允许 / 禁止是否与实现一致；
2. D6 与 D7 冲突时是否按更严格者执行；
3. gate 放行与阻断是否正确；
4. 外发包主证是否满足原始回链要求；
5. 演示版是否强制免责声明与强脱敏；
6. 对外 API 是否只返回白名单对象与白名单字段；
7. 高限制联系对象是否被严格阻断；
8. 例外与私有补证是否无法绕开正式边界。

### [D7-R-054] 13.2 一票失败条件

- 客户版看到 `field_lineage_record` 原文；
- 外发包包含 `rule_gate_decision` / `evidence_gate_decision` 明细；
- 对外 API 泄露 `contact_target` 或高限制自然人字段；
- 演示版被标识为正式交付物；
- D 层输入经由例外或补证进入正式主交付对象；
- 对象矩阵与 D6 字段策略不一致且实现选择了更宽松的一侧。

## [D7-R-055] 14. 完成定义

D7 只有同时满足以下条件，才算正式完成：

1. 正式交付形态与 release level 冻结；
2. 核心对象矩阵冻结；
3. release gate 与一票阻断规则冻结；
4. 审批链、审计链、SoD 要求冻结；
5. 与 D6、D13、D11 的承接关系冻结；
6. 机器资产存在唯一承接面；
7. 测试回归与阻断样例已经补齐；
8. 页面、接口、模板都不再自造对象放行逻辑。

## [D7-R-056] 15. 正式禁止事项总表

- 不得以演示预览版冒充正式客户交付版；
- 不得以对象存在为由，默认对象可见；
- 不得以字段已脱敏为由，放宽对象级禁止；
- 不得通过导出、API、日志、缓存、截图、压缩包绕开矩阵；
- 不得以例外机制永久放行本应阻断的对象；
- 不得把 D 层、隔离链或内部解释对象伪装成正式外发主对象；
- 不得形成第二套 release gate 或第二套客户可见矩阵。
