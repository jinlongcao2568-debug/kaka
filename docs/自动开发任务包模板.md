# 自动开发任务包模板

> 本模板只回答一件事：`task packet` 应该怎么写。  
> 正式 machine-readable `task packet` 统一落在 `control/current_task.yaml -> currentTask.task_packet`。  
> 本模板不承担当前开发模式说明书、repo readiness 说明书或执行优先级说明书职责。

## 1. 使用范围

- 用途：起草、评审、补齐 `task packet` 字段，确保当前执行窗口有明确目标、范围、门禁、脚本和停机条件。
- 上位约束：`L0.md`、`裁决总表.md`、`D1_研发_Codex执行手册.md`
- 适用对象：`activation`、`scoped_execution`、`closeout`
- 口径约束：本模板只描述任务包怎么写，不解释当前 repo readiness、conditional-go、当前产品开发模式总规则，也不解释 `current_task` / `product_task_library` / 路线图之间的优先级。

## 2. 三种使用方式

- `activation`
  - 用于切换当前 active packet、同步最小控制面语义、锁定下一执行窗口。
  - machine-readable `execution_mode` 默认写 `ACTIVATION_ONLY`。
  - 典型落笔面：`control/current_task.yaml`、`control/repo_status.md`，必要时最小同步导航提示。
- `scoped_execution`
  - 用于在已激活包范围内执行正文收平、机器资产补齐或受控实现。
  - machine-readable `execution_mode` 默认写 `SCOPED_EXECUTION`。
  - 典型落笔面：以当前包 `declared_changed_paths` / `allowed_modification_paths` 为准。
- `closeout`
  - 用于当前窗口收尾、停机汇报、准备提交或关闭当前执行语义。
  - `closeout` 是工作流阶段，不要求单独新增 `execution_mode` 枚举；当前仓库仍以 `ACTIVATION_ONLY / SCOPED_EXECUTION` 这两个 machine-readable 值承接。

说明：
- 上述 3 种是任务包使用方式，不要求再造第二套 machine-readable 结构。
- 正式写入时，仍以 `control/current_task.yaml -> currentTask.task_packet` 中现有字段表达。
- 若当前窗口会消费 `currentTask.operator_assignment_roster.stage7` / `stage8` / `stage9`，必须同时保留 `currentTask.operator_assignment_roster_source_ref=control/operator_assignment_roster_defaults.yaml#defaults`，并从该稳定来源继承对应 roster。
- governance 过渡包 / control 对齐包可以合法存在于 `current_task` 中，但不应误写进 `control/product_task_library.yaml` 的主线候选池。

## 4. 修改范围模板

- `declared_changed_paths`
  - 声明本轮预计会落笔的真实路径
- `baseline_dirty_paths`
  - 可选但强烈建议填写
  - 写入“当前 packet 激活前工作区里已存在的 tracked dirty paths”
  - validator 会按：`actual_changed_paths - baseline_dirty_paths - generated_runtime_artifacts` 做范围校验
  - 不允许把本轮准备新增修改的真实目标路径伪装进 baseline
- `allowed_modification_paths`
- `forbidden_modification_paths`
- `impacted_assets.docs`
- `impacted_assets.control`
- `impacted_assets.contracts`
- `impacted_assets.handoff`
- `impacted_assets.scripts`
- `impacted_assets.tests`
- `impacted_assets.runtime`

补充约束：
- `control/current_task.yaml` 只允许用于当前 active packet 的 machine-readable 承接与必要的 execution semantics 同步。
- `currentTask.operator_assignment_roster.stage7` / `stage8` / `stage9` 不应再作为每次切包时手工发明的临时值；应继承自 `control/operator_assignment_roster_defaults.yaml#defaults`，并同步写入 `currentTask.operator_assignment_roster_source_ref`。
- 路径声明要能覆盖实际修改，但不能放宽成与本轮主题无关的大范围白名单。
- 在真正改文件前，先运行一次 preflight：
  ```powershell
  $paths = @(
    'control/current_task.yaml',
    'control/repo_status.md'
  )
  pwsh -NoProfile -ExecutionPolicy Bypass -File scripts/check-task-packet.ps1 -PlannedTargetPaths $paths
  ```
- `planned target paths / intended changed files` 必须先同时匹配 `declared_changed_paths` 与 `allowed_modification_paths`，且不得命中 `forbidden_modification_paths`。
- `scripts/check-task-packet.ps1` 对单字符串逗号拼接形式有兼容处理，仅用于工具链兜底；首选仍是 PowerShell 数组变量传参。
- 文档只负责提醒；真正阻断以 `control/automation_task_packet_rules.yaml` 和 `scripts/check-task-packet.ps1` 为准。

## 5. Review Gate 模板

- `human_review_required`
  - `MANDATORY_HUMAN_REVIEW` 默认必须为 `true`
- `owner_reviews_required`
  - 默认至少包含：`architecture_owner`、`governance_owner`、`testing_owner`
  - 若 `change_domains` 包含 `automation_control_core`，还必须包含：`automation_owner`
  - 若 `change_domains` 包含 `governance_release_core`、`future_external_unlock_prereq_core`、`future_external_unlock_decision`，还必须包含：`release_approver`
  - 若 `change_domains` 包含 `provider_vendor_source_policy_core`、`future_external_unlock_prereq_core`、`future_external_unlock_decision`，还必须包含：`model_governance_owner`
- `review_evidence.declared`
- `review_evidence.signoff_required`
- `review_evidence.signoff_status`
  - `MANDATORY_HUMAN_REVIEW` 默认必须为：`declared=true`、`signoff_required=true`、`signoff_status=REQUESTED_NOT_APPROVED`

## 6. 脚本与停机模板

- `required_scripts`
- `script_timeouts`
  - 建议显式填写
  - `scripts/check-final-gate.ps1` 默认建议 `300000` ms
  - 工具层 120 秒超时但脚本未返回 FAIL/非零时，不得直接判定为脚本失败；必须按本字段更长超时重跑一次
- `stop_conditions`
- 连续失败阈值：同一脚本 `>= 2` 次停机
- 若任务包修改 `scripts/**`，必须同时补对应 tests，避免 validator / release gate 继续依赖过期 token

## 7. 完成标准模板

- `definition_of_done`
  - 写清本轮结束时必须已经成立的状态
- `deliverables`
  - 写清本轮交付物，而不是泛泛写“已完成”
- `notes`
  - 用于补充 transition-safe 口径、当前包与 next candidate 的关系、以及不进下一包等窗口说明

## 8. 提交前收尾提醒

- 不得混入 `报告*.md`
- 不得混入 `__pycache__/`、`*.pyc`、`.pytest_cache/`
- 提交前收尾步骤应包含：`pwsh -NoProfile -ExecutionPolicy Bypass -File scripts/clean-python-cache.ps1`
- 提交前若还没做 preflight，先运行：
  ```powershell
  $paths = @(
    'control/current_task.yaml',
    'control/repo_status.md'
  )
  pwsh -NoProfile -ExecutionPolicy Bypass -File scripts/check-task-packet.ps1 -PlannedTargetPaths $paths
  ```
- 提交前应核对 `actual changed paths` 是否仍全部落在 `allowed_modification_paths` 内，并且不要超出 preflight 已声明的 `planned target paths`
- 若工作区存在不属于当前包的脏改，先停机汇报，不要混入当前提交

## 9. 输出格式

- 修改文件清单
- 每个文件修改了什么
- 脚本结果
- 当前阻断项
- 是否建议进入下一步
- 若不建议，最小补齐路径
