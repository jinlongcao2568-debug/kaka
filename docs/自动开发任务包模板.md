# 自动开发任务包模板

> 本模板只回答一件事：`task packet` 应该怎么写。  
> 正式 machine-readable `task packet` 统一落在 `control/current_task.yaml -> currentTask.task_packet`。  
> 本模板不承担当前开发模式说明书、repo readiness 说明书或执行优先级说明书职责。

## 1. 使用范围

- 用途：起草、评审、补齐 `task packet` 字段，确保当前执行窗口有明确目标、范围、门禁、脚本和停机条件。
- 上位约束：`L0.md`、`裁决总表.md`、`D1_研发_Codex执行手册.md`
- 适用对象：`activation`、`scoped_execution`、`closeout`
- 口径约束：本模板只描述任务包怎么写，不解释当前 repo readiness、conditional-go、当前产品开发模式总规则，也不解释 `current_task` / `task_packet_library` / 路线图之间的优先级。

## 2. 三种模式

- `activation`
  - 用于登记新包、激活当前包、同步最小控制面语义。
  - 重点字段：`source_blueprint_batch_id`、`packet_id`、`declared_changed_paths`、`allowed_modification_paths`、`required_scripts`
- `scoped_execution`
  - 用于在已激活包范围内执行正文收平、机器资产补齐或受控实现。
  - 重点字段：`objective`、`non_goals`、`impacted_assets`、`stop_conditions`、`definition_of_done`
- `closeout`
  - 用于当前窗口收尾、停机汇报、关闭当前执行语义，不自动进入下一包。
  - 重点字段：`deliverables`、`definition_of_done`、`stop_conditions`、当前窗口结束说明

说明：
- 上述 3 种模式是任务包的使用方式，不要求额外新增 machine-readable 字段。
- 正式写入时，仍以 `control/current_task.yaml -> currentTask.task_packet` 中现有字段表达，不另起第二套结构。

## 3. 必填字段模板

- `route_map_segment`
- `source_blueprint_batch_id`
  - 必须来自：`control/task_packet_library.yaml#blueprint_registry`
- `packet_id`
- `title`
- `objective`
- `depends_on`
- `non_goals`
  - 必须包含：`not an external unlock implementation`
- `affected_stages`
- `risk_level`
- `change_class`
- `change_domains`
  - 默认必须包含：`automation_control_core`
  - 若触及 `src/shared/capability_runtime.py`、`src/shared/policy_executor.py`、`src/shared/runtime_validator.py`、`src/shared/contracts_runtime.py`、`src/shared/pipeline.py`，还必须包含：`shared_runtime_core`
  - 若触及 `docs/D12`、`docs/D13`、`docs/D14`、`docs/技术实现决策页.md`、`docs/正式业务代码开发开工裁决页.md`、`docs/文档与资产状态板.md`、`docs/AX9S_开发执行路由图.md`、`control/repo_status.md`、`control/milestone_status.yaml`、`control/release_manifest.yaml`、`control/model_release_manifest.yaml`，还必须包含：`future_external_unlock_prereq_core`
- `human_review_required`
- `owner_reviews_required`
- `review_evidence`
- `required_scripts`
- `stop_conditions`
- `definition_of_done`
- `deliverables`

## 4. 修改范围模板

- `declared_changed_paths`
  - 声明本轮预计会落笔的真实路径
- `baseline_dirty_paths`
  - 可选但强烈建议填写
  - 写入“当前 packet 激活前工作区里已存在的 tracked dirty paths”
  - validator 会按：`actual_changed_paths - baseline_dirty_paths - generated_runtime_artifacts` 做范围校验
  - 不允许把本轮准备新增修改的真实目标路径伪装进 baseline
- `allowed_modification_paths`
- `forbidden_modification_paths`
- `impacted_assets.docs`
- `impacted_assets.control`
- `impacted_assets.contracts`
- `impacted_assets.scripts`
- `impacted_assets.tests`
- `impacted_assets.handoff`
- `impacted_assets.runtime`

补充约束：
- `control/current_task.yaml` 只允许用于当前 active packet 的 machine-readable 承接与必要的 execution semantics 同步。
- 路径声明要能覆盖实际修改，但不能放宽成与本轮主题无关的大范围白名单。

## 5. Review Gate 模板

- `human_review_required`
  - `MANDATORY_HUMAN_REVIEW` 默认必须为 `true`
- `owner_reviews_required`
  - 默认至少包含：`architecture_owner`、`governance_owner`、`testing_owner`
  - 若 `change_domains` 包含 `automation_control_core`，还必须包含：`automation_owner`
  - 若 `change_domains` 包含 `governance_release_core`、`future_external_unlock_prereq_core`、`future_external_unlock_decision`，还必须包含：`release_approver`
  - 若 `change_domains` 包含 `provider_vendor_source_policy_core`、`future_external_unlock_prereq_core`、`future_external_unlock_decision`，还必须包含：`model_governance_owner`
- `review_evidence.declared`
- `review_evidence.signoff_required`
- `review_evidence.signoff_status`
  - `MANDATORY_HUMAN_REVIEW` 默认必须为：`declared=true`、`signoff_required=true`、`signoff_status=REQUESTED_NOT_APPROVED`

## 6. 脚本与停机模板

- `required_scripts`
- `script_timeouts`
  - 可选但建议填写
  - `scripts/check-release.ps1` 默认建议 `300000` ms
  - 工具层 120 秒超时但脚本未返回 FAIL/非零时，不得直接判定为脚本失败；必须按本字段更长超时重跑一次
- `stop_conditions`
- 连续失败阈值：同一脚本 `>= 2` 次停机
- 若任务包修改 `scripts/**`，必须同时补对应 tests，避免 validator / release gate 继续依赖过期 token

## 7. 完成标准模板

- `definition_of_done`
  - 写清本轮结束时必须已经成立的状态
- `deliverables`
  - 写清本轮交付物，而不是泛泛写“已完成”

## 8. 提交前收尾提醒

- 不得混入 `报告*.md`
- 不得混入 `__pycache__/`、`*.pyc`、`.pytest_cache/`
- 提交前应核对 `actual changed paths` 是否全部落在 `declared_changed_paths` / `allowed_modification_paths` 内
- 若工作区存在不属于当前包的脏改，先停机汇报，不要混入当前提交

## 9. 输出格式

- 修改文件清单
- 每个文件修改了什么
- 脚本结果
- 当前阻断项
- 是否建议进入下一步
