# 文档方向同步与合并治理计划

## 1. 结论

需要同步，但不应一次性重写 `docs/` 全部文件。

原因：

- `docs/` 里有正式 D1-D14、L0、裁决总表、专题草案、路线图、状态板和历史任务契约，权威等级不同。
- 新增的双线业务方向已经写入 `AGENTS.md`、`README.md`、`docs/业务方向_候选公示后证据包与投前预测双线契约.md` 和 `contracts/evaluation/business_direction_strategy_contract.json`。
- 现有部分文档仍把“公开市场机会发现 + 证据包商业化”写得较宽，容易让后续开发继续把“招标公告文件链路 smoke”当成业务主线。
- `专题_投前预测...` 文档很大，包含大量能力候选，不应删除，但必须明确它是辅助产品线和规则候选池，不是核心商业主线本身。

正确处理方式：

1. 先固定文档权威层级。
2. 再给每类文档打同步优先级。
3. 只修会误导开发入口、运行命令、验收口径的段落。
4. 重叠能力只保留一个主入口，其他文档改成引用，不复制整套口径。
5. 大文档不做无意义全文替换，按业务影响面小批次修。

## 4. 重叠功能合并原则

### 4.1 不能合并成一个大文档

不建议把 `专题_来源覆盖...`、`专题_投前预测...`、D8、D11 全部合并成一个文档。它们负责不同层：

- 来源路由：负责哪里抓、怎么归档、来源状态。
- 投前经验库：负责投前预测和规则候选。
- D8：负责真实竞争者、买家适配、可售对象。
- D11：负责验收和回归。
- 双线契约：负责业务方向和默认路由。

正确做法是“单源主口径 + 多文档引用”：

- 业务方向只在 `业务方向_候选公示后证据包与投前预测双线契约.md` 定义。
- 机器路由只在 `business_direction_strategy_contract.json` 定义。
- 其他文档只引用，不复制。

### 4.2 重叠能力归属

| 能力 | 主归属 | 其他文档如何处理 |
|---|---|---|
| 候选公示后证据包分析 | 双线方向契约 + D8 | 来源规范负责采集路由，D11负责验收 |
| 投前预测分析 | 投前专题 + 双线方向契约 | 不得替代证据包主线 |
| 控标/定制标线索 | Stage5/规则 seed + 投前专题 | 后验项目可作为受益链和证据包线索 |
| 围标/串标/陪标线索 | 候选公示后证据包线 + D8 | 投前只能做弱线索或风险提示 |
| 真实竞争者识别 | D8 | 不能在投前线形成结论 |
| 文件下载/解析 clean-v1 | professional clean 契约 | 只证明文件链路，不代表商业主线完成 |

## 5. 建议实施顺序

### 第 1 批：轻量同步，防止开发跑偏

范围：

- `docs/文档与资产状态板.md`
- `docs/专题_来源覆盖与采集路由规范.md`
- `docs/专题_投前预测评审风控规则救济与履约结算经验库草案.md`
- `docs/AX9S_实战运行图纸与验收契约.md`

动作：

- 增加双线契约引用。
- 明确候选公示后证据包为核心商业主线。
- 明确 tender_file smoke 是投前/文件链路验证，不是最终业务入口。
- 明确 YGP 继续隔离。

### 第 2 批：验收和销售承接同步

范围：

- `docs/D8_真实竞争者识别可售对象与销售推进规范.md`
- `docs/D11_测试验收与金标回归清单.md`
- `contracts/testing/*` 如后续需要。

动作：

- D8 绑定候选公示后全流程回溯。
- D11 增加业务方向契约测试和 post-candidate backtrace 验收。

### 第 3 批：正式对象和规则同步

范围：

- `docs/D2_正式对象契约与字段字典.md`
- `docs/D3_正式规则码总表与判定说明书.md`
- `docs/D13_公开可查边界能力清单.md`
- 对应 contracts/schema/rules/enums。

动作：

- 等 `POST_CANDIDATE_BACKTRACE_V1` 代码入口确定后，再补正式对象和规则字段。
- 避免先写一堆对象，最后代码没有消费。

## 6. 暂不建议做的事

- 不建议删除 D1-D14 或 L0。
- 不建议把大文档全部合并成一份。
- 不建议现在重写 `专题_投前预测...` 全文；它是能力池，应该降权定位，不是删除。
- 不建议现在把所有旧路线图改成“已完成”；当前更重要的是绑定新方向和运行验收。
- 不建议把广州 tender_file 下载成功写成商业主线完成；它只是广州文件链路能力跑通。

## 7. 当前最优下一步

先执行第 1 批轻量同步。

完成后再做代码层下一步：

1. 新增或改造 `POST_CANDIDATE_BACKTRACE_V1` 运行入口。
2. 广州候选公示 3-5 个项目入池。
3. 自动回溯招标公告、招标文件、答疑澄清、开标、候选、结果、投标文件公开。
4. 每个项目输出项目级全流程审计表。
5. 再接控标/围标/串标/陪标和真实竞争者判断。
