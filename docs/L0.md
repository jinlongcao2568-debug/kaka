## 平台定位修订

- 当前目标不是对外售卖软件、客户可见工作台或开放平台能力，而是建设**内部线索运营平台 / 情报生产平台 / 销售作战平台**。
- 对外交付只允许以**线索包 / 机会包 / 情报包 / 销售推进结果**形态输出，不交付客户可见软件平台本体。
- 平台对外交付的正式对象仅限：线索包、机会包、情报包、推进结果（均受门禁与审计约束）。
- 统一放行层级以 D13 为准：`DEV_ALLOWED`（研发 / 设计 / 验证阶段可用）、`INTERNAL_OPERABLE`（内部运营与销售可用）、`LEADPACK_DELIVERABLE`（线索包可对外交付；必须满足门禁与审计）、`EXTERNAL_CONTROLLED_OPENING`（对外阻断）。
- “全国覆盖 / 成熟度”只允许作为**内部 coverage / maturity 地图**、**内部资源优先级**与**自动化程度提示**使用，不得作为对外统一产品承诺。
- 新增线索包质量门定义：
  - **可交销售线索包**：证据回链完整、双闸门通过、字段策略满足、审计链完整，满足外发门禁。
  - **待审线索包**：证据不足或覆盖不完整，需进入人工复核或审批链后才能升级。
  - **必须退回**：触达合规、证据或来源不合规；或违反交付矩阵与字段策略。
  - **证据不足不得入包**：未形成可审计证据链的线索不得进入外发包。
- 新增销售使用门总纲定义：仅当线索包质量门通过、触达合规与交付矩阵未阻断、审批链与审计链可成立时，销售使用才允许成立。
- 新增结果回写与误判回收总纲定义：触达、交付、未成交与误判必须形成正式回写事件并进入治理反馈与复核回收，用于阶段 6-7 的持续校正。
- 真相主链、双闸门、统一事实、审计链与合规门禁保持不变，任何对外交付必须受其约束。

## 基线摘要

- 本文固定  建设工程域的唯一正式主基线，并把**公开可查、可回链、可审计**确立为领域主链、规则命中、统一事实、可售判断与外发治理的总边界。
-  的正式价值不是“自动替代监管、评标或司法作终局裁决”，而是在公开边界下形成**核心候选集合攻击面、排序 / 不排序候选关系、伪竞争者剔除、真实竞争者识别、法律动作建议与商业转化编排**。暗箱 / 内定型异常默认只能进入 CLUE 或 review_request，不得越界写成终局违法定性。
- 全闭环正式主链固定为：任务编排与时钟管理、公开链抓取、结构化解析与关键对象抽取、关键对象定向公开核验与冲突预判、规则与证据、人工复核、统一事实、真实竞争者识别与正式报告、商业承接与可售对象、联系对象与销售触达、订单支付交付与治理反馈；其中“人工复核、统一事实、真实竞争者识别与正式报告”属于同一正式阶段，不得拆成并列新阶段。
- 真实竞争者识别属于  的核心正式能力，不属于可后置的销售附加项；其研发优先级可以前置进入首批可开工基线，但运行时序不得早于阶段 5 双闸门与阶段 6 统一事实形成。
- 页面、接口、交付、机器资产与后续配套文档均只能细化本文，不得绕开本文形成第二套主裁决。
- V3.1 的直接目标不是扩写 D1-D14 正文，而是把 L0 升级为可稳定派生下位文档、接口契约、测试断言与机器资产的上位母版。

## 0. 规范语言与阅读规则

本文档为  建设工程域在中国公开证据边界下的全闭环最高约束性权威总文档。本文档的任务不是罗列所有实现细节，而是裁决：系统是什么、哪些对象是正式对象、哪些结果是正式结果、哪些页面与接口是正式消费面、哪些事项只能进入复核或治理链、以及 1-9 阶段如何在公开边界下闭环成立。

本文档必须能够回答以下问题：第一， 为什么既不是单纯异议工具，也不是脱离证据的销售平台；第二，公开证据边界下哪些能力可以自动成立、哪些只能形成线索或请求核验；第三，统一项目事实、真实竞争者、可售对象、联系对象、订单交付与治理反馈如何串成闭环；第四，后续所有配套文档、机器可读资产与实现发生冲突时，应当以什么优先级裁决。

### 0.1 规范语言

- **必须 / 不得 / 只允许**：硬约束。产品、架构、研发、测试、页面、接口、导出、销售、交付、治理与 AI 执行代理均必须遵守。
- **应 / 不应**：强推荐。除非存在受控例外，否则应按本文执行。
- **可 / 允许**：可选能力。仅在不突破本文主边界与主路线时允许采用。

### 0.2 本文档直接规定的内容

- 全闭环产品定位与总路线裁决。
- 九阶段正式输入、处理、输出、窗口期与承接定义。
- 正式结果语义、公开边界能力矩阵、正式对象清单与核心聚合字段。
- 3 个正式 SKU、真实竞争者、可售对象与联系对象的最低成立规则。
- 页面 / API / 导出 / 外发的主消费原则。
- 治理、交付、发布门禁、审计留痕与人工覆盖边界。
- 后续文档体系与机器可读资产骨架的总索引。

### 0.3 本文档不直接展开的内容

- 仓库目录、分支、PR、worktree、CI 流程细节。
- 全量规则码逐条算法说明与测试样本详情。
- 全量字段策略字典与对象级交付矩阵的机器可读细表。
- 所有公开源适配器、抓取脚本、调度脚本、部署命令与监控告警实现。
- 价格体系、报价策略、合同条款与销售话术模板的具体商业文本。

上述细节统一由后续执行手册、规则码总表、对象契约、字段策略字典、交付矩阵、可售对象规范、触达规范、订单交付规范、测试验收清单与机器可读资产承接，但不得突破本文定义的总路线、正式对象和公开边界。

### 0.5 L0 与 D1-D14 的单向约束关系

## 2. 产品定位、边界与总路线裁决

### 2.1 正式产品定位

-  是基于公开证据的建设工程域全闭环系统，不是单纯异议包生成器，也不是脱离证据真相层的销售平台。
-  的核心职责是：发现公开链、形成结构化对象、围绕关键对象执行定向公开核验、运行规则、固化证据、组织人工复核、形成统一项目事实、识别真实竞争者与可售对象、组织销售触达，并把订单、交付与治理反馈回写系统。
-  对外提供的是公开证据核查、风险线索发现、证据回链、人工复核、异议辅助、项目推进判断、可售对象编排、客户交付与治理支持；它不是行政处罚系统、司法裁判系统或评标委员会替代系统。
-  当前正式主基线以建设工程招标投标程序为主。若项目适用政府采购质疑 / 投诉制度或其他专门救济程序，必须切换对应制度时钟、主管机关与动作口径，不得直接沿用建设工程招标投标异议 / 投诉 / 递补语义。
- 本文中的“第一名 / 前三候选人 / 递补收益”，仅适用于存在排序推荐且地方规则允许按顺位解释的候选人模式；对不排序推荐、评定分离或其他候选集合模式，应统一解释为核心候选集合、定标结果关系与行动受损关系，不得为了套用规则而伪造顺位。

### 2.2 产品总路线裁决

 的正式路线裁决为：

1. **公开证据链必须成立**：阶段 2-5 必须在公开边界下真实打通。
2. **统一事实中枢必须成立**：阶段 6 必须是唯一统一事实中枢与法律动作建议中枢。
3. **商业闭环必须成立**：阶段 7-9 必须真实消费阶段 6 的结果，而不是绕开它各自成立。
4. **治理门禁必须成立**：coverage、field policy、delivery audit、release gates、公开边界门禁必须形成正式门禁链。

任何只做前五阶段的实现，只能算半闭环；任何绕开证据真相层直接做销售触达的实现，只能算漂移系统；任何把非公开数据静默混入主模型的实现，都不得视为正式版本。

### 2.3 中国公开证据边界硬约束

- 自动核查只允许使用公开、可检索、可回链、可审计来源的证据。
- 招投标平台后台日志、登录轨迹、上传 / 下载 IP、设备指纹、预算软件私有比对结果、非公开社保、非公开人员档案、非公开合同附件、客户内部底稿、私下获取材料，均不得作为自动命中的直接输入。
- 上述材料只允许进入 review_request、人工复核、交付治理或客户补证链，不得直接形成 AUTO_HIT。
- 客户自行提供的内部材料，不得静默覆盖阶段 3 的原始真相层，只允许作为阶段 5、阶段 9 或治理链补证。

### 2.4 法律动作导向硬约束

 的正式输出不得只停留在“发现异常”。系统必须同时回答：

- 当前仍可采取的正式动作是什么；
- 该动作的公开依据是什么；
- 该动作是否受时钟、窗口期、证据等级、交付边界或可售条件阻断。

正式动作建议至少包括：招标文件异议、开标异议、评标结果异议、质疑建议、投诉建议、建议递补、建议重新招标、建议保持观察与继续补证。

- 对存在中标候选人公示的依法必须招标项目，中标候选人公示期内异议是第一层正式动作，不是补充动作；招标人答复前应视制度要求暂停后续招投标活动。
- 法律动作建议必须先识别 `procurement_regime`、`candidate_order_mode` 与主管机关边界；只有动作家族可以作为全国统一正式能力，具体动作实例化不得脱离适用制度。
- 当制度类型、候选推荐模式或主管机关边界尚未确认时，只允许输出 review_request、观察结论或补证动作，不得把工程招投标与政府采购的救济口径混写为同一正式建议。
- legal_action_recommendation 必须显式记录 `competent_authority_scope` 或等价正式字段，用于说明当前建议动作所依附的主管机关边界、受理落点或制度作用域；不得把“适用制度已识别”替代“动作将向谁发起或由谁受理”的正式表达。

### 2.5 正式公开能力边界裁决

本文将公开能力按以下四层裁决，而不再沿用“凡法规中存在即可视为市场侧公开能力”的宽口径。

**A 层：全国稳定公开可得主能力**

允许直接进入正式主链，且可作为全国最小公开能力承诺的内容包括：

- 公开链完整性核验
- 中标候选人公示期异议时钟与动作门控
- 评标方法与项目类型分型
- 候选集合攻击面核验（排序推荐模式下包含第一名 / 前三候选人攻击面）
- 公开版本冲突检测（联合体成员、资格条件、项目负责人、关键字段前后不一致）
- 限制性条款与规则定向识别
- 全国统一公开信用链核验（中国执行信息公开网、信用中国等）
- 真实竞争者识别
- 法律动作建议对象化
- 公开证据等级治理

**B 层：工程建设域公开增强能力**

允许进入正式主链，但必须受地区覆盖、平台成熟度、数据等级与治理状态约束，不得写成全国统一最小能力。典型包括：

- 建筑市场监管公共服务平台及地方住建公开平台中的项目业绩信息
- 建筑市场监管公共服务平台及省级一体化平台中的项目经理注册信息、在建工程、企业业绩与项目负责人业绩
- 施工许可、竣工验收备案等工程建设域增强字段
- 合同订立信息、部分地区已稳定公开的合同履行与变更信息
- 工程项目公开处罚、信用、履约类增强字段
- 国家企业信用信息公示系统中的单位负责人、股东及控股 / 管理关系核验

**C 层：条件性公开线索能力**

仅在当地平台、项目公示或公开页面颗粒度足够时才允许启用；默认只能形成 CLUE、OBSERVATION 或 review_request，不得被视为全国稳定主能力。典型包括：

- 公开附件内容核验
- 评分细项异常、主观分离群、评审明细异常
- 完整项目管理班子比对
- 履约异常、合同变更异常、后评估异常等条件性公开线索

**D 层：非公开 / 监管 / 取证侧能力**

下列能力不得纳入正式公开主模型：

- 交易平台后台日志、登录轨迹、IP、设备指纹、私有 XML
- 投标保证金同源转出等保证金系统内部特征
- 同一单位或个人编制投标文件、办理投标事宜、投标文件相互混装等取证侧特征
- 非公开社保、人事档案、内部合同、内部履约材料
- 非法或灰色获取的联系人信息

### 2.6 对外承诺受控开放边界

- 不得承诺自动认定围标串标。
- 不得承诺自动判断中标无效或自动替代监管、评标或司法结论。
- 不得承诺全国全地区、全专题、全链路同等成熟覆盖。
- 不得把线索型专题包装成已成立违法定性。
- 不得把“明招暗定 / 暗箱操作”作为单条自动规则结论对外输出。

### 2.7 正式目标客户与购买角色

|---|---|---|
| 投标 / 跟标侧 | 投标负责人、跟标团队、法务 | 在有效窗口期内发现硬伤、异常与可用证据，形成异议辅助或项目研判 |

3. 商业封装链
4. 触达编排链
5. 商务交付与反馈链

**经营主链与阶段映射关系**

| 阶段范围 | 映射到的经营主链 | 备注 |
|---|---|---|
| 阶段 1-6 | 高价值线索发现链 + 正式依据基础 | 形成公开证据、统一事实与可售前提 |
| 阶段 7 | 证据包与可售判断链 + 商业封装链 | 形成可售判断与商业封装 |

| 阶段 9 | 商务交付与反馈链 | 形成交付与治理回写闭环 |

### 3.2 三个正式 SKU

| SKU | 正式名称 | 核心价值 | 主要消费对象 | 默认交付形态 |
|---|---|---|---|---|
| SKU-A | 资格与程序打击包 | 聚焦项目负责人资格、程序公开充分性、无效标与资格后审，形成快速可用的证据与异议辅助 | 投标 / 跟标 / 法务 / 顾问 | 项目核查简报、证据包、异议辅助草稿 |
| SKU-B | 评审与竞争异常包 | 聚焦评分异常、报价模式、真实竞争者识别、评审可见异常，形成强化研判与补强火力 | 投标 / 顾问 / 经营 / 风控 | 项目研判报告、竞争格局摘要、评分异常证据集 |
| SKU-C | 组织行为与履约风险包 | 聚焦限制性资格条件、参数定向、代理行为风险、疑似拆包与中标后异常变更 | 经营 / 风控 / 顾问 / 交付 | 专题风险报告、履约风险追踪、组织行为风险摘要 |

### 3.2A LeadPack 商业封装层（A/B/C/D）

- `LeadPack SKU A/B/C/D` 仅用于线索包商业封装层（包装、交付形态、报价带与分层呈现），不等于能力边界层。
- D13 的 A/B/C/D 仍是正式能力边界分层，不得被 `LeadPack SKU A/B/C/D` 替代或映射。

### 3.3 SKU 与专题关系裁决

SKU 是对外售卖与交付组织方式，不等于底层只存在三类规则。底层可以保留更丰富的专题、对象与画像对象，但首卖与交付必须通过正式 SKU 输出统一产品形态，不得让客户直接面对无组织的专题堆叠。

### 3.4 覆盖成熟度与可售边界

- 地区、专题、源族的正式覆盖状态只允许通过 coverage_governance_registry 表达，不得并行维护第二套对外正式状态。
- `coverage_sellable_state` 是唯一正式可售状态，且只允许使用 `NOT_READY / VALIDATING / SELLABLE / RESTRICTED / SUSPENDED / RECOVERING` 六状态。
- “已实现 / 已验证 / 已可售 / 已可交付”只允许作为内部研发、验证或交付准备属性存在，不得替代、映射偷换或并行覆盖正式可售状态。
- 未进入 coverage registry 的地区或专题，不得被阶段 7 视为可售对象来源。
- 任何宣称可售的地区 / 专题 / 源族，都必须证明其核心能力位于公开边界内可成立。

### 3.5 SKU 承诺边界与禁止表达

| SKU | 允许承诺 | 禁止承诺 | 必须附带限定语 |
|---|---|---|---|
| SKU-A | 资格、程序、公开链与异议辅助能力 | 自动废标、自动认定违法 | 公开边界、人工复核辅助、证据回链成立 |
| SKU-B | 竞争格局、评分异常、真实竞争者识别与研判能力 | 自动认定围标串标、自动判断中标无效 | 受证据等级、coverage 与双闸门约束 |
| SKU-C | 履约、组织行为与风险研判能力 | 仅作为内部 coverage / maturity 地图与交付准备参考，不对外承诺全国同成熟度 | 受地区覆盖、公开颗粒度与交付矩阵约束 |

## 4. 九阶段正式定义与时钟驱动流水线

 建设工程域必须以九阶段闭环方式成立。阶段成立的最低标准统一为：输入存在、处理真实发生、输出真实形成、输出已被下一阶段消费；并且所有阶段都必须受窗口期和公开边界约束。

| 阶段 | 主要输入 | 主要输出 | 正式承接 | 硬约束 |
|---|---|---|---|---|
| 阶段 1 任务编排与时钟管理 | 启动指令、地区、时间范围、策略模板、窗口期规则 | 任务、项目归一策略、复核承接车道、执行上下文、动作优先级 | 阶段 2 | 必须先判断是否仍在有效动作窗口，不得直接形成业务结论 |
| 阶段 2 公开链抓取 | 任务、公开源入口、项目线索 | 原始页面、附件、抓取诊断、public_chain 初步节点、clock_chain_profile、notice_version_chain | 阶段 3 | 只负责发现与固化，不负责最终业务归一 |
| 阶段 3 结构化解析与关键对象抽取 | 原始页面、附件、抓取 hint | project_base、field_lineage_record、候选人、项目经理、业绩、资格能力、profile 对象 | 阶段 4 | 是正式结构化真相层，不得被第二套画像替代 |
| 阶段 4 关键对象定向公开核验与冲突预判 | 阶段 3 对象、公开源补抓策略、窗口期状态 | public_attack_surface、focus_bidder_verification_profile、qualification_gap_clue、pm_conflict_clue、pseudo_competitor_signal_set、evidence_grade_profile | 阶段 5 与阶段 6 | 只允许围绕公开可查对象做二次核验，不得混入非公开资料 |
| 阶段 5 规则与证据 | 阶段 3-4 对象、规则配置、证据素材 | rule_hit、evidence、rule_gate_decision、evidence_gate_decision、review_request | 阶段 6 | 只允许基于公开可回链证据形成 AUTO_HIT |
| 阶段 6 人工复核、统一事实、真实竞争者识别与正式报告 | 前五阶段结果、治理状态、人工复核结果 | project_fact、sale_gate_status、legal_action_recommendation、challenger_candidate_profile、review_queue_profile、report_record | 阶段 7-9 + 页面/API | 唯一统一事实中枢；正式报告必须消费 project_fact |
| 阶段 7 商业承接与可售对象 | project_fact、challenger_candidate_profile、SKU、buyer fit 规则 | sales_lead、challenger_buyer_fit、legal_action_actor_profile、procurement_decision_actor_profile、saleable_opportunity、offer_recommendation、account_context | 阶段 8 与阶段 9 | 不得直接消费底层原始对象重算主价值 |
| 阶段 8 联系对象与销售触达 | saleable_opportunity、contact_target、outreach_plan | touch_record、客户反馈、推进状态 | 回写阶段 6，必要时进入阶段 9 | 只允许使用公开、授权或合规接入的联系来源 |
| 阶段 9 订单支付交付与治理反馈 | opportunity、订单、支付、交付状态 | order_record、payment_record、delivery_record、opportunity_outcome_event、governance_feedback_event | 回写阶段 6，必要时触发阶段 5 补证 | 交付与治理不能反向篡改阶段 3 真相层 |

### 4.1 阶段 1：任务编排与时钟管理

阶段 1 是启动控制面。其职责是定义系统何时、何地、按什么规则去发现项目、重跑项目、刷新项目、追踪履约，并把制度类型、候选人推荐模式、中标候选人公示期、异议窗口、投诉窗口、结果公示窗口、履约追踪窗口纳入正式调度输入。阶段 1 必须支持按地区、时间窗、专题、SKU 目标、回查周期、重跑策略与动作时效优先级进行编排，但不得直接形成业务结论。
阶段 1 还必须为每个任务声明项目归一策略与复核承接车道。对于同一项目的公告、更正公告、补遗、候选人公示、结果公示、标段拆分、重招链与联合体版本变化，不得在进入阶段 2 后再由页面、销售脚本或人工备注临时归并，而必须在正式调度输入中明确 project_root、notice_version、lot、retender chain 的归一原则。对于窗口期极短的项目，阶段 1 必须支持标准复核与窗口快速复核两类承接车道；快速复核只能改变阶段 6 的 SLA 与最小输出级别，不得绕开阶段 6 统一事实中枢。

### 4.2 阶段 2：公开链抓取

阶段 2 必须把公开公告、依法公开的公示、候选人公示、结果公示、公开开标信息、公开处罚信息等固化为可审计原始载体，并形成 public_chain 初步节点。public_chain 不是简单链接集合，而是围绕法定公开节点、节点缺失、节点时序与节点合规性形成的正式判断对象。对于附件、履约、合同变更、评分明细等内容，只有在地区平台或项目页面确有稳定公开时，才允许作为 B 层增强能力或 C 层条件性能力进入抓取，不得默认承诺全国稳定可得。
阶段 2 还必须同时固化多时钟链与多版本链。至少应并行记录：法定发布时间钟、首次见证时钟、更正 / 补遗时钟、答复时钟、救济时钟；并且必须记录公告版本序列、版本替代关系、首见时间、末次取回时间与多源冲突裁决状态。未形成正式并行时钟上下文或正式版本链的项目，不得在后续阶段被简化为单一窗口或单一版本事实。

### 4.3 阶段 3：结构化解析与关键对象抽取

阶段 3 是工程域唯一结构化真相层。所有正式业务对象与画像对象都必须在此阶段落成并可回链到公开来源。阶段 3 不仅要形成 project_base，还必须明确抽出核心候选集合、候选人推荐模式、排序推荐模式下的第一名 / 前三候选人、项目负责人、资格能力承诺、公开业绩、公开处罚、公开信用与窗口期相关对象。
阶段 3 对关键字段不得只做到对象级可追溯，而必须做到字段级可追溯。凡被阶段 4-9 消费的关键字段，至少必须能够回答：字段来源于哪一份公开载体、哪一个版本、哪一个切片、采用了哪条归一 / 规范化规则、当前解析置信度是多少、是否存在多源冲突未解。未形成字段级来源切片与置信说明的关键字段，只允许停留在解析候选或 review_request，不得直接抬升为正式事实依据。

### 4.4 阶段 4：关键对象定向公开核验与冲突预判

阶段 4 是正式前置增强层。其职责不是直接作结论，而是围绕核心候选集合、排序推荐模式下的第一名 / 前三候选人、项目经理、项目业绩、资格能力、公开信用处罚、限制性条款、异常低价 / 严重不平衡报价与窗口期进行二次公开核验，并形成攻击面对象、目标候选人核验画像、伪竞争者信号集、异常低价画像与证据等级画像。

阶段 4 必须同时遵守以下分层规则：

- A 层能力可直接作为主核验输入；
- B 层能力只有在已验证地区、已验证平台和已声明覆盖的工程建设域场景下才允许启用；
- C 层能力只有在公开颗粒度足够时才允许启用，且默认以线索化表达；
- D 层能力不得进入阶段 4 正式主链。

因此，阶段 4 不得把公开附件、评分细项、完整项目管理成员、履约异常或合同变更异常写成全国默认前置核验项；也不得把保证金同源、同人编标、同人办标、文件混装等取证侧特征伪装成公开核验能力。

阶段 4 对目标候选人的上位核验边界至少包括：违法违规风险、资质 / 业绩 / 人员资格 / 信用 / 财务状况 / 生产条件是否符合中标条件、经营或财务状况重大变化是否影响履约、履约保证金是否按招标文件要求提交。其中，凡可通过公开平台、公开公示、公开信用链、公开证书链、公开业绩链或公开主体信息完成回链的部分，允许进入正式公开核验链；其余只能降级为 review_request 或客户补证事项。履约保证金提交情况默认不属于全国稳定公开最小集合，通常只允许作为 B/C 层条件性能力或 review_request，不得被误写为全国 A 层常规核验项。

### 4.5 阶段 5：规则与证据

阶段 5 负责在公开证据边界内运行规则、固化证据并生成 review_request。任何内部权限依赖事项都必须降级为 review_request，不得写成 AUTO_HIT。阶段 5 输出必须同时进入阶段 6 的统一事实中枢。
阶段 5 必须实行规则命中闸门与证据可见性闸门双闸门裁决。规则命中闸门只回答“规则是否命中、命中了哪些前提、命中是否被版本冲突或制度不确定性阻断”；证据可见性闸门只回答“证据是否达到客户可见、外发可用、正式建议可用的最低阈值”。任一闸门未通过时，结果只允许停留在 CLUE、OBSERVATION 或 review_request，不得借由另一闸门单独通过而升级为 AUTO_HIT、正式客户版主结论或正式法律动作建议。
- 涉及项目经理社保一致性的能力，公开一致性部分只允许形成 CLUE；凡依赖非公开社保缴纳明细、内部人事材料或其他 D 层输入的部分，一律只能形成 review_request，不得直接升级为 AUTO_HIT、正式客户版主结论或当然废标结论。

### 4.6 阶段 6：人工复核、统一事实、真实竞争者识别与正式报告

阶段 6 是  的唯一统一事实中枢。它不仅聚合核查结果，还必须形成统一项目事实、可售门状态、交付风险状态、覆盖治理状态、法律动作建议，并完成真实竞争者识别。所谓真实竞争者，是指在公开证据边界内，对目标候选结果具有较强依法挑战收益、明确受损关系、较高组织化行动可能与较高窗口紧迫度的真实受损者。页面、接口、销售话术、外发模板与经营判断都必须优先消费 project_fact 与正式挑战者画像，而不是再次穿透到底层原始对象。
阶段 6 必须支持标准复核与窗口快速复核双车道。窗口快速复核仅适用于临近异议、投诉、递补、重新招标建议或其他关键制度动作窗口的项目，其最低输出至少包括：统一事实摘要、主要攻击点、窗口状态、最低证据等级、legal_action_recommendation 与是否允许进入阶段 7 受限对象化。快速复核可以先形成最低放行版 report_record，但不得跳过人工确认、不得绕开 project_fact，也不得在证据未达阈值时直接放行正式外发。
真实竞争者识别属于  的核心正式能力，而不是可后置的销售增强项；其研发优先级可以前置进入首批可开工基线，但运行时序不得早于阶段 5 双闸门与 project_fact 统一事实形成。

### 4.7 阶段 7：商业承接与可售对象

阶段 7 的职责是把统一事实与阶段 6 已识别出的真实竞争者候选人转化为正式商业对象，使系统从“发现了价值”进入“知道最应该卖给谁、卖什么、优先跟谁”。阶段 7 不负责重新判断谁最值得卖，它只负责将阶段 6 已确认的候选人正式对象化。
对于竞争驱动型机会，阶段 7 必须正式区分两类 actor：第一，程序上最可能依法发起异议、投诉、复核、递补请求或其他正式动作的法律行动主体；第二，组织内最可能批准采购、推动签约或承担付款的采购决策主体。两类 actor 可以重合，但不得被默认视为同一对象；未完成该拆分时，只允许停留在一般线索或受限机会，不得进入竞争驱动型正式销售编排。

### 4.8 阶段 8：联系对象与销售触达

阶段 8 的职责是把可售对象推进到真实客户互动。阶段 8 不是重新做判断，而是基于正式对象选择联系对象、生成触达计划、记录反馈与推进状态。阶段 8 只允许使用公开可审计的联系来源、客户主动提供的授权联系方式，或企业合规接入的 CRM 数据。
为避免正式链路因下位触达规范缺失而失控，阶段 8 在总纲层至少必须完成以下最小门槛校验：`channel_family`、`contact_legal_basis`、`frequency_policy_state`、`opt_out_state`、`quiet_hours_policy_state`。任一校验项缺失时，不得执行正式触达。D9 可以继续细化具体渠道、频控与同意规则，但不得低于本总纲最小门槛。
正式链路默认优先选择组织联系方式与组织角色对象；自然人联系方式属于高限制触达对象，默认 `requires_manual_review = true`，除非 D9 已明确声明该场景可自动放行且当前对象满足相应渠道规则。

### 4.9 阶段 9：订单支付交付与治理反馈

阶段 9 的职责是把商业承接结果转化为收入与交付，并把交付治理结果反写回系统。阶段 9 必须形成 order_record、payment_record、delivery_record 与 governance_feedback_event。必要时，阶段 9 应触发阶段 5 补证、阶段 6 统一事实刷新或外发阻断升级。
阶段 9 除记录成交、支付与交付外，还必须正式沉淀未成交、窗口失效、误判、找错角色、联系失败、客户拒绝、交付放弃等负结果回写事件，以支撑阶段 6-7 的后续校正。该类负结果不得仅停留在运行备注、销售私有笔记或临时聊天记录中。

### 4.10 阶段进入条件矩阵

| 阶段 | 最低进入条件 | 若条件不满足的正式处理方式 |
|---|---|---|
| 阶段 1 | 已声明地区、时间窗、制度口径、动作优先级与项目归一策略 | 不得启动正式任务 |

- public_chain 是法定公开节点判断对象，不是简单 URL 集合。它必须能够回答：当前制度下哪些节点应公开、哪些节点已取得、哪些节点缺失、缺失是否影响动作、以及节点时序是否成立。
- 当法定关键节点缺失、错序、回链失效或制度要求的公开节点未被满足时，相关判断不得直接升级为 AUTO_HIT 或正式法律动作建议。
- 目标候选人的正式公开核验路径族至少包括：全国建筑市场监管公共服务平台及地方一体化平台的注册 / 在建 / 业绩信息链，信用中国与中国执行信息公开网的全国公开信用链，以及国家企业信用信息公示系统的主体关系链。凡未进入上述或等价正式公开路径的核验结果，不得冒充公开主链事实。
- 同一项目的公告、更正、补遗、候选人公示、结果公示、标段、重招链与联合体版本变化，不得靠页面、销售话术或临时脚本在阶段 6 之后再做主归一；必须在阶段 1-3 形成正式的 project_root、notice_version、lot、retender chain 与 consortium snapshot 关系，并在统一事实中保持可追溯。
- 字段级关键事实必须有 field_lineage_record；缺失来源切片、来源版本、解析置信度或归一规则 ID 的字段，不得被 project_fact、public_attack_surface、legal_action_recommendation 或正式客户版报告消费。
- private_supplement_record 只允许服务于 review_request、交付治理、客户补证、阶段 9 结果回写与合规留痕，不得直接写入阶段 3 公开真相层，也不得在未声明 lawful_basis 与 usable_scope 前被 project_fact、public_attack_surface 或 legal_action_recommendation 消费。

### 5.1 project_fact 硬约束

- project_fact 是阶段 6 唯一正式统一事实对象，必须可追溯到阶段 3-5 正式对象与治理侧正式资产。
- 页面层、接口层、经营层、销售层、导出层不得绕过 project_fact 重新拼装顶层主判断。
- manual_override_status 只允许覆盖结果层，不得篡改阶段 3 真相层。
- coverage_sellable_state、delivery_risk_state 只允许来源于正式治理资产或正式同步视图，不得由前端、接口或销售脚本临时推断。
- project_fact 的每次聚合都必须绑定清晰的项目归一上下文；若 project_root、notice_version、lot 或 retender chain 尚未完成正式归一，只允许停留在 review_request、OBSERVATION、受限 sales_lead 或受限 report_record，不得升级为正式 saleable_opportunity、正式客户版报告或正式法律动作建议。
- project_fact 的关键结论升级必须同时消费 rule_gate_decision 与 evidence_gate_decision；任一闸门未通过时，project_fact 不得把对应事项升级为 AUTO_HIT、正式客户版主结论或正式法律动作建议。
- legal_action_recommendation 必须与 rule_hit_summary、window_clock_state、evidence_grade_profile 保持可解释一致。
- 凡被 project_fact、public_attack_surface、legal_action_recommendation 消费的输入，都必须同时声明 `public_capability_tier`；任何 D 层输入不得被静默提升为 A/B/C 层事实依据。
- legal_action_recommendation、客户版证据包与外发异议辅助，不得仅以截图、OCR 文本或模型转述作为唯一主证；若缺少原始载体回链或稳定交叉印证，只允许停留在 CLUE、OBSERVATION 或 review_request。
- 任何正式外发主证、正式法律动作建议或正式客户版报告，必须同时满足 evidence_grade_profile 的来源可追、固定可核与必要时可调阅要求；任一条件不成立时，不得升级为正式客户可见主结论。

### 5.1.1 项目归一失败降级面

当 project_root、notice_version、lot、retender chain、多版本裁决或多时钟裁决尚未完成正式归一时，只允许按以下固定降级面处理：

| 输出或动作 | 是否允许 | 约束 |
|---|---|---|
| 内部观察项目 / OBSERVATION | 允许 | 必须显式标注归一未完成 |
| review_request | 允许 | 必须说明缺失的是哪一类归一条件 |
| 进入人工复核队列 | 允许 | 必须进入 review_queue_profile，不得绕过人工复核 |
| 受限 report_record / 内部 brief | 允许 | 仅限内部研判，不得作为正式客户版主报告 |
| sales_lead | 允许 | 仅限受限 lead，不得进入正式销售优先级池 |
| saleable_opportunity | 禁止 | 直至归一完成 |
| 正式客户版报告 | 禁止 | 直至归一完成 |
| formal legal_action_recommendation 外发放行 | 禁止 | 直至归一完成 |
| 正式触达 | 禁止 | 直至归一完成 |

### 5.2 真实竞争者识别硬约束

- challenger_candidate_profile 是阶段 6 的正式受损竞争者识别对象，用于识别“在公开证据边界内最可能依法行动、最可能直接受益的真实受损者”。
- 竞争驱动型机会不得绕开 challenger_candidate_profile 直接生成；未完成真实竞争者识别的项目，最多只允许停留在一般 sales_lead。
- 真实竞争者识别必须至少可解释以下维度：真实竞争者成立、受损程度、目标候选结果可攻击度、候选关系收益、执行就绪度、窗口期紧迫度。
- 阶段 6 负责主识别，阶段 7 只允许对象化阶段 6 已确认的候选人；阶段 8 不得绕开阶段 6-7 自行判断主要竞争者。
- pseudo_competitor_signal_set 命中高风险伪竞争者信号的对象，只允许作为保守过滤或复核输入；在信号未被解除前，不得直接晋升为 challenger_candidate_profile，更不得直接进入竞争驱动型 saleable_opportunity。

### 5.3 saleable_opportunity 硬约束

- saleable_opportunity 只能来源于阶段 6 的正式 project_fact，不得直接由单条 rule_hit、单页公告或销售手工备注生成。
- 一般 saleable_opportunity 必须同时绑定 buyer_fit 与 offer_recommendation；竞争驱动型 saleable_opportunity 还必须绑定 challenger_candidate_profile 或 challenger_buyer_fit。
- saleability_status 必须与 sale_gate_status、coverage_sellable_state、delivery_risk_state 保持可解释一致，不得出现明显冲突。
- 正式进入销售优先级池的 saleable_opportunity，必须补齐最小销售运营编排字段：`crm_owner_state`、`expected_close_days_band`、`expected_contract_value_band`、`expected_delivery_cost_band`；这些字段服务于经营编排，不得反向覆盖阶段 6 的统一事实、法律动作建议或公开证据结论。
- 对于竞争驱动型 saleable_opportunity，legal_action_actor_id 与 procurement_decision_actor_id 不得被默认视为同一对象；若尚未完成两类 actor 的正式拆分与解释，最多只允许停留在 challenger_buyer_fit、一般 sales_lead 或受限 opportunity。
- 对于异议打击型、竞争分析型、评审追击型机会，若 real_competitor_count < 1、competitor_quality_grade 低于 B 或 execution_readiness_score 未达到正式阈值，不得升级为正式 saleable_opportunity。

### 5.4 contact_target 与触达对象硬约束

- contact_target 只允许来源于公开可审计联系方式、客户主动提供的授权联系方式或企业合规接入的 CRM 数据。
- contact_target 不得通过非法采集、灰色渠道、不可审计外包名单或未经授权的私人信息形成。
- 触达动作必须绑定 outreach_plan 与 touch_record；没有触达记录的“联系成功”一律不成立。
- 正式触达放行必须同时满足 sale_gate_status 允许推进、contact_target 来源合规且 delivery_risk_state 不构成外发阻断；任一条件不成立时，只允许保留机会、补证或回写复核，不得执行正式触达。
- 正式触达前，至少必须完成 `channel_family`、`contact_legal_basis`、`frequency_policy_state`、`opt_out_state`、`quiet_hours_policy_state` 五项最小校验；任一校验缺失、未知或未放行时，不得执行正式触达。
- 正式链路默认优先组织联系方式与组织角色对象；自然人联系方式属于高限制触达对象，默认 requires_manual_review 为 true，除非 D9 已明确允许该场景自动放行且当前渠道策略全部通过。
- 对已经合法公开的自然人信息，只允许在与原公开场景、原公开目的和个人合理预期一致的范围内处理；超出合理范围或对个人权益有重大影响的，必须进入人工审查或另行取得合法基础。
- 不得仅凭自动化评分结果对自然人实施高强度触达、差别化报价或差别化交付承诺。
- 短信、电话、电子信息及其他触达渠道的具体放行条件、同意要求、拒绝方式与频控规则，一律以下位 D9 联系对象、触达策略与客户跟进规范为准；未经该规范放行的渠道，不得在正式链路中启用。

### 5.5 AI 与模型治理条件硬约束

- 仅当模型输出直接进入正式对象、正式报告、正式触达建议、正式外发材料或正式法律动作建议时，模型治理才被提升为主基线必选项。

- model_governance_record 必须至少说明：模型被用于什么范围、输入边界是什么、输出进入哪个正式对象、是否要求人工复核、以及当前发布状态是否允许正式使用。

### 5.7 状态机冻结原则

除非本文或 D2 / D3 另有更高位裁决，正式状态字段只允许按以下原则流转；未被允许的跨级跳转、静默回滚或无审计状态变更，一律视为违规实现。

| 状态字段 | 允许的主流转 | 禁止事项 |
|---|---|---|
| `coverage_sellable_state` | `NOT_READY -> VALIDATING -> SELLABLE / RESTRICTED / SUSPENDED`; `SELLABLE -> RESTRICTED / SUSPENDED`; `SUSPENDED -> RECOVERING -> VALIDATING / SELLABLE` | 不得 `NOT_READY -> SELLABLE` 直跳 |
| `sale_gate_status` | `REVIEW <-> HOLD`; `REVIEW / HOLD -> OPEN`; 任意状态在重大阻断出现时可进入 `BLOCK` | 不得在 coverage / delivery 阻断仍存在时进入 `OPEN` |
| `manual_override_status` | `NONE -> PENDING -> CONFIRMED / REJECTED` | 不得 `NONE -> CONFIRMED` 直跳；已确认覆盖不得无新审批静默复写 |
| `release_state` | `ISOLATED -> REVIEW_ELIGIBLE -> IMPACT_ELIGIBLE -> RELEASED`; 任一阶段可进入 `REJECTED` | 不得 `ISOLATED -> RELEASED` 直跳 |
| `exception_state` | `REQUESTED -> APPROVED -> EXPIRED / REVOKED` | 不得以“长期批准”替代有效期与回收计划 |
| `review_queue_bucket` | 允许按重算结果在 `CRITICAL_WINDOW / HIGH / NORMAL / LOW` 间迁移 | 不得无审计把低优先级项目人工插队为最高优先级 |

补充裁决：

- 状态超时、窗口过期、核心源失效或审批失效时，必须按降级方向流转，不得维持原高状态不变。
- 任何正式状态回滚，都必须说明触发原因、责任对象、审批或重算依据与影响范围。

## 6. 结果语义、公开边界能力矩阵与可售裁决

### 6.1 正式结果语义

| 结果类型 | 正式含义 | 允许使用条件 | 不得做的事 |
|---|---|---|---|
| AUTO_HIT | 公开证据充分、可回链、前置对象齐全、边界说明完整的正式命中 | 必须有正式对象、证据引用、边界说明、最低证据等级 | 不得作为行政、司法或终局违法结论 |
| CLUE | 具有较强公开依据、但仍需人工复核、补证或外部说明的强线索 | 允许作为主摘要或主卖点，但必须保留风险语义 | 不得伪装成已坐实结论 |
| OBSERVATION | 弱异常、披露不足、链路不完整或信息不足观察 | 可作为补充说明与弱信号 | 不得单独作为外发首页主结论 |
| review_request | 需要内部材料、外部说明或人工专项核验才能闭环的事项 | 可进入复核、交付治理或客户补证链 | 不得单独转写为 AUTO_HIT |

补充升级裁决：

- AUTO_HIT 只允许在以下条件同时满足时成立：适用能力层级允许进入正式主链、public_chain 法定节点满足当前动作所需最低完整度、evidence_grade_profile 达到客户可见阈值、requires_manual_confirmation 未触发、且不存在未闭环的核心来源冲突。
- 任何一项核心条件不满足，但主要攻击点、主要异常或主要价值点已具有较强公开依据时，只允许降级为 CLUE。
- 当公开链缺失关键节点、证据颗粒度不足、来源冲突未解、制度模式尚未确认或只存在弱单点信号时，只允许进入 OBSERVATION 或 review_request，不得越级升级。
- 凡进入 review_request 的事项，必须明确缺失条件属于节点缺失、证据等级不足、制度口径未定、非公开补证需求或人工裁量需求中的哪一类。

### 6.2 公开边界能力矩阵

| 层级 | 正式含义 | 默认结果语义 | 使用门槛 | 典型范围 |
|---|---|---|---|---|
| A 层：全国稳定公开可得主能力 | 可作为全国最小公开能力承诺，允许直接进入正式主链 | AUTO_HIT / CLUE / 事实对象 | 公开可查、可回链、可审计，且不依赖地区特殊权限 | 公开链完整性、候选集合攻击面、限制性条款识别、法律动作建议 |
| B 层：工程建设域公开增强能力 | 可进入正式主链，但必须受地区覆盖、平台成熟度与数据等级约束 | CLUE / review_request / 条件性事实对象 | 仅在已验证地区、已声明覆盖的平台和工程建设域场景中启用 | 四库一平台业绩、施工许可、竣工验收备案、合同订立与部分公开履约信息 |
| C 层：条件性公开线索能力 | 只有公开颗粒度足够时才允许启用，默认不得写成全国稳定主能力 | CLUE / OBSERVATION / review_request | 必须满足公开页面已披露、可回链且颗粒度足以支撑判断 | 公开附件、评分细项异常、主观分离群、完整项目管理班子比对、履约/变更异常 |
| D 层：非公开 / 监管 / 取证侧能力 | 不得纳入正式公开主模型 | 禁止 | 任何依赖后台、内部系统、取证材料或不可审计来源的能力一律排除 | 保证金同源、同人编标、同人办标、文件混装、后台日志、IP、设备指纹 |

补充裁决：

- 项目经理在建冲突默认属于 **B 层 + C 层交界能力**：若仅依赖弱公开或企业自行填报信息，默认只能形成 CLUE / review_request；若位于已验证工程建设域覆盖地区且证据等级达到外发阈值，才允许进一步升级。
- 项目经理近期变更注册单位与社保一致性强核查默认属于 **B 层 + C 层交界能力**：仅凭近期变更不得直接写成社保不合格；若招标文件明确要求本单位近 1-3 个月社保，且项目经理变更时间贴近投标节点，同时注册单位、安管 B 证单位、投标单位或公开任职轨迹出现公开冲突时，允许升级为强 CLUE 或 review_request。
- 评分异常不得默认按“所有项目均可细粒度建模”处理；只有公开评分颗粒度足够时，才允许启用 C 层评分异常模型，否则只能做“评标情况与结果不协调”的弱线索。
- 真实竞争者识别本身属于 A 层正式能力，但其输入不得静默消费 D 层取证特征。
- 目标候选人核验的上位边界至少包括四类：违法违规风险、资质 / 业绩 / 人员资格 / 信用 / 财务状况 / 生产条件是否符合中标条件、经营或财务状况重大变化是否影响履约、履约保证金是否按招标文件要求提交。该四类边界可进入正式主链，但具体核验点只能按公开可检索程度分别落入 A/B/C 层。
- 单位负责人同一人、控股或管理关系冲突，默认属于 **B 层 + C 层交界能力**：只有在国家企业信用信息公示系统、招标文件限制条款与公开投标主体信息同时成立时，才允许形成 CLUE 或条件性事实对象。
- 评标报告复核触发条件默认属于 **C 层条件性能力**：只有评标报告、复核说明或等效公开材料足以回链时，才允许进入正式判断链；否则只能停留在 review_request。
- 异常低价与严重不平衡报价识别默认属于 **B 层 + C 层交界能力**：只有在公开报价、公开评标办法、公开澄清 / 复核依据或其他足以支撑的公开材料同时存在时，才允许形成 CLUE 或条件性事实对象；缺少公开复核依据时，不得直接写成 AUTO_HIT、否决结论或当然废标结论。
- 截图、OCR 文本、模型转述摘要只能作为辅助证据；在缺少可回链原始页面 / 原始附件，且无多源交叉印证时，不得单独支撑 legal_action_recommendation、客户版首页主结论或外发异议辅助主证。

### 6.3 sale gate 正式含义

| 状态 | 正式含义 | 最常见触发 |
|---|---|---|
| OPEN | 可以正式推进为可售对象 | 统一事实稳定、主要价值点清晰、无重大治理与交付阻断 |
| REVIEW | 存在需要人工确认的证据、字段、外发、买方适配或窗口期问题 | 价值明确但仍需复核或补证 |
| HOLD | 存在关键证据缺失、窗口期问题、客户适配不清或交付条件不足 | 暂缓推进，等待条件满足 |
| BLOCK | 存在重大合规限制、地区不可售、严重交付阻断、外发禁止或主价值不成立 | 不得推进为正式可售机会 |

补充裁决：

- sale_gate_status 只反映统一事实层对项目价值与推进条件的判断，不替代 coverage 或 delivery 的治理结论。
- coverage_sellable_state 是正式可售范围门，delivery_risk_state 是正式外发 / 交付门；二者均对正式推进具有一票否决效力。
- 正式机会、正式触达、正式交付的组合裁决顺序固定为：先看 coverage_sellable_state，再看 delivery_risk_state，最后看 sale_gate_status。
- 只要 coverage_sellable_state 不为 `SELLABLE`，不得生成可对外推进的正式机会；只要 delivery_risk_state 构成 BLOCK，不得执行正式触达、客户版导出或正式交付；只有 coverage 允许且 delivery 不阻断时，sale_gate_status 才能决定是否进入 OPEN / REVIEW / HOLD 流程。
- `sale_gate_status` 的 canonical 词表固定为 `OPEN / REVIEW / HOLD / BLOCK`；历史实现中的 `ALLOW` 自本轮起废止，不得再进入 contracts、runtime、测试或下位文档补表。

### 6.4 从线索到可售机会的裁决

- 线索不等于可售对象。只有当统一事实成立、窗口期有效、地区与专题可售、推荐 SKU 明确、买方适配成立、交付阻断不为 BLOCK 时，线索才允许升级为 saleable_opportunity。
- 对于竞争驱动型可售机会，除一般条件外，还必须识别出“真实竞争者”，并且能够解释：主要攻击点、候选关系收益、窗口期紧迫度与经营可转化性。
- 可售对象不等于联系人。只有在 buyer_fit / challenger_buyer_fit 成立且 contact_target 来源合规时，阶段 8 才允许生成正式触达计划。
- 成交不等于闭环完成。只有 payment 与 delivery 结果回写，并形成 governance feedback event 时，商业闭环才算成立。

### 6.5 全局裁决优先序

当公开边界门禁、项目归一状态、双闸门、治理门禁、可售门、人工覆盖或受控例外同时出现时，必须按以下固定顺序裁决；高优先级结论压低优先级结论，低优先级不得反向推翻高优先级。

| 优先序 | 裁决层 | 正式含义 |
|---|---|---|
| 1 | 公开边界与法定硬门禁 | D 层禁止、法定关键节点缺失、不可回链、不可审计、公开边界违规，一票否决 |
| 2 | 项目归一与版本 / 时钟完整性 | project_root、notice_version、lot、retender chain、多版本、多时钟未解时，只允许降级，不得升级 |
| 3 | 规则命中闸门与证据可见性闸门 | rule_gate_decision 与 evidence_gate_decision 必须共同成立，任一未通过即不得升级为正式主结论 |
| 4 | coverage / delivery 治理门禁 | coverage_sellable_state 与 delivery_risk_state 决定是否允许正式机会、正式触达、正式交付 |
| 5 | sale_gate_status 与业务对象化 | 仅在前述门禁允许后，sale_gate_status 才决定 OPEN / REVIEW / HOLD / BLOCK 与机会推进方式 |
| 6 | 人工覆盖与受控例外 | 只允许在结果层做受审计的确认、降级、限制、暂停或受限放行，不得推翻更高优先级硬门禁 |

补充裁决：

- 当优先序 1-3 任一层为 BLOCK、UNRESOLVED 或等价禁止状态时，人工覆盖不得将结果直接提升为正式客户版主结论、正式法律动作建议或正式外发放行。
- 受控例外不得改变优先序本身；它只能在不突破更高优先级硬门禁的前提下，调整低优先级层的时效、展示或流程承接方式。

### 6.6 结果升级、降级与阻断矩阵

| 当前结果 | 允许去向 | 禁止去向 |
|---|---|---|
| OBSERVATION | OBSERVATION、review_request | 直接升级为 AUTO_HIT |
| review_request | review_request、CLUE | 直接升级为客户版主结论 |
| CLUE | CLUE、AUTO_HIT（仅在双闸门、边界、证据与治理条件全部成立时） | 终局违法定性、自动监管替代 |
| AUTO_HIT | AUTO_HIT、因重算或治理原因降级为 CLUE / REVIEW / BLOCK | 越过治理门禁直接外发 |
| BLOCK | 回退重算、review_request、受控限制展示 | 人工直接改写为 PASS |

### 6.7 主结论、客户可见与外发可用对应关系

| 结果语义 | 可内部展示 | 可客户可见 | 可正式外发 |
|---|---|---|---|
| AUTO_HIT | 是 | 条件允许 | 条件允许 |
| CLUE | 是 | 条件允许 | 受限允许 |
| OBSERVATION | 是 | 受限允许 | 原则上不允许作为首页主结论 |
| review_request | 是 | 原则上不允许作为主结论 | 不允许 |

## 7. 角色、权限、审计与人工覆盖

### 7.1 最小角色集

| 角色 | 主要权限 | 禁止事项 |
|---|---|---|
| 产品管理员 | 配置系统、规则、字段策略、交付矩阵、coverage 状态 | 不得绕开审批流程静默修改外发边界或公开边界 |
| 核查分析员 | 查看 project_base、public_chain、public_attack_surface、focus_bidder_verification_profile、rule_hit、evidence、field_lineage_record、rule_gate_decision、evidence_gate_decision、review_queue_profile 等内部研判对象 | 不得直接执行客户外发与人工覆盖；不得查看 `private_supplement_record` 与 `controlled_exception_record` 的未授权正文材料 |
| 人工复核员 | 处理 review_request、确认结果层、执行人工覆盖、批准导出、确认高限制补证对象对阶段 5 / 6 的影响范围 | 不得修改阶段 3 真相层原始对象；不得绕开审批链直接放行 D 层材料进入正式公开主链 |
| 经营 / 销售用户 | 查看 project_fact、challenger_candidate_profile、saleable_opportunity、buyer_fit / challenger_buyer_fit、推荐 SKU；在合规门槛满足时创建 / 修改 `contact_target`、编排 `outreach_plan`、记录 `touch_record` | 不得访问未授权底层敏感字段，不得自行改写项目结论；不得绕过联系来源合规、频控、quiet hours 或审批门禁执行正式触达 |
| 交付 / 治理用户 | 审批交付、外发、脱敏、发布门禁、审计事件；执行 `order_record`、`payment_record`、`delivery_record`、`governance_feedback_event` 的正式状态变更；审批高限制对象的可见范围与释放条件 | 不得把治理阻断静默写成项目风险结论；不得越过证据、交付与 release gates 直接放行正式外发 |
| 客户只读用户 | 查看授权范围内的项目摘要与正式交付物 | 不得访问内部真相层与中间态结果 |

### 7.1.1 高限制对象最小权限映射

下列对象的最小权限要求必须与对象级交付矩阵、field policy dictionary 与外发审批链保持一致；任何页面、接口或导出若与本表冲突，以更严格者为准。

| 对象族 | 默认可见角色 | 默认可操作角色 | 默认外发属性 |
|---|---|---|---|
| `field_lineage_record`、`rule_gate_decision`、`evidence_gate_decision`、`review_queue_profile` | 产品管理员、核查分析员、人工复核员、交付 / 治理用户 | 以系统生成与复核确认为主；人工复核员只允许确认结果层影响，不得改写阶段 3 原始事实 | 默认内部研判版，不得直接对客户开放 |
| `private_supplement_record`、`controlled_exception_record` | 人工复核员、交付 / 治理用户；产品管理员如因治理配置需要，仅可查看必要状态元数据，不得默认查看正文材料 | 人工复核员、交付 / 治理用户按审批链处理；不得由核查分析员、经营 / 销售用户或客户只读用户直接操作 | 默认高限制治理对象，不得默认客户可见或外发 |
| `legal_action_actor_profile`、`procurement_decision_actor_profile` | 核查分析员、人工复核员、交付 / 治理用户；经营 / 销售用户仅在对象级交付矩阵批准的交付形态下查看必要摘要 | 以系统生成与治理审批为主；不得由经营 / 销售用户自行改写主体结论 | 必须按角色、场景与交付形态单独审查，不得默认可外发 |
| `project_fact`、`legal_action_recommendation`、`report_record` | 核查分析员、人工复核员、经营 / 销售用户、交付 / 治理用户；客户只读用户仅限授权范围内正式交付版本 | 人工复核员、交付 / 治理用户在审批链内放行；经营 / 销售用户不得改写正式结论 | 仅在证据等级、交付矩阵与审批链同时满足时允许客户可见或外发 |

### 7.1.2 阶段 8 / 9 执行角色最小定义

- `contact_target`、`outreach_plan` 与 `touch_record` 的默认执行角色为经营 / 销售用户；涉及自然人高限制字段、批量触达、超出原公开场景合理预期的触达，或频控策略例外时，必须先经交付 / 治理用户审批后方可执行。
- `order_record`、`payment_record`、`delivery_record` 与 `governance_feedback_event` 的默认执行角色为交付 / 治理用户；经营 / 销售用户可以提交操作申请、查看授权范围内状态，但不得直接完成正式状态变更。
- `review_request`、`review_queue_profile`、`private_supplement_record` 与 `controlled_exception_record` 的处理链必须由人工复核员或交付 / 治理用户承担，不得把复核 / 治理对象下放为销售执行动作。
- 任何阶段 8 / 9 的正式执行动作，只要会影响客户可见交付、正式触达、正式订单 / 交付状态或阶段 6 统一事实回写，都必须存在明确执行角色、审批角色与审计留痕；缺少任一角色定义时，不得宣称该动作已经形成正式能力。

### 7.2 人工覆盖原则

- 人工覆盖只允许作用于阶段 5 / 6 的结果层，不允许直接改写阶段 3 真相层。
- manual_override_status 仅允许 NONE / PENDING / CONFIRMED / REJECTED。
- 任何人工覆盖都必须记录覆盖原因、责任人、审批人、时间与影响范围。
- 人工覆盖默认只允许确认、降级、限制、暂停或驳回结果；不得把优先序更高层已阻断的事项直接升级为正式客户版主结论、正式法律动作建议、正式机会或正式外发放行。
- 当 rule_gate_decision 或 evidence_gate_decision 处于 `REVIEW` 时，人工覆盖只有在完成正式审批且证据包已满足目标使用场景阈值时，才允许把结果层由待审转为可见；当任一闸门处于 `BLOCK` 时，人工覆盖不得直接改写为通过，必须回到规则、证据、版本、归一或治理链重算。
- 人工覆盖进入客户版导出、正式外发异议辅助或正式触达放行前，至少应经过复核或治理侧二次批准；单人审批链不得同时完成创建、覆盖与放行。

### 7.3 审计留痕最低要求

- 查看完整证据载体与敏感字段。
- 执行人工覆盖或驳回。
- 生成客户导出与外发包。
- 创建或修改 contact_target。
- 执行订单、支付、交付状态变更。
- 修改 coverage 状态、field policy、delivery matrix、release gates 或公开边界状态。

## 8. 页面、接口、导出与消费原则

### 8.1 正式页面

| 页面 | 主消费对象 | 核心用途 |
|---|---|---|
| 任务编排页 | 阶段 1 任务对象 | 启动、重跑、窗口期与计划管理 |
| 项目总览页 | project_fact | 统一事实、主价值点、sale gate、coverage / delivery 状态 |
| 公开链与核验工作台 | public_chain、clock_chain_profile、notice_version_chain、public_attack_surface、evidence_grade_profile | 查看公开链完整性、多时钟、多版本、攻击面、证据等级与冲突摘要 |
| 证据工作台 | rule_hit、evidence、rule_gate_decision、evidence_gate_decision、review_request、field_lineage_record | 查看规则解释、双闸门、字段溯源、证据与补证事项 |
| 报告与复核工作台 | report_record、review_queue_profile、legal_action_recommendation、private_supplement_record | 核查简报、证据包、复核排队、隔离补证、异议辅助草稿与动作建议 |

| 订单与交付台 | order_record、payment_record、delivery_record、opportunity_outcome_event | 管理商业状态、交付状态、负结果回写与治理反馈 |
| 治理后台 | coverage registry、controlled_exception_record、field policy、delivery matrix、audit logs | 控制可售、受控例外、外发、脱敏、审批与发布门禁 |

### 8.2 页面硬约束

- 项目页、经营页、销售页必须优先消费 project_fact。
- 公开链与核验页必须消费 public_chain、public_attack_surface 与 evidence_grade_profile，不得由前端自行拼装第二套解释。
- 可售机会页不得直接消费阶段 3 原始对象重算机会等级；必须优先消费 saleable_opportunity 与 challenger_candidate_profile。
- 触达页不得展示来源不合规或未经审批的敏感联系方式。
- clock_chain_profile、notice_version_chain、field_lineage_record、rule_gate_decision、evidence_gate_decision、review_queue_profile、legal_action_actor_profile、procurement_decision_actor_profile、private_supplement_record、controlled_exception_record、opportunity_outcome_event 必须拥有正式消费面；可以作为独立页面，也可以作为现有工作台的正式子视图，但不得仅存在于离线脚本、内部表或人工备注中。

### 8.3 最小正式接口集

- 项目与事实：GET /projects, GET /projects/{id}, GET /projects/{id}/facts, POST /projects/{id}/rebuild
- 公开链、时钟、版本与溯源：GET /projects/{id}/public-chain, GET /projects/{id}/clock-chain, GET /projects/{id}/version-chain, GET /projects/{id}/field-lineage
- 规则、证据与双闸门：GET /projects/{id}/rule-hits, GET /projects/{id}/evidence, GET /projects/{id}/rule-gates, GET /projects/{id}/evidence-gates, GET /projects/{id}/review-requests
- 报告、复核与隔离补证：GET /projects/{id}/report-record, GET /projects/{id}/review-queue, GET /projects/{id}/private-supplements, GET /projects/{id}/legal-actions, POST /reports/{id}/export
- 挑战者识别、双 actor 与可售对象：GET /projects/{id}/challenger-candidates, GET /projects/{id}/actors, GET /saleable-opportunities, GET /saleable-opportunities/{id}, POST /saleable-opportunities/{id}/refresh
- 触达：GET /contact-targets, POST /outreach-plans, POST /touch-records
- 订单、交付与负结果回写：GET /orders, POST /orders, POST /payments, POST /deliveries, GET /projects/{id}/opportunity-outcomes
- 治理与受控例外：GET /coverage-status, GET /delivery-matrix, GET /projects/{id}/exceptions, POST /exception-records, POST /exception-records/{id}/revoke, POST /governance-feedback-events

上述路径为最小正式接口能力示例；D4 可以在不改变对象边界与消费原则的前提下采用等价路径设计，但不得让正式对象处于“有对象、无正式接口承接”的状态。

### 8.4 导出与外发原则

- 任何客户可见导出都必须通过正式交付白名单层，不得从内部对象直接透传。
- 线索型专题默认使用风险线索语义，不得在首页以终局定性表达。
- 没有正式 report_record 或正式 delivery_record 的文件，不得冒充正式交付物。
- 任何导出必须附带公开信息核查与人工复核辅助免责声明。

### 8.5 正式消费优先级

- 页面、接口、导出、销售与交付消费面的正式优先级固定为：`project_fact / legal_action_recommendation -> gate objects -> lineage / version / clock objects -> truth objects -> 备注 / 私有补证 / 临时说明`。
- 低优先级对象只允许解释、补充、追溯或触发复核，不得反向改写高优先级对象的正式主结论。
- 若消费面展示需要穿透到低优先级对象，必须同时保留其与高优先级对象的关联解释，不得孤立展示形成误导。

### 8.6 禁止展示与禁止透传清单

| 对象或内容 | 默认策略 |
|---|---|
| `private_supplement_record` 正文材料 | 禁止默认展示、禁止默认透传 |
| `controlled_exception_record` 正文与治理备注 | 禁止默认客户可见 |
| 未审批的自然人高限制联系方式 | 禁止展示、禁止导出 |
| 未冻结枚举或未声明语义的字段 | 禁止正式接口暴露 |
| 仅有截图、OCR 文本或模型摘要而缺原始载体回链的材料 | 禁止作为主证透传 |

## 9. 治理、交付与外发门禁

### 9.1 coverage 治理

- 必须维护 coverage_governance_registry 或等价正式同步视图，用于记录地区、专题、源族的覆盖等级、验证状态、可售状态、限制条件与治理事件。
- coverage_sellable_state 仅允许使用 NOT_READY / VALIDATING / SELLABLE / RESTRICTED / SUSPENDED / RECOVERING。
- coverage_governance_registry 必须把验证准备、交付准备与正式可售状态分开表达；不得再以“已实现 / 已验证 / 已可售 / 已可交付”形成第二套对外正式状态。
- 当制度类型、候选推荐模式或定标模式会改变可售、可交付、可承诺边界时，coverage_governance_registry 必须同步按 `procurement_regime`、`candidate_order_mode`、`award_determination_mode` 分流，不得继续按单一地区 / 专题 / 源族口径混算。
- coverage_governance_registry 还必须至少同时记录 `public_granularity_state`、`fixation_readiness_state`、`retrieval_stability_state` 三个现实可执行维度；未满足最低公开颗粒度、原始载体固定就绪度或取回稳定性阈值的地区 / 专题 / 源族，不得升级为 SELLABLE。
- 任何地区或专题一旦发生重大合规、客户投诉、核心源失效、验证窗口过期或外发事故，必须触发降级、暂停或重新验证。

### 9.2 公开边界门禁

- 任何新增数据源、字段、规则、触达来源、外发模板，都必须声明其公开能力层级：`A_PUBLIC_CORE`、`B_PUBLIC_ENHANCED`、`C_PUBLIC_CONDITIONAL`、`D_NON_PUBLIC_OR_FORENSIC`。
- A 层能力可以作为全国最小正式承诺；B 层能力必须绑定地区覆盖、平台成熟度、数据等级和治理状态；C 层能力必须绑定公开颗粒度条件与默认线索语义；D 层能力一律不得进入正式公开主模型。
- 任何客户可见交付、API 或销售话术，均不得引用来源不明、不可回链、灰色获取或 D 层来源的信息。
- 任何把 B 层或 C 层能力包装成“全国稳定默认能力”的页面、文案、售前材料或导出模板，均视为边界违规。

### 9.3 字段策略与交付矩阵

- 客户可见字段必须进入 field policy dictionary，声明字段分类、审批要求、脱敏规则、外发审计要求。
- 至少对 project_base、public_chain、clock_chain_profile、notice_version_chain、field_lineage_record、rule_gate_decision、evidence_gate_decision、private_supplement_record、controlled_exception_record、project_fact、public_attack_surface、legal_action_recommendation、challenger_candidate_profile、legal_action_actor_profile、procurement_decision_actor_profile、report_record、review_queue_profile、saleable_opportunity、opportunity_outcome_event、delivery_record 维护对象级交付矩阵。
- 对象级交付矩阵必须区分内部研判版、客户交付版、外发异议辅助包、演示版页面、对外 API。
- field_lineage_record、rule_gate_decision、evidence_gate_decision、review_queue_profile 原则上默认内部研判版；private_supplement_record 与 controlled_exception_record 默认高限制治理对象；legal_action_actor_profile 与 procurement_decision_actor_profile 必须按角色、场景与交付形态单独审查其客户可见范围，不得因其属于正式对象就默认可外发。

### 9.4 delivery risk 正式含义

delivery_risk_state 只反映交付、外发、审批、脱敏、证据等级与审计链相关风险，不替代项目核查结论本身。任何销售或页面都不得把 delivery risk 误用为项目风险结论。

### 9.5 归档、调阅与履约追踪

- 与正式项目判断、正式法律动作建议、正式客户交付、正式治理反馈相关的资料，归档保存期限不得低于 15 年；如适用制度另有更高要求，从其要求。
- 归档资料必须支持按项目、报告、订单、交付与治理事件进行正式调阅；不能调阅、不能定位或不能证明来源链的资料，不得视为完成正式归档。
- 履约跟踪、合同变更、交付回写与后续治理事件，一旦进入正式对象链，即属于阶段 9 正式归档范围，不得视为可丢弃的运行日志。
- 与正式外发主证、正式法律动作建议、正式客户版报告相关的原始页面、原始附件、固定包、抓取时间、原始载体类型、内容哈希与调阅路径，必须一并归档；截图、OCR 文本与模型摘要必须与原始载体区分保存，不得混同为同等级主证。
- 未成交、误判、窗口错过、联系失败、角色错配、客户拒绝、预算不成立或交付放弃等负结果，必须形成 opportunity_outcome_event 或等价正式对象并归档，不得仅停留在销售私有备注或临时沟通记录。

### 9.6 外发审计与审批

- 查看或导出自然人高限制字段、执行客户版导出、执行外发异议包、修改对象级交付矩阵、调整 coverage sellable state、确认人工覆盖，均必须留痕。
- 新增专题、新增字段、新增导出模板、新增地区可售承诺，必须经过正式审批与回归。
- 涉及公开边界状态变化的数据源与规则，必须经过专项边界审查。
- 批量触达渠道、自然人高限制触达、超出原公开场景合理预期的触达，或频控策略发生变化的触达规则，必须经过专项审批、边界复核与审计留痕。
- private_supplement_record 的默认 release_state 必须为 `ISOLATED`；仅在 lawful_basis、usable_scope、非公开边界说明、审批角色与影响目标对象全部明确后，才允许进入 `REVIEW_ELIGIBLE`。
- 只有当补证材料已经完成正式 review_request 关联、证据等级达到目标使用场景最低阈值、且审批角色确认允许影响阶段 5 或阶段 6 时，private_supplement_record 才允许进入 `IMPACT_ELIGIBLE` 或 `RELEASED`。未满足条件时，只允许隔离保存，不得影响 rule_gate_decision、evidence_gate_decision、project_fact 或 legal_action_recommendation。
- private_supplement_record 进入 `RELEASED` 的审批主体不得与创建者为同一角色链；至少应由复核或治理侧正式角色批准，并留下审批时间、审批依据与影响范围。

### 9.7 发布门禁

- 若影响正式对象、正式枚举、字段策略、delivery matrix、coverage registry、release gates 或 project_fact 聚合逻辑，必须触发回归、审批或重新验证。
- 若影响公开边界、证据等级、legal_action_recommendation 或 contact_target 来源，必须额外触发边界回归。
- 任何绕开治理门禁的“先上线后补审计”做法，都不得视为正式发布。

### 9.8 主证规则与证据等级分层

| 使用场景 | 最低主证要求 |
|---|---|
| 内部研判 | 可接受 CLUE 级组合证据，但必须可解释来源链 |
| 客户版报告 | 必须存在可回链原始载体，且证据等级达到客户可见阈值 |
| 外发异议辅助包 | 必须存在原始主证、固定包与适用制度 / 窗口说明 |
| 正式法律动作建议 | 必须同时满足制度口径、窗口状态、主证回链与双闸门要求 |

### 9.9 对外交付分级与禁止组合

| 交付等级 | 允许内容 | 禁止组合 |
|---|---|---|
| 内部研判版 | gate objects、lineage、review queue、内部解释对象 | 不得冒充客户版交付 |
| 客户交付版 | project_fact、report_record 白名单字段、受控 legal_action_recommendation | 不得附带高限制治理对象正文 |
| 外发异议辅助包 | 正式主证、窗口说明、动作建议、必要免责声明 | 不得只含截图 / OCR / 模型摘要 |
| 演示版 / 预览版 | 脱敏摘要、受限截图、示例化表达 | 不得冒充正式交付物或正式外发包 |

## 10. 真实竞争者识别、可售对象与联系对象闭环

### 10.1 可售对象的正式定义

可售对象不是“有线索的项目”，而是“已经满足最低推进条件、能够被正式销售或交付编排的机会对象”。 中，可售对象至少由 sales_lead、buyer_fit / challenger_buyer_fit、offer_recommendation、saleable_opportunity 四个对象共同构成。

### 10.2 竞争驱动型可售对象的正式定义

竞争驱动型可售对象，特指以下场景中的正式机会对象：异议打击、竞争分析、评审异常追击、目标候选结果可被依法挑战且候选集合中存在明确行动受益方的项目推进。此类对象必须回答三个问题：第一，谁是真实竞争者；第二，谁最有动力依法发起行动；第三，谁最可能为该线索与服务付费。

### 10.3 真实竞争者的正式定义

“真实竞争者”不是情绪词，也不是简单的第二名。其正式含义为：在公开证据边界内，对目标候选结果具有明确受损关系、较强依法挑战收益、较高窗口行动性与较高组织化执行可能的真实受损者。该对象必须同时满足以下四层解释：

- **独立竞争性成立**：不是陪跑、不是明显伪竞争者、不是信号污染样本。
- **受损关系成立**：因目标候选结果、异常评分、程序瑕疵、异常无效标或公开链缺陷而存在明确受损。
- **收益关系成立**：目标候选结果被纠正后，该对象具有较强递补收益、重新竞争收益或项目推进收益。
- **行动可能性成立**：当前仍在能够触发异议、复核、项目研判或快速咨询采购的有效窗口期内，并具备一定组织化行动可能。

### 10.4 真实竞争者的主识别阶段

真实竞争者的主识别必须在阶段 6 完成，而不是在阶段 8 触达时临时判断。阶段 6 应至少形成 challenger_candidate_profile，并计算以下代理维度：

- focus_bidder_attackability_score：目标候选结果可攻击度
- challenger_pain_score：受损程度
- succession_gain_score：候选关系收益强度
- execution_readiness_score：行动执行就绪度
- challenger_window_urgency_score：窗口期紧迫度
- challenge_actionability_score：综合行动成立度

### 10.5 线索升级为一般可售机会的最低条件

- 项目处于有效窗口期，且窗口期与推荐 SKU 匹配。
- sale_gate_status 不为 BLOCK。
- coverage_sellable_state 允许进入当前地区 / 专题的正式承诺范围。
- 至少存在 1 个主要价值点，并且主要价值点能被 report_record 或等价证据对象解释。
- buyer_fit 成立，且至少匹配一种目标购买角色。
- offer_recommendation 明确，SKU、推荐交付形态与主要前提条件清楚。
- 对进入正式销售优先级池的对象，至少已形成 `crm_owner_state`、`expected_close_days_band`、`expected_contract_value_band` 与 `expected_delivery_cost_band`；未形成时，最多只允许停留在 sales_lead 或待分配机会队列。
- delivery_risk_state 不为 BLOCK。

### 10.6 线索升级为竞争驱动型可售机会的最低条件

除一般条件外，竞争驱动型机会还必须同时满足：

- real_competitor_count >= 1。
- competitor_quality_grade 至少达到 B。
- 至少存在 1 个主要攻击点能够解释“为何目标候选结果可被依法挑战”。
- 已识别出 challenger_candidate_profile，且 execution_readiness_score 已达到正式阈值。
- 已能说明该挑战者为何最有可能付费购买该线索、服务或快速交付。
- 推荐 SKU 明确属于资格与程序打击包、评审与竞争异常包或其组合。
- 当前窗口期与翻盘收益匹配，未过关键行动窗口。

### 10.7 buyer_fit 与 challenger_buyer_fit 的分工

- buyer_fit 用于一般买方适配，适合合规、履约、监测、咨询类机会。
- challenger_buyer_fit 用于对抗型买方适配，专门回答“谁最可能为竞争驱动型行动与快速交付付费”。
- 竞争驱动型机会必须优先消费 challenger_buyer_fit，而不是仅依赖一般 buyer_fit。
- challenger_buyer_fit 至少要说明：挑战者是谁、其组织类型、其角色簇、其购买意愿、其支付能力、其窗口行动性，以及推荐销售话术角度。
- 在竞争驱动型机会中，buyer_fit / challenger_buyer_fit 只能回答适配与付费意愿，不得替代法律行动主体与采购决策主体对象。legal_action_actor_profile 负责回答“谁最可能依法发起正式动作”，procurement_decision_actor_profile 负责回答“谁最可能拍板采购、推动签约或承担付款”。

### 10.8 联系对象的正式定义

联系对象不是“任何能找到电话的人”，而是与可售机会有明确关联、来源合规、联系方式可审计、角色适配成立的 contact_target。联系对象至少要回答：这个人属于哪个组织、扮演什么角色、为什么联系他、联系渠道是什么、来源是否合法。
对于自然人联系对象，总纲层最小必答还包括：渠道类别、法律基础、频控状态、拒绝 / 退订状态与静默时段状态。D9 可以继续细化各渠道策略，但不得低于上述最小字段要求。
正式链路默认优先组织联系方式；自然人联系方式默认更严，除非已经明确证明组织触达路径不成立、当前场景存在清晰合法基础且已通过人工审查或 D9 自动放行条件，否则不得优先选择自然人路径。

### 10.9 典型阻断情形

| 阻断情形 | 阻断层级 | 正式处理方式 |
|---|---|---|
| 项目归一未完成或版本链冲突未解 | 统一事实层 / 机会层 | 不得进入正式机会池、客户版报告或正式触达池 |
| 地区或专题不可售 | 机会层 | 不得生成正式 saleable_opportunity，可保留为内部观察项目 |
| 主要价值点证据不足 | 机会层 | 保持在 sales_lead 或进入 review_request，不得推进 |
| 没有明确买方适配 | 机会层 | 不得进入正式销售池 |
| 最小销售运营编排字段缺失 | 机会层 | 只允许停留在 sales_lead 或待分配池，不得进入正式销售优先级池 |
| 未识别出真实竞争者 | 竞争驱动机会层 | 禁止升级为异议打击型或竞争分析型机会 |
| 联系来源不合规 | 触达层 | 禁止生成 contact_target 与触达计划 |
| 外发或交付阻断 | 交付层 | 允许保留机会，但不得导出或交付客户版结果 |
| 客户反馈与公开事实冲突未闭环 | 统一事实层 | 必须回写阶段 6，并按需要重新复核 |

### 10.10 双 actor 裁决清单

| 问题 | `legal_action_actor_profile` 回答 | `procurement_decision_actor_profile` 回答 |
|---|---|---|
| 谁最可能依法发起正式动作 | 是 | 否 |
| 谁最可能拍板采购、推动签约或承担付款 | 否 | 是 |
| 谁更适合作为阶段 8 的优先触达入口 | 条件回答 | 条件回答 |
| 谁的判断可以直接替代另一类 actor | 不允许 | 不允许 |

## 11. 测试验收、发布门禁与运行要求

### 11.1 P0-P8 正式验收门槛

| 阶段 | 最低通过件 | 一票失败信号 |
|---|---|---|
| P0 统一底座 | 正式对象、正式枚举、阶段口径、公开边界口径、文档优先级冻结 | 存在第二套对象名、第二套结果语义或第二套边界口径 |
| P1 抓取成立 | 至少 1 个公开源族稳定抓取；A 层公开链抓取稳定成立；多时钟链与多版本链可形成并可追溯原始载体 | 原始页面 / 附件不可追溯，或临时抓取内容被当作正式主证 |
| P2 解析成立 | `project_base`、候选人、项目经理、处罚与至少 2 类 profile 对象稳定形成；关键字段 `field_lineage_record` 稳定形成 | 关键字段缺来源切片、来源版本、解析置信度或归一规则 ID 却进入正式主链 |
| P3 公开核验成立 | `public_attack_surface`、`pseudo_competitor_signal_set`、`evidence_grade_profile` 稳定形成；A/B/C/D 四层边界正确打标 | D 层输入静默进入公开核验对象或正式页面消费面 |
| P4 规则与证据成立 | 至少 1 类资格专题、1 类程序专题、1 类评审专题或工程建设域增强专题稳定输出；`rule_gate_decision` 与 `evidence_gate_decision` 同步成立 | 只跑规则、不跑证据；或双闸门任一缺失却升级正式主结论 |
| P5 统一事实成立 | `project_fact`、`sale_gate_status`、`coverage_sellable_state`、`delivery_risk_state`、`legal_action_recommendation` 稳定形成且冲突可解释 | 项目归一未完成、版本冲突未解或闸门未过却直接形成正式客户版主结论 |
| P6 商业识别成立 | `challenger_candidate_profile`、`challenger_buyer_fit`、`sales_lead`、`saleable_opportunity`、`review_queue_profile` 真实消费统一事实 | `saleable_opportunity` 由单条 rule、单页公告或销售手工备注直接生成 |
| P7 商业触达与收入回写成立 | `contact_target`、`touch_record`、`order_record`、`payment_record`、`delivery_record`、`governance_feedback_event` 真实回写统一事实 | 联系来源不合规、触达无审批 / 审计、订单 / 交付状态不能回写统一事实 |
| P8 边界一致性成立 | 所有规则、页面、导出、触达来源、对外表述与 A/B/C/D 四层能力矩阵保持一致；B/C 层条件可解释；D 层输入零渗透 | 对外表述越界、客户版字段越界、B/C 包装成全国默认能力、D 层渗透正式主链 |

### 11.2 最小必测对象与机制清单

下列对象与机制必须进入 D11 与 `/contracts/testing`，否则不得视为完成正式验收：

| 类别 | 必测对象 / 机制 | 最低要求 |
|---|---|---|
| 真相层 | `project_base`、`bidder_candidate`、`project_manager`、`field_lineage_record` | 必须存在正向样本、负向样本、缺失样本与冲突样本 |
| 公开链 | `public_chain`、`clock_chain_profile`、`notice_version_chain` | 必须覆盖多时钟并行、版本替代、错序、缺节点与补抓重算 |
| 核验层 | `public_attack_surface`、`focus_bidder_verification_profile`、`evidence_grade_profile` | 必须覆盖公开充分 / 不充分、回链成立 / 不成立、边界正确打标 |
| 规则证据层 | `rule_hit`、`evidence`、`rule_gate_decision`、`evidence_gate_decision`、`review_request` | 必须覆盖 PASS / REVIEW / BLOCK 三态与升级 / 降级 / 阻断路径 |
| 统一事实层 | `project_fact`、`legal_action_recommendation`、`review_queue_profile` | 必须覆盖项目归一成功 / 失败、正式建议成立 / 不成立、快速复核与标准复核分流 |

| 治理机制 | `private_supplement_record` release state、`controlled_exception_record`、coverage、delivery matrix | 必须覆盖隔离保存、复核可用、影响可用、释放失败与审批阻断 |

### 11.3 一票失败条件

出现以下任一情形时，相关版本不得通过正式验收、不得进入正式发布：

1. D 层输入进入正式公开主模型、正式页面、正式接口或正式规则消费面。
2. 关键字段缺失 `field_lineage_record` 最小必有信息，却被 `project_fact`、`legal_action_recommendation` 或客户版报告消费。
3. `rule_gate_decision` 或 `evidence_gate_decision` 任一缺失，却仍升级为正式主结论。
4. `saleable_opportunity` 未消费阶段 6 正式 `project_fact`，而是由原始页面、单条规则或销售备注直接生成。
5. `contact_target` 来源不合规，或正式触达缺审批链、频控、quiet hours 或审计留痕。
6. 正式客户版证据包、法律动作建议或外发主证仅以截图、OCR 文本或模型摘要作为唯一主证。
7. 页面、接口、导出、销售表述与本文 A/B/C/D 边界矩阵不一致。
8. 文档已改但对应 D11 / 机器资产 / 对象级交付矩阵未同步，且变更已影响正式对象、字段、边界或发布门禁。

### 11.4 变更触发回归矩阵

| 变更类型 | 必须重跑的回归范围 | 最低附带材料 |
|---|---|---|
| 正式对象 / 正式枚举 / 状态机变化 | D11 对象契约回归、页面 / API 回归、发布门禁回归 | 变更说明、影响对象清单、迁移说明 |
| 规则码、默认结果语义、证据等级变化 | 规则回归、双闸门回归、客户版证据可见性回归 | 命中变化、误报 / 漏报说明 |
| 公开边界、A/B/C/D 打标或能力矩阵变化 | 边界回归、销售表述回归、导出模板回归 | 边界说明、外发影响清单 |

### 11.5 发布前检查

发布前至少必须完成以下检查：

1. 是否扩大客户可见承诺或自然人字段处理范围。
2. 是否改变地区 / 专题 / 源族覆盖等级与 sellable 状态。

4. 是否影响 `project_fact`、`challenger_candidate_profile`、`saleable_opportunity`、`delivery_record` 的正式字段与枚举。
5. 是否影响多时钟链、多版本链、字段级 lineage、双闸门、隔离区 release state、review queue 优先级或 coverage 的现实可执行维度。
6. 是否将 B 层能力误写为全国统一承诺，或将 C 层能力误写为默认稳定能力。
7. 是否已经完成对应审批、回归、降级声明或迁移说明。

## 13. 基线实施优先级与变更治理

### 13.1 当前范围声明

- 当前版本已正式冻结 L0 总纲边界、九阶段主链、正式对象体系、结果语义、双闸门、角色权限与发布门禁。

- 本文中对 D8-core、D8-plus、D9、D10 及其他下位文档的表述，属于正式承接边界与目标约束，不构成这些文档已经存在、已经生效或已经实现完成的证明。

### 13.2 基线分层裁决

- `L0` 是领域总纲 / 领域宪法，只负责冻结领域边界、九阶段正式定义、正式对象、正式闸门、消费原则、治理优先级与禁区，不负责承载所有实现细节。

- `L2` 是 MVP 商业包，用于回答“首版卖什么、先验证什么、怎么衡量是否成功”。`L2` 至少包括 P1 MVP 范围文档、P2 北极星与护栏指标、P3 首批客户成功路径、P4 报价与签约模型、P5 关键页面任务流。
- `L1` 与 `L2` 只能细化 `L0`，不得改写 `L0`；凡 `L1`、`L2` 与 `L0` 冲突，以 `L0` 为准。

## 附录 C 核心裁决复述

 的正式主链固定为：

**任务编排与时钟管理 -> 公开链抓取 -> 结构化解析与关键对象抽取 -> 关键对象定向公开核验与冲突预判 -> 规则与证据 -> 人工复核、统一事实、真实竞争者识别与正式报告 -> 商业承接与可售对象 -> 联系对象与销售触达 -> 订单支付交付与治理反馈。**

在此主链中，正式公开能力必须按 **A/B/C/D 四层** 统一治理：A 层是全国稳定公开可得主能力，B 层是工程建设域公开增强能力，C 层是条件性公开线索能力，D 层是非公开 / 监管 / 取证侧能力。只有 A、B、C 层才允许在各自条件下进入公开版判断链；凡 D 层能力，均不得进入正式公开主模型。暗箱 / 内定型异常、评分异常、项目经理在建冲突等事项，在纯公开边界下原则上只能形成强线索、条件性结论或复核请求，不得越界替代监管、评标或司法作终局裁决。
