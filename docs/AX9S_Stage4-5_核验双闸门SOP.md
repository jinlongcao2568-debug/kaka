# AX9S Stage4-5 核验双闸门SOP

**版本**: 2026-05-01 v1

**定位**
- 本文件是 `docs/AX9S_Stage1-9_执行矩阵与子漏斗.md` 在 **Stage4-5** 维度的 L3/L4 展开文档。
- `docs/AX9S_产品主图与验收总则.md` 只保留 L1 总口径；`docs/AX9S_Stage1-9_执行矩阵与子漏斗.md` 只保留 L2 路径、双门回退和验收门；本文件才维护 Stage4/5 的具体操作规程。
- future Stage4-5 规则资料、候选来源和 rule basis 吸收入口统一看 `docs/专题_投前预测_Stage1-9 全链规则与判定资料池.md` 的 Stage4-5 主规则模块；不再维护单独的 Stage4-5 规则资料文档。
- 本文件不把 owner 实战经验、产品假设、代码现状混成一种权威。每个规则必须标明可信等级、实现状态和待验证事项。
- Stage4 只生成公开核验 carrier、readback、证据风险和 review 信息；Stage5 才做 `rule_gate_decision` 和 `evidence_gate_decision`。任何 Stage4 结果都不得直接变成正式违法结论、正式可售机会或客户交付结论。

## 0. 总原则

1. **先定对象，再定来源，再定规则**
   - 先从 Stage3 的 `project_base`、`bidder_candidate`、`project_manager`、`field_lineage_record` 生成核验目标。
   - 再按核验目标选择官方公开源。
   - 最后把公开源证据送入 Stage5 双闸门。

2. **不做泛搜索裁决**
   - 搜索可以用来发现入口，但正式核验必须落到公开可回放页面、附件、API readback 或快照。
   - 查不到不等于事实不存在；只能 review、换源或补证。

3. **按业务类型分流核验**
   - 施工/EPC、监理、设计、勘察、勘察设计、设备材料、服务采购不能套同一套项目经理规则。
   - 类型分流只决定核验路线和优先级，不作为源头删除真实候选的理由。
   - 设计/勘察/监理项目缺施工项目经理时，不得直接判字段失败；必须核验对应负责人、注册证书和企业资质链。

4. **双闸门不合并**
   - `rule_gate_decision` 只回答规则前提是否成立。
   - `evidence_gate_decision` 只回答证据是否足够、公开、可回放、可外发。
   - 任一缺失、REVIEW 或 BLOCK，不得升级正式主结论。

5. **证据等级分层**
   - 页面标题、搜索列表、模型摘要、截图、OCR 只能作为辅助线索。
   - 原始公开页、附件、字段血缘、快照、hash、source slice 才能支撑正式证据门。

6. **公开注册信息匹配，不核实自然人本人**
   - Stage4 不得写“是不是本人”“确认本人”等表述。
   - Stage4 只能核验候选公示或 `08 投标文件公开` 中的姓名、候选公司/联合体成员、证书号、角色字段是否与四库或地方注册人员公开记录匹配。
   - `姓名 + 注册单位/候选公司` 匹配时，身份线索先成立；证书号、注册类别、注册专业、有效期、状态匹配时增强证据强度；不匹配时才进入风险线索。

## 1. 核验单元模板

每个核验单元必须补齐以下字段：

| 字段 | 含义 |
| --- | --- |
| 核验目标 | 这项核验要证明或排除什么 |
| 公开来源 | 官方站点、地方站点、公告附件或内部 readback |
| 搜索方式 | 输入哪些字段，如何定位页面 |
| 翻页/消歧 | 如何处理分页、同名、同公司简称、标段/包号 |
| 抽取字段 | 需要拿哪些字段 |
| rule gate | 哪些条件构成规则命中、REVIEW 或 BLOCK |
| evidence gate | 哪些证据足够、哪些证据不足 |
| 失败补救 | 失败后回哪个阶段、换哪个来源、补哪个字段 |
| 当前实现状态 | IMPLEMENTED / PARTIAL / MISSING / NEEDS_REAL_PAGE_VERIFY / NEEDS_TEST |
| 可信等级 | AUTHORITY / CODE_CONFIRMED / REAL_PAGE_VERIFIED / USER_FIELD_EXPERIENCE / PRODUCT_HYPOTHESIS / TO_VERIFY |

## 1.1 多源核验链矩阵

Stage4/5 不得把四库一平台当成唯一核验源。四库/JZSC 更适合作为身份、资质、人员、项目归档补强；项目是否真实进入履约、项目经理责任窗口是否释放、是否有变更或处罚线索，必须优先交叉省市公共资源平台、地方住建/行政审批/施工许可平台和信用处罚源。

| 核验链 | 主来源 | 可核验字段 | Stage4 carrier | Stage5 用法 | 不足时状态 |
| --- | --- | --- | --- | --- | --- |
| 公告承诺链 | 省级/市级公共资源平台、政府采购平台 | 招标公告、招标文件、资格条件、中标候选人、中标结果、澄清/变更、附件、项目编号、标段号、候选公司、拟派项目经理、金额、工期 | `public_notice_timeline`, `bid_candidate_notice`, `award_result_notice` | 锚定投标/中标承诺、规则适用边界、版本优先级和当前有效公告 | 缺公告版本链只能 review，不得按旧公告或单页裁决 |
| 履约时间链 | 地方住建/行政审批/施工许可/工程合同信用信息 | 施工许可证号、注册项目经理、施工单位、合同价、合同开工/竣工日期、许可状态、合同备案号 | `construction_permit`, `contract_public_info` | 判断真实履约窗口，校验项目经理是否处于同一时间段多项目任职 | 缺许可/合同时间只能 review，不得以中标日期单独判冲突或无冲突 |
| 竣工释放链 | 地方住建竣工验收、竣工备案、消防/联合验收 | 竣工验收日期、备案编号、验收状态、施工单位、项目状态 | `completion_filing` | 判断项目是否释放项目经理责任窗口 | 找不到竣工/验收/解除锁定证据时保持 `UNKNOWN/REVIEW`，不得判无风险 |
| 项目经理变更链 | 住建/招投标监管项目经理变更公示、施工许可变更 | 原项目经理、新项目经理、证书编号、变更日期、变更原因、无在建声明 | `project_manager_change_notice` | 切分项目经理责任窗口，避免把已变更项目错误算作持续在建 | 只有自由文本或无证书编号时 review |
| 身份归档链 | 四库一平台企业/人员/项目页、省级建筑市场平台 | 企业主体、人员证书号、注册单位、专业等级、有效期、历史项目 | `enterprise_public_record`, `personnel_public_record`, `performance_public_record` | 消歧同名、确认注册单位和证书有效性，补历史项目 | JZSC 查不到不能判无风险，只能换源或 review |
| 风险信号链 | 信用中国、执行信息公开网、国家企业信用信息公示系统、地方住建/行政处罚公示 | 行政处罚、失信被执行、经营异常、严重违法失信、拖欠工资、质量安全处罚、投诉处理决定 | `credit_penalty_blacklist`, `administrative_penalty_public_record`, `complaint_or_supervision_decision` | 进入风险线索和商业钩子；严重有效限制再进入规则候选 | 站点阻断或主体不一致只能 fail-closed/review |

**多源链硬规则**
- 项目经理重叠必须用“公告承诺链 + 履约时间链 + 竣工释放链 + 身份归档链”交叉判断。
- 变更公告可以切分责任窗口；没有变更公告时不能假设已经变更。
- 处罚、投诉、异议、监督决定不只是阻断项，也可以成为可售风险线索，但客户交付前仍要过 Stage5 双门和后续审批审计。
- 任一链只有搜索结果、截图、模型摘要或列表标题，不能支撑 evidence gate PASS。

**项目负责人释放证据链**
- 当前项目必须先有 `07 中标候选人公示` 或附件中的候选公司/联合体、负责人、证书号/公开人员 ID、排名/报价等绑定证据。
- 可能占用项目必须尽量取得中标结果、施工许可、合同备案/工程合同信用信息、履约记录或项目经理变更等公开记录；只有旧项目候选公示时，只能作为中等线索。
- 释放证据包括合同约定工作验收合格或工程移交、竣工验收/备案、项目经理变更或施工许可变更、非承包方原因停工超过 120 天且建设单位同意、同一工程相邻分段/分期例外；广州项目还要关注建筑施工项目安全生产标准化考评结果告知书。
- 缺竣工、缺备案、缺变更或缺安全生产标准化考评释放材料时，只能输出“项目负责人未释放风险线索”或 `REVIEW_REQUIRED`，不得输出“在建冲突成立”。
- 查询未命中、站点阻断、接口失败和四库/JZSC 未出现项目，不得推断“无在建”“无风险”；必须保留公开源 readback 缺口。

## 1.1.1 核验范围策略

核验范围按“核验对象”决定，不按当前搜索地区一刀切。广东项目只说明当前项目的属地，不说明项目经理历史责任窗口只在广东。

| 核验对象 | 范围模式 | 为什么 | 当前机器落点 |
| --- | --- | --- | --- |
| 当前项目公告链 | 固定当前项目属地：省/市公共资源平台、政府采购平台 | 公告、澄清、中标候选人和结果公告由项目所在地平台发布 | `current_project_notice_chain` |
| 当前项目许可/合同/竣工 | 固定当前项目属地：地方住建/行政审批/施工许可/竣工备案 | 施工许可、合同备案、竣工备案属于项目所在地监管记录 | `current_project_permit_contract_completion` |
| 当前项目项目经理变更 | 当前项目属地优先，必要时用全国历史责任链补强 | 当前项目变更公告在本地，但变更前后责任窗口要和全国历史项目交叉 | `current_project_manager_change` |
| 项目经理身份 | 全国优先 + 注册单位/地方源补强 | 同名消歧、证书号、注册单位和人员公开 ID 不能被当前省份限制 | `project_manager_identity` |
| 公司/项目经理出现地区发现 | 全国索引发现 + 命中地区定向核验 | 先用候选公司、已消歧项目经理、证书号/人员公开 ID 找近几年实际出现的项目和地区，避免无脑扫所有省市 | `company_manager_project_region_discovery` |
| 项目经理在建冲突 | 全国发现后定向深核 | 项目经理可能在任意省市有未竣工/未验收/未变更释放项目；但执行策略应先发现命中地区，再查这些地区的合同/许可/竣工/变更 | `project_manager_active_conflict` |
| 企业信用/处罚/黑名单 | 全国优先 + 项目地/注册地地方源补强 | 企业处罚、失信、经营异常和欠薪投诉可能出现在国家源或其他经营地 | `bidder_company_credit_and_penalty` |

**范围硬规则**
- `project_manager_active_conflict` 必须是 `NATIONAL_DISCOVERY_THEN_TARGETED_REGIONAL_VERIFICATION`。
- 只查当前项目所在省，只能作为地方补强，不能得出“无在建冲突”。
- “全国”不是 31 个省市无脑全扫；必须先执行 `company_manager_project_region_discovery`，用候选公司优先、项目经理证书/人员 ID 辅助，产出 `appeared_region_codes`、历史项目 refs、候选冲突时间窗。
- 命中地区必须继续做定向深核：中标/候选公告、施工许可、合同备案、竣工备案、项目经理变更、处罚/投诉线索。
- `no_conflict` 必须同时满足：项目经理身份已消歧、公司/项目经理出现地区发现已完成、命中地区已核验或 fail-closed、当前项目时间窗口已确定、冲突项目竣工/变更释放状态已确定、证据可回放。
- 当前机器落点：`src/stage4_verification/verification_scope_policy.py`、`regional_hard_defect_source_plan.verification_scope_policy`、`project_manager_active_conflict_readback.verification_scope_policy`。

- 不允许把 `run_internal_chain` 或公告详情快照 `MATCHED` 当作客户可售证据闭合。

## 2. 企业核验

| 项 | 内容 |
| --- | --- |
| 核验目标 | 确认候选/中标企业主体存在、名称一致、主体状态可公开回放 |
| 公开来源 | 四库一平台企业入口、国家企业信用信息公示系统、地方住建企业库、公共资源公告附件 |
| 搜索方式 | 优先用统一社会信用代码；其次用企业全称；不得只用简称 |
| 翻页/消歧 | 企业名相近时核对统一社会信用代码、注册地址、法定代表人、资质证书号、公告主体字段 |
| 抽取字段 | 企业名称、统一社会信用代码、注册状态、资质入口、人员入口、项目入口、source URL、snapshot id |
| rule gate | 企业主体和候选公告主体不一致时进入主体不匹配规则候选；企业主体未匹配只进入 review，不直接判定不存在 |
| evidence gate | 需要企业公开页或公告附件快照；搜索结果列表不足以 PASS |
| 失败补救 | 回 Stage2 补详情/附件；换 GSXT/地方住建；补统一社会信用代码 |
| 当前实现状态 | PARTIAL：active conflict 已可消费人员项目记录；JZSC 人员项目渲染行 adapter 已能生成 `performance_public_record`、`contract_public_info`、`completion_filing` carrier；JZSC 公司优先浏览器采集计划已固化；真实浏览器点击人员详情、项目列表翻页执行器和地方住建兜底待补 |
| 可信等级 | AUTHORITY + CODE_CONFIRMED |

**必须避免**
- 只按企业简称匹配。
- 搜不到企业就生成“企业不存在”结论。
- 把公告里自由文本企业名当作企业公开记录。

## 3. 项目经理核验

| 项 | 内容 |
| --- | --- |
| 核验目标 | 确认项目经理是否属于候选企业、证书/专业/等级/注册状态是否匹配，并为在建冲突判断提供人员身份 |
| 公开来源 | 四库一平台企业页 -> 注册人员/项目负责人列表 -> 人员详情；地方住建人员查询；公告附件 |
| 搜索方式 | 企业优先：先查候选企业，再在企业人员列表中找项目经理；有人员公开 ID/证书号时可直达人员详情 |
| 翻页/消歧 | 企业人员列表必须翻页或调用等价公开接口；同名人员用注册单位、证书号、专业、等级、人员公开 ID 消歧 |
| 抽取字段 | 姓名、注册单位、证书号、注册专业、证书等级、注册状态、注册时间、变更时间、有效期、人员公开 ID、人员详情 URL；公告解析阶段还必须区分 `project_manager_certificate_type`、`project_manager_cert_specialty`、`project_manager_professional_title` |
| rule gate | 注册单位不符、证书/专业/等级不满足、注册时间晚于投标/资格关键节点、注册状态无效时进入规则候选 |
| evidence gate | 需要企业人员页或人员详情页快照；`personnel_public_record` 必须为 `MATCHED` 且 `review_required=false` 且有 `source_url/source_snapshot_id` 才能支撑身份链；只搜姓名、只看搜索列表、只看第一页均不足以 PASS |
| 失败补救 | 公告缺证书编号但有候选公司 + 项目经理/项目负责人姓名时，必须生成 `JZSC_COMPANY_FIRST_REQUIRED`：先查企业、翻人员列表、拿证书号/人员公开 ID、写回 `project_manager_public_identifier_optional`，再恢复在建/合同/业绩/变更核验；同名无法消歧进入 `AMBIGUOUS_PUBLIC_MATCH` |
| 当前实现状态 | PARTIAL：active conflict carrier 已表达企业优先路线、禁止姓名泛搜最终证明；JZSC 企业人员列表渲染行 adapter 已能从公司内唯一人员记录派生证书编号、人员详情 URL 和注册时间；JZSC 公司优先浏览器采集计划已定义入口、翻页、人员详情和项目记录步骤；真实公告缺证书编号时 operator readback 已显式生成公司优先身份补全计划；真实浏览器执行器仍待补 |
| 可信等级 | USER_FIELD_EXPERIENCE + AUTHORITY + CODE_CONFIRMED；真实页面路径待验证 |

**公开注册信息匹配口径**
- 本核验只判断公开注册信息是否匹配，不判断自然人“是不是本人”。
- `姓名 + 候选公司/联合体成员` 与公开注册人员记录中的姓名、注册单位匹配时，先视为该候选组身份线索已解析。
- 证书号、注册类别、注册专业、有效期、状态与公告或投标文件字段继续核对；不匹配才进入注册信息不一致线索。
- 推荐路线顺序：`姓名 + 公司/注册单位` 直接查人员；失败或不稳定时走“公司搜索 -> 人员列表”；再失败时走姓名-only 枚举并按注册单位过滤。
- 点击人员详情页是二次核验和补字段手段；详情页文本短时间未渲染出姓名，不得覆盖人员列表行已经完成的公司+姓名+证书匹配事实。

**候选后项目负责人早取证与补证顺序**

`ResponsiblePersonEarlyProbe v1` 是候选后证据包进入 Stage4 live 核验前的前置策略。它的目标是先拿到可核验身份字段，而不是先全量解析大体积投标文件。

1. `07` 详情 HTML 和小 PDF 已有候选公司、负责人姓名、证书号时，直接进入 Stage4 证书号核验和注册单位核验。
2. `07` 多候选人表必须先形成 `候选组`：同一候选行的联合体牵头方、成员方、负责人、证书、报价、排名绑定在一起，不得跨候选行混配。
3. 联合体项目必须核验同一候选组的全部成员公司；任一成员与负责人/证书注册单位匹配，即视为该候选组身份线索已解析，其他成员未匹配不得单独升级为冲突。
4. `07` 只有候选公司 + 负责人姓名、缺证书号时，生成 `COMPANY_FIRST_CERTIFICATE_SUPPLEMENT_REQUIRED`，先查候选公司或联合体成员公司，在企业人员列表中找同名人员；企业内唯一匹配时回填证书号、人员公开 ID 和人员详情 URL。
5. 公司优先未命中时，生成 `NAME_ENUMERATION_FALLBACK_REQUIRED`，允许按姓名枚举公开人员，但必须逐个核对注册单位是否匹配候选公司或联合体成员。姓名枚举只用于补证和消歧，不能单独 PASS。
6. 同一候选组全部成员经公司优先和姓名枚举都未匹配候选公司时，生成 `FLOW_08_TARGETED_PARSE_REQUIRED`，风险等级上调，但不得直接判定冲突成立。
7. `08 投标文件公开` 默认只登记存在、URL、附件清单和后续可解析目标，不默认下载或解析；只有 `07 + Stage4` 缺字段、不匹配、证书类别/专业不足或需要核对 `08` 声明业绩时才做定向解析。定向解析优先资格、项目负责人、项目经理、总监、建造师、证书、商务、投标函等文件；抽到证书号即停止该文件解析并回到 Stage4 核验。
8. `08` 回填证书号后，若证书注册单位仍不匹配同一候选组任一候选公司或联合体成员，生成 `STRONG_CONFLICT_CLUE_AFTER_08`，进入 Stage5 证据门和规则门；输出只能是“项目负责人注册单位不一致线索”或“证书号回填后注册单位仍不一致，高风险线索”，不得输出法律定性。
9. `08` 定向解析仍拿不到证书号时，生成 `INSUFFICIENT_EVIDENCE_AFTER_08`，保持高风险 review，并决定是否进入 OCR 或更深解析。

**Stage3 项目负责人别名和身份字段归一标准**
- 施工/EPC lane：`项目经理`、`项目负责人`、`拟派项目负责人`、`施工负责人` 归一为 `project_manager_name`。
- 监理 lane：`总监理工程师`、`总监`、`监理负责人` 进入 `chief_supervision_engineer_name`，并为了兼容现有身份核验链同步进入 `project_manager_name`。
- 设计 lane：`设计负责人`、`项目设计负责人`、`建筑专业负责人`、`结构专业负责人` 进入 `design_lead_name` 和 `primary_responsible_person_name`，不直接当施工项目经理。
- 勘察 lane：`勘察负责人`、`项目勘察负责人`、`岩土负责人` 进入 `survey_lead_name` 和 `primary_responsible_person_name`，不直接当施工项目经理。
- 勘察设计 lane：没有明确设计/勘察标签时，先进入 `primary_responsible_person_name` 和 `primary_responsible_role=survey_design_project_lead`，待 Stage4 按企业/人员证书补具体角色。
- 服务/咨询 lane：`项目总负责人`、带服务/咨询上下文的 `项目负责人` 进入 `primary_responsible_person_name` 和 `primary_responsible_role=service_project_lead`，不直接当施工项目经理。
- 广东评标报告候选表常见 `项目负责人姓名及资格证书编号`、`项目总负责人姓名及资格证书编号`、`项目负责人`、`项目总负责人` 列；Stage3 可从 `公司 + 报价 + 姓名/证书号`、`公司 + 报价 + 姓名 证书号` 或 `公司 + 报价 + 下浮率 + 姓名` 补 `primary_responsible_person_name`，有证书号时同步补 `project_manager_certificate_no`，无证书号时进入 Stage4 公司优先身份补全。
- `负责人` 只有在同一标签出现 `项目`、`拟派`、`标段`、`施工`、`监理`、`服务`、`咨询` 等上下文时才可归一为负责人字段；采购负责人、联系人、经办负责人不得归一。
- `一级建造师`、`二级建造师`、`注册建造师`、`注册监理工程师`、`注册建筑师`、`注册结构工程师`、`注册土木工程师（岩土）`、`注册公用设备工程师`、`注册电气工程师` 等归一为 `project_manager_certificate_type`。
- `机电`、`市政`、`建筑`、`公路`、`水利`、`土木`、`岩土`、`结构`、`给排水`、`暖通`、`电气` 等归一为 `project_manager_cert_specialty`。
- `工程师`、`高级工程师`、`正高级工程师` 归一为 `project_manager_professional_title`，不得单独当作项目经理姓名或证书编号。

**判定禁令**
- 不得只搜项目经理姓名。
- 不得因为同名太多就判定无冲突。
- 不得因为企业页第一页没看到人员就判定公司没有该项目经理。
- 不得因为中标候选公示没有证书编号就直接判定项目经理核验失败；必须先走四库企业优先身份补全。
- 不得把 `REVIEW_REQUIRED`、同名未消歧或未匹配的人员记录当成身份核验通过。
- 企业内唯一匹配人员记录必须把证书编号/人员公开 ID 写回 `project_manager_public_identifier_optional`，后续在建、合同、业绩核验优先用该编号，不退回姓名泛搜。
- 人员行带注册单位时，注册单位与目标公司冲突必须进入 `registered_unit_name_conflicts_with_target_company`，不得只凭姓名/证书号 PASS。
- 缺 `source_url` 或 `source_snapshot_id` 的人员/项目 carrier 只能 REVIEW，不得支撑 `OVERLAP_RISK`。
- 不得把刚注册/变更直接判成违法；只能把时间事实送入 Stage5 双闸门。

**范围要求**
- 本核验不得限制在当前项目省份。当前项目在广东，只决定当前项目许可/合同/竣工/变更优先查广东；项目经理过往和同时在建项目必须先走全国公司/人员/项目索引发现实际命中地区，再用命中地区地方源补强。
- 公司/项目经理出现地区发现本身是独立核验环节：它既服务在建冲突，也服务公司项目履历、合同/投标/中标时间重叠、业绩真实性和后续商业线索判断。
- 历史候选、中标、中标结果和合同服务期记录是低摩擦宽筛入口；它们只能形成重叠线索和命中地区，不得单独证明未释放，也不得单独排除问题。
- 施工许可、竣工验收、合同备案、项目经理变更和停工/例外材料是命中重叠线索后的定向释放证据补查，不作为负责人未释放/履约重叠宽筛第一动作的全量扫描对象。
- 命中同负责人、同主体和时间窗口重叠后，先记为 `A_STRONG_TIME_OVERLAP_SIGNAL`；施工许可/合同/履约/人员公开记录命中记为 `B_ENHANCEMENT_OFFICIAL_READBACK`，竣工/变更/停工/例外材料命中记为 `C_REVERSE_EXPLANATION_OFFICIAL_READBACK`，源阻断、未公开、接口不可用、验证码/登录或查询未命中记为 `D_INSUFFICIENT_OR_BLOCKED_READBACK`。D 只代表释放证据不足，不降级 A。
- 原文回溯阻断先进入分层解阻队列，不直接终止：先做 URL 规范化和占位值识别；无 scheme 公开域名先补 `https://`，传输失败时允许降级尝试 `http://` 并保留两次 `route_attempts`；页面可达但正文不可抽取时标记 `original_notice_body_not_extractable_review`；JS 壳或跳转页进入浏览器 readback；YGP 原文指针进入 YGP readback；验证码、登录、风控或授权页面进入人工/挑战队列。上述步骤仍失败时才标 D/阻断/证据不足，且不得写“无问题”。
- 释放证据补查先按历史重叠项目所在省/市进入所在地住建局或住建主管部门公开源；MVP 默认只选这一类主源。非广东历史项目按归属地住建局适配器做，不得回退到广东/广州源。全国建筑市场监管公共服务平台只作身份/地区发现和辅助交叉验证，不作为 B/C/D 默认释放主源；广东省工程建设项目审批管理系统只作辅助或人工挑战路线。只有历史项目地区缺失时才回退当前候选项目地区，并在任务中标记 fallback。
- 无脑全省市扫描不是默认策略；只有全国索引失效、目标价值足够高且人工批准扩大范围时，才进入更宽的地区遍历。
- 第一批重点省份定向源目录为浙江、四川、江苏、湖北、山东、湖南、河南；这些目录只用于生成 `ActiveConflictProbe` 的公开源任务和 adapter 优先级，默认状态为 `PLAN_ONLY_UNTIL_REGION_ADAPTER_VERIFIED`，不能当作已完成 live 核验。
- `MajorRegionQueryProbe v1` 可以按省份生成任务和只读可达性诊断，默认不联网；即使入口页可达，也只说明后续 adapter 有入口，不代表施工许可、合同、竣工或项目经理变更字段已经核验成功。
- 只拿到广东住建或广东公共资源结果时，`current_region_only_is_insufficient=true`，只能 review，不能判无冲突。
- 四库/JZSC 的人员项目页和企业项目页只能作为身份消歧、历史项目和 `08` 声明业绩匹配的辅助来源；由于全国平台由地方逐级推送，存在同步滞后，不能作为实时在建冲突唯一依据。
- 在建/履约冲突必须优先结合地方公共资源公告、地方住建/行政审批施工许可、合同备案、竣工验收、项目经理变更、处罚/投诉处理等地方或网络公开线索。

**异议/投诉证据包口径**
- 对外异议或投诉材料需要事实、请求、有效线索和证明材料；系统内部只能生成证据线索包、公开源 readback、缺口和建议核验事项。
- 证据包至少包含当前项目 07 证据、候选组绑定、公开注册信息匹配/不匹配、可能占用项目的中标/许可/合同/履约记录、释放证据尝试、时间窗口表和招标文件无在建条款。
- 缺少释放证据时，建议表述为“未发现可回放释放材料，建议招标人或监管部门核验”，不得表述为“已确认未释放”。
- 正式证据包在进入客户交付前必须具备来源 URL、采集时间、访问路径、脚本版本、snapshot/readback、SHA-256/hash、字段血缘、脱敏日志、禁止词检查和反向解释；可信时间戳、存证编号或公证未接入时只能作为待增强项，不得包装成已完成正式存证。
- 交付内容只能是事实、线索、证据、反向解释和建议核验事项。不得做收费举报人、付费沉默、爆料号、公开点名、使用内部泄露材料或 AI 一键定性。

## 14. 推进方式

下一轮不直接一次性实现全部 Stage4/5。按最小可验证分支推进：

1. 先选 `项目经理企业内消歧 + 注册时间核验`。
2. 真实页面验证四库企业页、人员列表、分页、人员详情可行路径。
3. 把验证结果从 `USER_FIELD_EXPERIENCE` 升级或降级。
4. 再补代码 target generation / fetch / parser / readback。
5. 再补 Stage5 rule/evidence gate 测试。

如果真实页面路径与本规程冲突，先修本规程，不把错误路线硬塞进代码。
