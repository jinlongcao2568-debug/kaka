## 0. 总目标

本清单解决的不是“想法是否完整”，而是以下问题：

1. 是否已经形成唯一上位母版与唯一裁决源；
2. 是否已经补齐 D1-D14 文档层；
3. 是否已经补齐与文档一一对应的 contracts 机器契约层；
4. 是否已经补齐 stage1-stage9 的 handoff 合同、schema、example 与集成矩阵；
5. 是否已经补齐 golden、回归、发布前检查与治理校验；
6. 是否已经补齐 coverage、field policy、delivery matrix、release gates 等治理与放行资产；
7. 是否已经补齐 owner、当前任务、审批链与例外链等执行控制面；
8. 是否已经达到“可以开始写正式业务代码”的 Go 条件。

## 1. Go / No-Go 总门槛

### 1.1 必须全部满足的 Go 条件

只有以下条件全部成立， 才可进入正式业务代码开发：

- 只有一个 L0 上位母版；
- 只有一个正式裁决索引面；
- D1-D14 文档层已经全部成形，且职责边界清楚；
- 每份 D 文档都能找到唯一 machines contracts 承接面；
- stage1-stage9 的 handoff 合同已明确，至少 stage1-stage6 不再依赖口头承接；
- golden、fixture、回归 manifest、发布前脚本已能覆盖真实主链，而不只是治理骨架；
- coverage、public boundary、field policy、delivery matrix、release gates、approval chain 已齐备；
- owner、current task、例外链、审批链为真实值，而非 seed / tbd 占位；
- 没有缺失引用、失效链接、空目录被误判为 ready 的情况；
- README / 状态板、文档、contracts、scripts 之间没有双口径。

### 1.2 任一成立即 No-Go 的阻断项

以下任一项成立，必须判定为 No-Go：

- 并存第二个 L0 母版、第二个 D13 正式口径或第二套对象体系；
- `project_fact` 不是唯一统一事实中枢；
- 双闸门未冻结或缺一仍可升级正式主结论；
- stage1-stage6 handoff 仍无正式 schema / example / 校验入口；
- D8 / D9 / D10 / D12 / D14 缺失，却仍宣称“全套已准备完毕”；
- tests / golden / release 脚本只验证治理控制面，未验证业务主链；
- owner、审批、例外、治理状态仍为占位值；
- 页面 / API / 导出存在绕开正式对象重算主判断的可能；
- contracts 与 docs 已出现双口径。

## 4. contracts 机器契约层检查

### 4.1 必备目录

- `contracts/schemas/`
- `contracts/enums/`
- `contracts/rules/`
- `contracts/gates/`
- `contracts/api/`
- `contracts/ui/`
- `contracts/governance/`
- `contracts/release/`
- `contracts/exceptions/`
- `contracts/testing/`
- `contracts/sales/`（承接 D8 / D9 / D10 的商业对象与触达对象）

### 4.2 必查项

- 每份 D 文档是否有唯一 catalog / schema 承接面；
- schema 是否标注 `stage`、`layer`、`required`、`sourceL0Refs`；
- enum 是否统一引用，不存在平行枚举；
- rule catalog 是否完整声明规则码、默认结果、最低证据等级、公开边界层级；
- gate policies 是否完整声明 PASS / REVIEW / BLOCK、升级 / 降级规则与全局优先序；
- api catalog / permission matrix / error codes 是否齐备；
- ui workbench / button flow / export template 是否齐备；
- governance contracts 是否包括 coverage、public boundary、field policy、approval chain；
- release contracts 是否包括 delivery matrix、release gates；
- exceptions contracts 是否包括受控例外与治理补证策略；
- testing contracts 是否包括 L0 assertions、golden cases、regression manifests。

### 4.3 完成标准

- 不存在“文档已写、contracts 未落”的正式资产；
- 任一实现者能直接从 contracts 找到机器可读定义；
- scripts 能基于 contracts 做自动化校验，而不是基于手工约定。

## 12. 最终结论页

### 12.1 允许开工

只有当以下结论成立时，方可写正式业务代码：

- 文档全套齐；
- contracts 全套齐；
- handoff 全套齐；
- testing / governance / release 资产齐；
- 控制面齐；

### 12.2 不允许开工

以下情况一律不允许宣称“全套准备完成”：

- 只补 md，不补 contracts；
- 只补治理目录，不补业务主链 handoff；
- 只跑格式校验，不跑 golden / regression；
- 只写阶段 1-6，不补 D8-D10 却宣称“全闭环已准备好”；
- 只补页面 / 接口描述，不补 `project_fact` / gate objects / release assets；
- 仍存在占位 owner、占位 task、空目录、失效链接、第二套口径。
