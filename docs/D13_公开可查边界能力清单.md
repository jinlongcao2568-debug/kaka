## [D13-R-001] 0. 文档任务与裁决范围

本文档不是发散罗列“系统可以做什么”，也不是销售能力海报，更不是实现层为了方便展示而拼出的功能清单。

本文档的正式任务，是把  在中国公开证据边界下的建设工程域能力，收敛成一套可裁决、可落库、可测试、可回归、可对外约束、可被 AI 代理严格执行的正式能力边界。

本文档必须回答以下问题：

1. 哪些能力属于全国稳定公开可得的正式主能力；
2. 哪些能力属于建设工程域增强能力，但必须受 coverage、平台成熟度与证据等级约束；
3. 哪些能力只允许作为条件性线索、观察项或复核请求；
4. 哪些能力绝对必须先完成治理复核后按授权范围进入正式链路；
5. 每项能力属于哪一阶段、消费哪些正式对象、允许输出什么结果语义、受到哪些 gate 与治理状态约束；
6. 页面、接口、导出、销售与交付在消费这些能力时，哪些表达允许，哪些表达一票禁止；
7. 仓库中的 `public_boundary_registry`、`coverage_registry`、D11 金标与回归脚本应如何与本文保持单一口径。

本文档不直接展开以下内容：

- 正式对象的字段级定义，由 D2 展开；
- 规则码、证据等级与双闸门细则，由 D3 展开；
- 页面、接口、字段策略与对象级交付矩阵，由 D4、D5、D6、D7 展开；
- 测试断言、金标样例、发布前检查，由 D11 展开。

但上述所有文档都不得改写本文对公开能力层级、默认结果上限、治理复核能力与对外承诺边界的定义。

## [D13-R-002] 1. 生效规则与对齐原则

### [D13-R-003] 1.1 生效规则

- 本文是 L0 在“公开能力边界与正式能力承诺”维度的正式展开文档。
- 本文只允许细化上位总纲，不允许突破总纲。
- 本文与 L0 冲突时，以 L0 为准。
- 本文与 D2、D3、D4、D5、D6、D7、D11 冲突时，以 L0 为准，再以本文解释能力边界，不允许在下位文档中平行定义第二套能力分层。
- 页面、接口、导出、销售话术、演示稿、脚本、测试样例与临时台账，都不得脱离本文自行定义“系统还能做什么”。

### [D13-R-004] 1.2 能力边界对齐原则

所有正式能力都必须同时对齐以下七个维度：

1. **阶段归属**：能力位于阶段 1-9 中的哪个阶段；
2. **对象归属**：能力消费哪些正式对象，输出哪些正式对象；
3. **边界归属**：能力位于 A / B / C / D 中的哪一层；
4. **结果归属**：能力允许输出 `AUTO_HIT`、`CLUE`、`OBSERVATION`、`review_request` 中的哪几类；
5. **门禁归属**：能力受哪些双闸门、版本 / 时钟、coverage、delivery、release gate 约束；
6. **消费归属**：能力能否进入 `project_fact`、`legal_action_recommendation`、`saleable_opportunity`、`contact_target` 或正式交付物；
7. **测试归属**：能力必须映射到 D11 的正向、负向、降级、阻断样本。

### [D13-R-005] 1.3 能力层级不得改写的上位事项

本文不得改写以下事项：

- `project_fact` 是唯一正式统一事实中枢；
- `rule_gate_decision` 与 `evidence_gate_decision` 缺一不可；
- `saleable_opportunity` 不得由单条规则、单页公告或销售备注直接生成；
- `contact_target` 只能来源于公开、授权或合规接入的联系来源；
- 页面、接口、导出、销售不得形成第二套主判断体系。

## [D13-R-006] 2. 四层正式边界总定义

### [D13-R-007] 2.1 A 层：`A_PUBLIC_CORE`

A 层是全国稳定公开可得主能力。

A 层能力必须同时满足：

- 来源公开可查；
- 能稳定取回；
- 能回链；
- 能审计；
- 能进入正式对象链与正式回归样本；
- 在满足双闸门与上位门禁条件时，允许形成 `AUTO_HIT`、`CLUE` 或正式事实对象。

A 层能力可以作为全国最小正式能力承诺，但仍不得被表述成终局违法定性、自动废标、自动替代监管或司法裁判。

### [D13-R-008] 2.2 B 层：`B_PUBLIC_ENHANCED`

B 层是建设工程域公开增强能力。

B 层能力具有正式价值，但不属于全国无条件稳定能力。B 层能力只有在以下前提同时成立时，才允许进入正式主链：

- `coverage_governance_registry` 已登记并放行；
- 源族已进入 `sources_registry` 或等价正式源目录；
- `public_granularity_state`、`fixation_readiness_state`、`retrieval_stability_state` 达到最低阈值；
- 主体对齐、字段来源与证据等级达到相应阈值；
- 已进入 D11 的正反样本与降级样本。

B 层能力默认只允许输出 `CLUE`、`review_request` 或条件性事实对象，不得包装成“全国统一默认能力”。

### [D13-R-009] 2.3 C 层：`C_PUBLIC_CONDITIONAL`

C 层是条件性公开线索能力。

C 层能力只有在项目所在地区、项目公开页面、公开附件或公开材料颗粒度足够时才允许启用；其默认输出只允许是 `CLUE`、`OBSERVATION` 或 `review_request`。

C 层能力必须满足以下限制：

- 必须存在可回链公开载体；
- 必须显式保留条件语义与不确定性；
- 不得在颗粒度不足时升级；
- 不得被写成客户版首页稳定主结论；
- 不得被销售话术包装成全国默认能力。

## [D13-R-011] 3. 能力项统一描述模板

每一个正式能力项，至少必须说明以下字段：

- 能力 ID；
- 能力名称；
- 边界层级；
- 所属阶段；
- 输入对象；
- 核心处理；
- 输出对象；
- 默认结果语义；
- 可升级上限；
- 升级条件；
- 降级条件；
- 阻断条件；
- 禁止事项；
- 依赖治理条件；
- D11 测试要求；
- 机器承接条目。

允许的展开方式只有两种：

1. **逐项完整展开**：A 层能力、B/C 交界高风险能力、首批必须开工能力、面向正式页面 / API / 导出 / 销售承诺的能力，必须逐项完整展开；
2. **能力族统一模板承接**：同构的 B 层或 C 层能力，可以在能力族层统一展开，但该模板必须同时说明输入对象、核心处理、输出对象、默认结果、升级 / 降级 / 阻断条件与治理前提。

任何能力只要存在独立法律边界、独立商业承诺边界、独立治理资产依赖或独立禁止事项，都不得只列名称而不落模板。

## [D13-R-012] 4. A 层正式能力清单

### [D13-R-013] A-01 公开链完整性核验

- **边界层级**：`A_PUBLIC_CORE`
- **所属阶段**：阶段 2、阶段 6
- **输入对象**：`task_record`、`execution_context`、`public_chain`、`clock_chain_profile`、`notice_version_chain`
- **核心处理**：识别当前制度下法定公开节点是否齐全、节点时序是否成立、缺失节点是否影响后续动作与事实聚合
- **输出对象**：`public_chain`、`project_fact`、`legal_action_recommendation`
- **默认结果语义**：正式事实对象 / `CLUE`
- **可升级上限**：`AUTO_HIT`
- **升级条件**：法定节点完整、版本与时钟无核心未解冲突、原始载体可回链、来源可审计
- **降级条件**：存在节点缺失但不影响当前仅内部使用场景，或存在待人工确认的轻微冲突
- **阻断条件**：关键法定节点缺失、时钟链未解、版本链冲突未解、公开链不可固定
- **禁止事项**：不得在缺失关键节点时继续输出正式外发主结论或正式动作建议
- **依赖治理条件**：无额外 coverage 前提，但受 D3 双闸门与 D11 阻断样本约束
- **D11 测试要求**：必须覆盖节点完整 / 节点缺失 / 时钟冲突 / 版本冲突四类样本
- **机器承接**：`public_boundary_registry.A-01`

### [D13-R-014] A-02 公示期与动作窗口门控

- **边界层级**：`A_PUBLIC_CORE`
- **所属阶段**：阶段 1、阶段 2、阶段 6
- **输入对象**：`clock_chain_profile`、`public_chain`、`project_fact`
- **核心处理**：识别当前公示窗口、异议窗口、投诉窗口或等价动作窗口是否仍然成立
- **输出对象**：`clock_chain_profile`、`legal_action_recommendation`
- **默认结果语义**：正式事实对象 / `CLUE`
- **可升级上限**：正式建议对象
- **升级条件**：制度类型明确、窗口状态明确、主管机关边界明确
- **降级条件**：窗口边界存在待确认事项，但仍可形成内部复核与观察结论
- **阻断条件**：制度类型不明、窗口状态不明、时钟冲突未解
- **禁止事项**：不得把窗口已失效项目包装成仍可正式发起动作
- **依赖治理条件**：依赖 D2 中 `clock_chain_profile` 完整，依赖 D3 双闸门与制度裁决
- **D11 测试要求**：必须覆盖窗口有效、窗口失效、窗口不明、时钟冲突四类样本
- **机器承接**：`public_boundary_registry.A-02`

### [D13-R-015] A-03 评标方法与项目类型分型

- **边界层级**：`A_PUBLIC_CORE`
- **所属阶段**：阶段 3、阶段 5
- **输入对象**：`project_base`、`field_lineage_record`
- **核心处理**：识别 `procurement_regime`、`procurement_category`、`bid_eval_method`、`candidate_order_mode`、`award_determination_mode`
- **输出对象**：`project_base`、`rule_gate_decision`、`project_fact`
- **默认结果语义**：正式事实对象
- **可升级上限**：正式事实对象
- **升级条件**：字段来源可追溯、制度口径明确、字段归一完成
- **降级条件**：字段缺失、术语存在待确认映射但不影响内部观察
- **阻断条件**：制度类型无法裁决，导致后续规则与动作口径无法成立
- **禁止事项**：不得混用工程招投标与政府采购动作口径，不得为了套用规则伪造候选排序关系
- **依赖治理条件**：依赖 D2 真相层字段与 D3 规则前提目录
- **D11 测试要求**：必须覆盖制度明确 / 制度歧义 / 不排序候选 / 评定分离等样本
- **机器承接**：`public_boundary_registry.A-03`

### [D13-R-016] A-04 候选集合攻击面核验

- **边界层级**：`A_PUBLIC_CORE`
- **所属阶段**：阶段 4、阶段 6
- **输入对象**：`project_base`、`bidder_candidate`、`public_chain`
- **核心处理**：形成核心候选集合、识别排序 / 不排序关系、固化目标候选人的公开攻击面
- **输出对象**：`public_attack_surface`、`project_fact`
- **默认结果语义**：正式事实对象 / `CLUE`
- **可升级上限**：正式事实对象
- **升级条件**：候选集合完整、项目归一完成、公开链成立
- **降级条件**：候选集合存在轻微缺口，但仍可形成受限攻击面观察
- **阻断条件**：项目归一未完成、候选集合不完整且无法补齐、版本链未解
- **禁止事项**：不得为了套用规则伪造顺位、伪造核心候选集合或伪造攻击点
- **依赖治理条件**：依赖 D2 `bidder_candidate`、D3 版本 / 时钟 gate
- **D11 测试要求**：必须覆盖排序候选 / 不排序候选 / 伪顺位阻断样本
- **机器承接**：`public_boundary_registry.A-04`

### [D13-R-017] A-05 公开版本冲突检测

- **边界层级**：`A_PUBLIC_CORE`
- **所属阶段**：阶段 2、阶段 3、阶段 5
- **输入对象**：`notice_version_chain`、`field_lineage_record`、`project_base`
- **核心处理**：检测联合体成员、资格条件、项目负责人、关键字段前后版本冲突
- **输出对象**：`notice_version_chain`、`rule_gate_decision`、`review_request`
- **默认结果语义**：`CLUE` / `review_request`
- **可升级上限**：`CLUE`
- **升级条件**：版本链完整、冲突可回链、冲突对当前规则结果有明确影响
- **降级条件**：冲突存在但影响面尚不明确，需人工裁决
- **阻断条件**：版本链不完整、冲突无法解释却影响正式结论
- **禁止事项**：不得忽略版本冲突继续升级正式结论
- **依赖治理条件**：依赖 D2 `notice_version_chain` 与 D3 `PROC-002` 等版本冲突规则
- **D11 测试要求**：必须覆盖可解释冲突 / 不可解释冲突 / 无冲突样本
- **机器承接**：`public_boundary_registry.A-05`

### [D13-R-018] A-06 限制性条款与规则定向识别

- **边界层级**：`A_PUBLIC_CORE`
- **所属阶段**：阶段 4、阶段 5
- **输入对象**：`project_base`、`field_lineage_record`、资格条款相关画像对象
- **核心处理**：识别疑似限制性资格条件、参数定向、明显不当门槛与条款性打击点
- **输出对象**：`rule_hit`、`evidence`、`project_fact`
- **默认结果语义**：`CLUE`
- **可升级上限**：`AUTO_HIT`
- **升级条件**：文本回链充分、前提完整、双闸门通过、证据达到目标外显阈值

## [D13-R-041] 9. 对外表达约束

### [D13-R-042] 9.1 允许表达

允许使用的正式表达包括：

- “基于公开可回链信息形成的正式核查摘要”；

- 把 B 层能力写成“全国所有地区稳定默认能力”；
- 把 C 层线索写成“已坐实”“当然废标”“当然违法”或终局结论；
- 把演示版、预览版、截图摘要、口头说明冒充正式交付物；
- 把 `AUTO_HIT` 写成自动认定围标串标、自动判断中标无效或自动替代监管 / 司法结论。

## [D13-R-044] 10. 页面、接口、导出、销售一致性要求

### [D13-R-045] 10.1 页面

- 页面顶层主判断只能消费允许进入正式公开主链的对象；
- `OBSERVATION` 不得作为客户版首页主结论；
- `review_request` 不得伪装成“已成立结论”。

### [D13-R-046] 10.2 接口

- API 只能返回 A / B / C 范围内已放行字段；

## 附录：统一 capability_mode 运行态补表（本轮）

本补表用于在既有 `A/B/C/D` 能力边界层与 `release_layer` 放行层之外，新增统一运行态维度 `capability_mode`；不改写既有 release 词表，也不放宽 `EXTERNAL_CONTROLLED_OPENING` 受控开放要求。

### A. capability_mode 正式词表

| capability_mode | 正式含义 | 是否可直接提升 release_layer |
|---|---|---|
| `PERMANENTLY_BLOCKED` | 永久阻断；当前与可预见批次内都不得启用 | 否 |
| `BUILDABLE_BUT_OFF_BY_DEFAULT` | 可建设、可接线、可验证，但默认关闭 | 否 |
| `INTERNAL_ONLY` | 仅内部可用，不进入对外消费面 | 否 |
| `INTERNAL_GOVERNED` | 内部可用，但必须受审批/审计/字段治理/对象治理约束 | 否 |
| `APPROVAL_REQUIRED` | 每次启用都必须先过人工审批链 | 否 |
| `SHADOW_MODE` | 只允许旁路观测、比对、解释或辅助，不得驱动主决策/主执行 | 否 |
| `DRY_RUN` | 只允许模拟执行与预留回执，不触发真实动作 | 否 |
| `REAL_RUN_READY` | 运行准备已具备，但仍需 release gate、审批链与审计链共同放行 | 否 |
| `EMERGENCY_OFF` | 紧急停机；高于普通 capability_mode，立即覆盖运行态 | 否 |

### B. 与 A/B/C/D、release_layer 的关系

- `A/B/C/D`：回答“能力边界性质是什么”。
- `release_layer`：回答“当前允许被谁消费/交付到哪一层”。
- `capability_mode`：回答“当前运行态如何执行、是否默认关闭、是否只允许 shadow/dry-run/approval”。
- `capability_mode` 不能替代 `release_layer`，也不能提升 `release_layer`。
- `PERMANENTLY_BLOCKED` 与 `EMERGENCY_OFF` 对运行态具有覆盖优先级，但**不能**把 `EXTERNAL_CONTROLLED_OPENING` 降级成可外放。
- `REAL_RUN_READY` 只表示“运行准备就绪”，**不等于** 已经允许 external release / external platform。

### C. 能力族收口视图

| 能力族 | 当前 capability_mode | 允许的运行态范围 | release_layer ceiling | 说明 |
|---|---|---|---|---|
| external source | `INTERNAL_ONLY` | `INTERNAL_ONLY / INTERNAL_GOVERNED / SHADOW_MODE` | `INTERNAL_OPERABLE` | 官方公开源可内部消费，不改成外部软件能力 |
| contact enrichment | `APPROVAL_REQUIRED` | `BUILDABLE_BUT_OFF_BY_DEFAULT / APPROVAL_REQUIRED / SHADOW_MODE / DRY_RUN` | `INTERNAL_OPERABLE` | 可建设但默认不直接放行 |
| execution vendor | `DRY_RUN` | `BUILDABLE_BUT_OFF_BY_DEFAULT / DRY_RUN / APPROVAL_REQUIRED / REAL_RUN_READY / EMERGENCY_OFF` | `INTERNAL_OPERABLE` | 不放开真实外部执行受控开放要求 |
| model provider | `BUILDABLE_BUT_OFF_BY_DEFAULT` | `BUILDABLE_BUT_OFF_BY_DEFAULT / INTERNAL_ONLY / SHADOW_MODE / INTERNAL_GOVERNED / EMERGENCY_OFF` | `INTERNAL_OPERABLE` | 只从“硬封死”解放到可建设，不等于放行 |
| tool provider | `SHADOW_MODE` | `BUILDABLE_BUT_OFF_BY_DEFAULT / SHADOW_MODE / INTERNAL_GOVERNED / EMERGENCY_OFF` | `INTERNAL_OPERABLE` | 默认只能旁路辅助 |
| Stage 8 execution | `DRY_RUN` | `DRY_RUN / APPROVAL_REQUIRED / REAL_RUN_READY / EMERGENCY_OFF` | `INTERNAL_OPERABLE` | 当前只允许模拟/审批前置 |
| Stage 9 execution | `SHADOW_MODE` | `BUILDABLE_BUT_OFF_BY_DEFAULT / SHADOW_MODE / DRY_RUN / APPROVAL_REQUIRED / REAL_RUN_READY / EMERGENCY_OFF` | `INTERNAL_OPERABLE` | 当前仍不允许真实支付/交付执行 |
| delivery/export variants | `INTERNAL_GOVERNED` | `INTERNAL_ONLY / INTERNAL_GOVERNED / APPROVAL_REQUIRED / DRY_RUN / EMERGENCY_OFF` | `LEADPACK_DELIVERABLE` | 只管运行态，不改 release 层 |
| risky automation | `APPROVAL_REQUIRED` | `APPROVAL_REQUIRED / SHADOW_MODE / DRY_RUN / EMERGENCY_OFF` | `INTERNAL_OPERABLE` | 高风险自动化默认不开 |
| emergency off | `EMERGENCY_OFF` | `EMERGENCY_OFF` | `EXTERNAL_CONTROLLED_OPENING` | 一键停机，不削弱 blocked 受控开放要求 |

### D. 受控开放要求保持说明

- `capability_mode` 新增后，`external controlled-opening-required` 总口径保持不变。
- `BUILDABLE_BUT_OFF_BY_DEFAULT` 只代表“工程上可以建设”，不代表“业务上默认打开”。
- `SHADOW_MODE` / `DRY_RUN` / `APPROVAL_REQUIRED` 只代表运行态，不代表可以绕过 D6/D7/D11/D12/D14。
