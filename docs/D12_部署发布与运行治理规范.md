## [D12-R-001] 0. 文档任务

本文档解决以下问题：

1.  在什么环境、什么批次、什么门禁下允许部署与发布；
2. 什么叫正式发布包，什么叫受限发布包，什么叫禁止发布；
3. 发布前必须校验哪些 contracts、测试、治理快照与审批链；
4. 线上运行时如何做监控、告警、降级、冻结、回滚与恢复验证；
5. 事故、例外、人工干预与运行治理如何留痕，并如何回写到正式审计链；
6. 如何保证运行层只消费正式 release 资产，而不反向改写对象、规则、页面、交付与模型治理口径。

本文档不是基础设施手册，不展开云资源购买、容器镜像构建细节、CI 平台具体按钮、具体监控面板实现命令。

## [D12-R-002] 1. 运行治理总原则

### [D12-R-003] 1.1 单向承接原则

- 部署与发布只能消费 L0、D4、D5、D6、D7、D11、D13、D14 已冻结的正式口径。
- 运行层不得通过热修、环境变量、隐藏开关、脚本覆盖或临时数据库变更，反向改写正式对象边界、结果语义、公开边界或交付矩阵。
- 任何运行层例外都只能降级、暂停、隔离或限制能力，不得提升 D 层输入为正式公开依据。

### [D12-R-004] 1.2 正式发布先于正式流量原则

- 未形成正式发布包，不得引入正式流量。
- 未通过 release gate，不得把 INTERNAL 或 DEMO 面向 CLIENT_DELIVERY、EXTERNAL_ACTION_ASSIST 或 EXTERNAL_API。
- 未完成回滚方案与恢复验证，不得进入生产级别放量。

### [D12-R-005] 1.3 更严格者优先原则

发生以下口径冲突时，以更严格者为准：

1. L0 与 D 文档冲突，以 L0 为准；
2. D7 交付矩阵与 D6 字段策略冲突，以更严格者为准；
3. D11 测试通过但 D7 / D13 / D14 治理阻断仍在时，以阻断为准；
4. 环境开关允许但正式 release gate 未放行时，以 release gate 为准。

## [D12-R-006] 2. 正式环境矩阵

| 环境 ID | 正式名称 | 用途 | 允许数据 | 允许发布形态 | 禁止事项 |
|---|---|---|---|---|---|
| `LOCAL_DEV` | 本地开发环境 | 开发、自测、契约联调 | 脱敏样例、最小测试数据 | `INTERNAL_ONLY` | 不得接入真实客户交付链 |
| `INTEGRATION` | 集成联调环境 | 跨模块联调、契约验证、回归预跑 | 脱敏样例、治理测试数据 | `INTERNAL_ONLY` | 不得承载正式客户交付 |
| `PRE_RELEASE` | 预发布环境 | release 检查、灰度前验证、最终门禁 | 脱敏样例、受控镜像快照 | `INTERNAL_ONLY`、受限 `CLIENT_VISIBLE` 验证 | 不得对外提供正式服务 |
| `PRODUCTION_INTERNAL` | 生产内部环境 | 正式内部运行、治理后台、内部复核 | 正式运行数据 | `INTERNAL_REVIEW` | 不得直接替代客户交付面 |
| `PRODUCTION_CLIENT` | 生产客户环境 | 客户版报告、客户投影、对外 API | 正式运行数据 | `CLIENT_DELIVERY`、受控 `EXTERNAL_API` | 未过 release gate 不得放行 |
| `PRODUCTION_ACTION` | 外发动作准备环境 | 正式外发异议辅助、动作包准备 | 正式运行数据 | `EXTERNAL_ACTION_ASSIST` | 未达 `EXTERNAL_ACTION_READY` 不得进入 |
| `DR_STANDBY` | 灾备待命环境 | 灾备切换、恢复演练 | 正式运行镜像与备份 | 与主生产一致但默认冻结 | 不得长期脱离主配置漂移 |

### [D12-R-007] 2.1 环境准入规则

- `PRODUCTION_CLIENT`、`PRODUCTION_ACTION`、`DR_STANDBY` 必须登记在 `runtime_inventory.yaml` 中。
- 任何未登记环境，不得被宣称为正式环境。
- 正式环境的配置差异必须可审计、可回滚、可比对。

## [D12-R-008] 3. 发布批次与运行放行关系

| 批次 | 允许发布的正式能力 | 最低前提 | 不得发布的能力 |
|---|---|---|---|
| `BATCH-1` | 文档、contracts、testing、脚本 | L0 / D1 / D2 / D3 / D11 可用 | API / 页面 / 商业闭环 |
| `BATCH-2` | 正式接口目录与页面消费契约 | D4 / D5 生效、对象与 gate 稳定 | 真正对外业务执行流 |
| `BATCH-3` | 字段治理、交付矩阵、公开边界、release gates | D6 / D7 / D13 生效 | 自动外发、正式触达 |
| `BATCH-4` | 真实竞争者、可售对象、联系对象、订单交付规范 | D8 / D9 / D10 生效 | 超出当前批次声明的全能力承诺 |
| `BATCH-5` | 正式业务代码与正式运行链路 | 前四批全部通过、回归与治理门禁通过 | 缺失 release 资产的灰度上线 |

### [D12-R-009] 3.1 批次门禁硬约束

- 不得在低批次环境中宣称高批次能力已正式上线。
- 任何保留态路径必须明确返回 `CAPABILITY_NOT_EFFECTIVE` 或等价正式错误。
- 批次状态变更必须触发 release gate 与治理回归。

## [D12-R-010] 4. 正式发布包定义

### [D12-R-011] 4.1 正式发布包最小组成

每一次正式发布必须形成 `release_bundle`，至少包含：

1. `release_manifest.yaml`
2. 版本号、提交号、构建号、时间戳
3. 环境目标与批次声明
4. `contracts` 快照引用
5. `testing` 快照引用
6. `governance` 快照引用
7. `release_gates` 判定结果
8. 回滚方案与恢复验证步骤
9. 审批链与操作人记录
10. 风险摘要与限制说明

### [D12-R-012] 4.2 正式发布包分类

| 发布包类型 | 正式含义 | 适用环境 | 必要说明 |
|---|---|---|---|
| `INTERNAL_RELEASE` | 内部受控发布 | `INTEGRATION`、`PRE_RELEASE`、`PRODUCTION_INTERNAL` | 不得自动对外开放 |
| `CLIENT_RELEASE` | 客户版正式发布 | `PRODUCTION_CLIENT` | 必须满足 `CLIENT_VISIBLE` |
| `ACTION_RELEASE` | 外发动作辅助正式发布 | `PRODUCTION_ACTION` | 必须满足 `EXTERNAL_ACTION_READY` |
| `HOTFIX_RELEASE` | 受控紧急修复发布 | 生产相关环境 | 必须附带例外原因、有效期与回收计划 |
| `ROLLBACK_RELEASE` | 回滚发布 | 所有正式环境 | 必须记录触发原因与恢复验证 |

## [D12-R-013] 5. 发布前检查清单

### [D12-R-014] 5.1 必须通过的正式检查

每次正式发布前，至少必须通过：

1. `validate-contracts`
2. `run-golden`
3. `run-governance-contracts`
4. `check-release`
5. 目标环境配置一致性检查
6. release manifest 完整性检查
7. 回滚资产完整性检查
8. 目标批次能力是否已正式生效检查

### [D12-R-015] 5.2 一票阻断项

以下任一项成立时，发布必须阻断：

1. D 层输入试图进入正式公开主链；
2. `project_fact`、双闸门、交付矩阵、字段治理、公开边界快照不一致；
3. 客户版或外发版所需 release level 不满足；
4. 回滚路径不存在或未验证；
5. 生产环境配置未登记或与快照不一致；
6. 热修试图绕开正式审批链；
7. `D8 / D9 / D10 / D14` 未生效却试图发布对应正式能力；
8. 文档、contracts、testing、runtime 配置四者任一脱节。

## [D12-R-016] 6. 灰度、冻结、降级与回滚

### [D12-R-017] 6.1 灰度放量规则

- 正式新版本必须支持最小灰度放量。
- 灰度期间必须可追踪：版本、流量比例、启用对象、阻断计数、错误码变化、回滚阈值。
- 灰度不得绕开 release gate，仅改变流量比例，不改变正式能力边界。

### [D12-R-018] 6.2 冻结规则

以下情形必须进入发布冻结：

- 正式事故未完成根因定位；
- 外发或客户版发生高限制字段泄露；
- D 层渗透、错误升级、错误触达等一票失败情形发生；
- 当前版本与 release snapshot 不一致。

### [D12-R-019] 6.3 降级规则

降级只允许做以下动作：

- 关闭非核心保留态或增强能力；
- 将 `CLIENT_DELIVERY` 降回 `INTERNAL_REVIEW`；
- 将自动流转改为人工复核；
- 关闭高风险接口、按钮、导出模板；
- 将环境切换到只读 / 受限服务模式。

降级不得：

## 附录：runtime governance guards 承接补表（本轮）

- 运行治理层现在至少要同时校验两类守卫：
  1. capability permission layer
  2. governance runtime guards（field policy / delivery matrix / release gates / usage policy）
- `check-release` 不仅要验证 contracts 和脚本存在，还必须验证 Stage 8 / Stage 9 的 risky path 已把上述治理资产接进 runtime 主链。
- 若 capability permission 与 governance runtime guard 冲突，运行治理一律采用更严格者。
- `EMERGENCY_OFF / PERMANENTLY_BLOCKED` 继续保留最高优先级；runtime governance guards 不得削弱该短路语义。

- 修改正式对象定义；
- 修改正式结果语义；
- 把 D 层输入升格为正式依据；
- 静默掩盖事故。

### [D12-R-020] 6.4 回滚规则

- 每次正式发布必须有明确上一稳定版本的回滚目标。
- 回滚后必须执行恢复验证，至少验证：核心接口、核心页面、关键导出、关键 gate、关键 release policy、监控恢复。
- 回滚是正式发布动作，必须单独留痕。

## [D12-R-021] 7. 监控与告警

### [D12-R-022] 7.1 最小监控面

正式环境至少监控：

1. API 可用性与错误率
2. 关键 job 成功率
3. release gate 拒绝率
4. D 层渗透阻断计数
5. 导出失败率
6. 高限制字段访问与释放事件
7. 触达阻断与异常发送计数
8. 回写失败率
9. 审批链延迟
10. 配置漂移与未登记环境变更

### [D12-R-023] 7.2 告警等级

| 等级 | 正式含义 | 要求 |
|---|---|---|
| `SEV-1` | 正式外发 / 客户版高风险事故 | 立即冻结相关发布与高风险能力 |
| `SEV-2` | 正式主链功能不可用或结果失真 | 降级或回滚 |
| `SEV-3` | 局部能力失败或延迟明显 | 限时修复并跟踪 |
| `SEV-4` | 预警、漂移、趋势异常 | 进入工作队列 |

### [D12-R-024] 7.3 值班与响应

- 生产相关环境必须有值班责任链。
- `SEV-1`、`SEV-2` 必须形成正式事故记录。
- 无值班、无责任链的能力不得进入正式生产放量。

## [D12-R-025] 8. 事故、例外与运行审计

### [D12-R-026] 8.1 正式事故记录

事故记录至少包含：

- 事故 ID
- 触发时间、发现时间、恢复时间
- 影响环境与影响范围
- 影响对象与交付形态
- 根因分类
- 临时处置
- 回滚 / 降级动作
- 审计链与操作人
- 后续整改项

### [D12-R-027] 8.2 运行例外规则

- 运行例外必须登记为 `controlled_exception_record` 或等价正式资产。
- 运行例外只能临时限制或受控放行，不得永久化。
- 运行例外必须有有效期、回收计划、审批人、审计记录。

## [D12-R-029] 9. 配置与密钥治理

### [D12-R-030] 9.1 配置原则

- 生产配置必须分层：运行配置、治理配置、密钥配置、环境开关。
- 运行配置不得与 release snapshot 脱离。
- 未登记开关不得进入正式环境。

### [D12-R-031] 9.2 密钥原则

- 密钥不得写入仓库明文。
- 密钥轮换必须可追溯。
- 环境之间不得共用不必要的高权限密钥。

### [D12-R-032] 9.3 配置漂移阻断

- 配置差异必须有审计记录。
- 未登记漂移一律视为风险事件。
- 配置漂移达到正式边界影响阈值时，必须冻结发布。

## [D12-R-033] 10. 与其他正式文档的衔接

### [D12-R-034] 10.1 与 D7 的衔接

- D12 不定义 release level，本文件只消费 D7 已冻结的 release gate 与交付矩阵。
- 运行层不得绕开 D7 放行本不允许的交付形态。

### [D12-R-035] 10.2 与 D11 的衔接

- D12 不定义功能正确性测试标准，只消费 D11 的测试通过结论。
- 未通过 D11 必要检查，不得进入正式发布。

### [D12-R-036] 10.3 与 D13 的衔接

- D12 不修改 A/B/C/D 能力边界。
- 运行层只能根据 D13 放行、受限或阻断相应能力，不得扩边。

### [D12-R-037] 10.4 与 D14 的衔接

- 若模型输出进入正式链路，运行放行必须同时满足 D14 的模型治理门禁。
- 模型版本变更属于正式发布影响项，必须进入 release manifest 与回归检查。

## [D12-R-038] 11. 机器承接规则

- `contracts/release/deployment_matrix.json`：环境 × 发布包 × 交付形态矩阵
- `contracts/release/runtime_policy_catalog.json`：灰度、冻结、降级、回滚策略目录
- `contracts/testing/release_checklist.json`：发布前检查、恢复验证清单
- `control/release_manifest.yaml`：每次正式发布的快照索引面
- `control/runtime_inventory.yaml`：正式环境、责任链、批次状态清单

任何运行规则若无法映射到上述机器资产，不得视为正式运行规则。

### [D12-R-038A] 11.1 运行层级映射（local / internal_pilot / future_external）

| 运行层级 | 对应环境 | 发布允许 | 外部状态 |
|---|---|---|---|
| `local` | `LOCAL_DEV` | 仅 `INTERNAL_ONLY` | `EXTERNAL_BLOCKED` |
| `internal_pilot` | `INTEGRATION`、`PRE_RELEASE`、`PRODUCTION_INTERNAL` | `INTERNAL_ONLY` / `INTERNAL_REVIEW` | `EXTERNAL_BLOCKED` |
| `future_external` | `PRODUCTION_CLIENT`、`PRODUCTION_ACTION`、`DR_STANDBY` | 受控 `CLIENT_DELIVERY` / `EXTERNAL_ACTION_ASSIST` | `EXTERNAL_BLOCKED`（当前） |
