# 业务方向_候选公示后证据包与投前预测双线契约

## 1. 文档目的

本文件固定当前招投标分析产品方向，避免后续开发或代理把“抓招标公告验证下载能力”误当成最终业务主线。

当前系统必须按双线产品理解：

| 产品线 | 默认入口 | 业务定位 | 输出 |
|---|---|---|---|
| 候选公示后证据包分析 | `07 中标候选人公示` | 核心商业主线 | 控标/围标/串标/陪标线索、候选人核查、真实竞争者识别、违规线索证据包、销售承接 |
| 投前预测分析 | `02 招标文件公示`、`03 招标公告/关联公告`、`04 澄清答疑` | 辅助产品线 | 控标/定制标预测、废标风险、是否值得投、澄清/质疑建议 |

除非人类明确要求做投前预测，涉及“证据包、销售、候选人、真实竞争者、围标、串标、陪标、违规线索”的任务，默认走“候选公示后证据包分析”。

本文件同时固定 `AnalysisStrategyPlan v1` 的产品分流口径：系统不能因为发现很多文件就全部下载、全部解析、全部调用大模型。任何项目必须先判断产品线，再决定流程范围、下载范围、解析深度、规则核验和大模型介入条件。

两条路线的网页入口默认只抓工作日 72 小时内的近期公告：候选后证据包优先筛选近期 `07 中标候选人公示`；投前预测优先筛选近期 `02/03/04`。30 天或更长窗口只用于广州 `01-12` 流程接口覆盖、历史适配验证或人类明确指定的回溯/历史场景；不得把 30 天作为默认生产入口窗口。

## 2. 默认业务主线

### 2.1 候选公示后证据包分析

默认从近期 `07 中标候选人公示` 入手，而不是只抓刚发布的招标公告，也不是默认从中标结果、合同或项目异常入手。开标记录、评标结果、中标结果、合同和异常材料是回溯或支撑阶段，不是当前销售窗口的默认入口。

候选公示后主线不是“少解析”。它必须先全流程登记，再按证据价值分层解析。因为该产品线要卖的是候选后证据包、真实竞争者/真实买家线索和违规线索承接，系统必须尽量识别控标、围标、串标、陪标、护航报价、伪竞争者和真实受损竞争者。若这些指数和竞争者画像不清晰，就无法判断谁可能是真实买家，也无法形成可销售线索。

标准流程：

1. Stage1 优先发现 `candidate_notice`，即 `07 中标候选人公示`。`opening_record`、`evaluation_result`、`award_result` 只作为回溯、支撑或后期核验阶段。
2. 用项目 ID、项目名称、标段、采购人、代理机构、交易编号回溯同一项目全流程。
3. Stage2 把同一项目的全过程材料归入同一个 `project_id` 目录和审计对象。
4. Stage3 解析招标文件、答疑澄清、评分办法、技术参数、开标/候选/结果文本。
5. Stage4-6 形成内部事实、规则线索、多指数判断和证据缺口。
6. Stage7-9 才能在受控边界内承接可售机会、买家适配、线索包/证据包和销售推进。

必须回溯的项目阶段材料：

- 招标公告。
- 招标文件附件。
- 答疑、澄清、补遗、变更。
- 开标记录或开标结果。
- 评标结果、中标候选人公示。
- 中标结果公告。
- 投标文件公开、资格审查结果、否决投标原因。
- 图纸、清单、合同条款、评分办法、资格审查条件等附件。

近期 `07 中标候选人公示` 项目通常还不会出现 `11 合同信息公开` 和 `12 项目异常`。因此，近期候选项目缺 11/12 不应算作当前销售窗口的回溯失败；只需记录“后期阶段尚未产生”。如果 11/12 已经出现，通常说明项目已进入后期或历史复盘，应转为历史/后期核验场景，单独判断是否仍有销售价值。

候选后 `AnalysisStrategyPlan v1` 处理策略：

| 流程 | 处理要求 | 解析策略 |
|---|---|---|
| 01 招标计划 | 作为背景和时间线补充 | 通常 `METADATA_ONLY` |
| 02 招标文件公示 | 作为前置文件版本补充 | 有正文则 `TEXT_PROBE` 或 `SECTION_PARSE` |
| 03 招标公告/关联公告 | 必须回溯，作为资格、评分、参数基准 | `SECTION_PARSE`，必要时 `DEEP_PARSE` |
| 04 澄清答疑 | 必须回溯，判断是否改变竞争条件 | `SECTION_PARSE`，命中冲突时 `DEEP_PARSE` |
| 05 开标信息 | 必须结构化，识别投标人、报价、递交情况 | 规则结构化优先 |
| 06 资审结果公示 | 必须结构化，识别通过、未通过和否决原因 | 规则结构化优先 |
| 07 中标候选人公示 | 默认入口，必须结构化排名、报价、评分、候选人 | 规则结构化 + 冲突时 AI |
| 08 投标文件公开 | 必须登记全部投标文件，但不默认下载或解析 | 只记录存在、URL、附件清单、文件名和后续可解析目标；只有 `07 + Stage4` 缺字段、不匹配或需要核对 `08` 声明业绩时才定向下载/解析 |
| 09/10 中标结果/中标信息 | 结果核验和受益链确认 | 规则结构化优先 |
| 11 合同信息公开 | 近期候选项目通常尚未产生；有则作为成交、金额、合同主体和履约变化核验 | 当前销售窗口不强制；`METADATA_ONLY` 或命中冲突时解析 |
| 12 项目异常 | 近期候选项目通常尚未产生；有则必须记录，作为强后验线索 | 当前销售窗口不强制；规则结构化优先 |

候选后规则与 AI 分工：

- 规则优先：流程完整性、投标人数量、报价规律、候选排名、否决原因、文件归属、同源元数据、缺流程/缺附件、控标/围标/串标/陪标多指数初筛。
- AI 介入：招标文件要求与候选人响应是否冲突、高分候选人是否满足关键条件、多家投标文件是否文本高度相似、陪标/护航报价是否存在合理解释、否决理由是否前后矛盾。
- 投标文件公开通常体积较大，必须先登记文件清单、大小、hash、流程归属、发布日期和来源 URL；默认不下载、不解析。只有 `FLOW_08_TARGETED_PARSE_REQUIRED`、公开注册信息不匹配、缺证书号、证书类别/专业不足或需核对 `08` 声明业绩时才定向下载/解析。

### 2.1.1 项目负责人早取证策略

候选后证据包路线必须优先执行 `ResponsiblePersonEarlyProbe v1`。目标不是尽快全量解析所有投标文件，而是用最低成本尽早拿到 `候选公司 + 项目负责人/总监/设计负责人 + 证书号/注册编号 + 报价/排名`。只要已拿到可核验的证书号，就可以先进入 Stage4 公开核验；未拿到证书号时再逐层加深解析。

固定顺序：

1. 先解析 `07 中标候选人公示` 详情 HTML。广州 07 页面常直接包含候选公司、报价、排名、拟派项目负责人、总监理工程师、项目负责人资质。
2. 再解析 `07` 的小 PDF，包括 `中标候选人公示.pdf`、`定标候选人公示.pdf`、`评标报告-脱敏.pdf`、`复评报告.pdf`。优先用 MarkItDown 文本探针和规则抽取；只有文本为空且目标字段缺失时才进入 OCR。
3. `07` 多候选人必须按候选行抽取，不能把项目级公司池和项目级负责人池混配。每一行候选人包括完整联合体：牵头方、成员方、该行负责人、证书号、报价和排名必须作为同一候选组保存。
4. 如果 `07` 已有候选组、负责人姓名和证书号，直接生成 Stage4 输入：同一候选组的全部联合体成员都要进入核验目标；任一成员与负责人/证书注册单位匹配，即视为该候选组身份线索已解析，其他成员未匹配不得单独升级为冲突。
5. 如果 `07` 有候选公司和负责人姓名但没有证书号，先走 Stage4 公司优先补证：查候选公司或联合体成员，在企业人员列表中找该姓名，找到唯一匹配人员后回填证书号和人员公开 ID。
6. 如果公司优先找不到唯一匹配，再走姓名枚举兜底：用姓名枚举公开人员，逐个核对注册单位；只要注册单位匹配候选公司或联合体成员，即可回填证书号。
7. 如果同一候选组全部联合体成员经公司优先和姓名枚举都找不到匹配公司，不能直接写“冲突成立”，但必须提高风险等级，进入 `FLOW_08_TARGETED_PARSE_REQUIRED`。
8. `08 投标文件公开` 默认只登记存在、URL、附件清单和后续可解析目标，不默认下载或解析。触发定向解析时，优先找文件名包含资格、项目负责人、项目经理、总监、建造师、证书、商务、投标函的文件；抽到证书号后立即停止该文件解析并回到 Stage4 复核。
9. 如果 `08` 拿到证书号后仍无法匹配同一候选组任一候选公司或联合体成员，输出强风险线索：`项目负责人注册单位不一致线索`、`证书号回填后注册单位仍不一致，高风险线索`，进入 Stage5 证据门；仍不得输出法律定性。

### 2.1.2 公开注册信息匹配与冲突口径

负责人核验不能表述为“是不是本人”。系统只能基于公开记录判断“候选公示或 `08 投标文件公开` 中的姓名、候选公司/联合体成员、证书号、角色字段，是否与四库或地方注册人员公开记录匹配”。

固定口径如下：

1. `姓名 + 候选公司/联合体成员` 与公开注册人员记录中的姓名、注册单位匹配时，先认定该候选组的身份线索成立。
2. `证书号 + 注册单位 + 注册类别/专业` 也匹配时，身份线索为强匹配，可继续核证书有效期、状态、注册/变更时间和后续业绩/履约线索。
3. 公开记录中的证书号、注册单位、证书类别、注册专业、有效期或状态与候选公示/`08` 文件不一致时，才进入“注册信息不一致风险线索”。
4. 如果 `07` 已经通过公司+姓名或证书号匹配，不默认进入 `08`；只有 `07` 缺字段、四库/地方注册信息不匹配、证书类别/专业不足或需要补投标文件声明业绩时，才定向解析 `08`。
5. 四库/JZSC 的项目业绩信息只用于辅助核对 `08 投标文件公开` 中声明的业绩是否有公开记录；由于数据来自地方逐级推送，存在同步滞后，不能把“四库未出现”单独写成业绩造假或在建冲突。
6. 在建/履约冲突不以四库作为唯一实时依据。应优先结合地方公共资源候选/中标公示、地方住建或行政审批施工许可、合同备案、竣工验收、项目经理变更、监管处罚、投诉处理和网络公开线索；四库只作身份消歧和历史项目辅助。

推荐执行顺序：

1. `姓名 + 公司/注册单位` 直接查人员，优先验证是否匹配。
2. 公司搜索后进入人员列表，作为公司名称规范化和简称兜底。
3. 姓名-only 枚举公开人员，再按注册单位过滤，作为最后兜底。

上述三条路线都必须保留；单一路线失败不能直接进入 `08` 或冲突结论。

### 2.1.3 `EvidenceReport v1` 三类输出

广州 5 项目核验报告第一版只做内部报告，不扩大样本，不默认解析 `08`，不调用大模型。报告必须分三类：

1. 核验线索/证据：候选组、联合体成员、负责人、证书、注册单位、报价、排名、公开注册信息匹配状态、是否需要 `08` 定向解析。
2. 过程稳定性：`07` 抓取、`07` PDF/HTML 提取、Stage4 路线、公司优先/姓名枚举、`08` 是否存在但未解析、失败 taxonomy。
3. 优化建议：可直接销售、需补查、需 `08` 定向解析、需地方源适配、解析/下载稳定性建议。

`ActiveConflictProbe v1` 第一版只生成外部核验任务清单，来源包括地方公共资源历史候选/中标、地方住建/行政审批、施工许可、合同备案、竣工验收、项目经理变更、处罚/投诉和可回放网络线索；不得用四库/JZSC 单独给出实时在建/履约冲突结论。核验业绩、在建和履约冲突不能固定广东，应先用全国平台发现企业、人员和项目出现地区，再按重点省份公开源定向核验；当前重点省份目录先覆盖浙江、四川、江苏、湖北、山东、湖南、河南，全部按 `PLAN_ONLY_UNTIL_REGION_ADAPTER_VERIFIED` 处理，未完成 adapter 前不得当成 live 结果。

禁令：

- 不得因为 `07` 没有证书编号就直接判定负责人核验失败。
- 不得跨候选行混配候选公司、负责人和证书号；例如第二/第三候选人的公司不能拿来核验第一候选人的负责人。
- 不得只核验联合体牵头方；联合体成员也必须进入同一候选组的核验目标。
- 不得把姓名泛搜结果当作最终 PASS；必须核对注册单位、证书号、专业、人员公开 ID 和来源快照。
- 不得因为公司优先补证未命中就直接写“冲突成立”；必须先做姓名枚举兜底和 `08` 定向解析。
- 不得把保险、服务类或非工程角色强行归为施工项目经理；应按项目总负责人、服务负责人或不适用处理。
- 不得写“核实是不是本人”；只能写公开注册信息是否匹配。
- 不得把四库/JZSC 当成实时在建冲突唯一依据。

项目级审计必须回答：

- 这个项目的来源网址是什么。
- 项目 ID、项目名、标段是否稳定。
- 这个项目一共有多少详情页和附件。
- 下载了多少文件，多少文件可 replay。
- 解析了多少文件，哪些章节解析到位。
- 哪些阶段材料缺失，缺失原因是什么。
- 哪些文件可用于控标/围标/串标/陪标判断。
- 哪些只是后验线索，不能直接推高投前控标指数。

### 2.2 投前预测分析

投前预测线只在项目还没有 `05 开标信息`、没有 `07 中标候选人公示`，且人类明确要求“新公告/投前/是否值得投”时作为默认路径。默认入口只允许 `02/03/04`，不把采购意向或招标计划包装成正式预测销售。

投前预测是时效性产品，必须先看投标截止日期、开标日期和流程进度。若已经出现 `05 开标信息`，说明项目已进入开标后阶段，投前预测已经来不及，必须直接转入后期开标/候选后核验证据产品线。

输入材料：

- 02 招标文件公示。
- 03 招标公告/关联公告。
- 04 答疑、澄清、补遗。
- 资格预审文件。

若只有 `02/03`，还没有 `04 澄清答疑/补遗`，预测只能标记为 `PREDICTION_BEFORE_CLARIFICATION`，属于半成品预测。后续一旦出现 `04`、补充招标文件、答疑或变更，必须输出 `PREDICTION_RECALC_REQUIRED`，重新下载相关文件并做二次预测。出现 `04` 后，才进入 `PREDICTION_AFTER_CLARIFICATION`。

投前预测硬门槛：

1. 如果已经出现 `07 中标候选人公示`：不卖投前预测，直接走候选公示后证据包。
2. 如果已经出现 `05 开标信息`，即使还没有 `07`：不卖投前预测，输出 `PRE_BID_NOT_ELIGIBLE_OPENING_STARTED`，转入开标后核验/候选后跟踪池。
3. 如果没有 `05`，但投标截止日期或开标日期已经过去：不卖投前预测，输出 `PRE_BID_NOT_ELIGIBLE_DEADLINE_PASSED`。
4. 如果距离截止或开标不足 72 小时：不作为正常投前预测销售，输出 `PRE_BID_NOT_ELIGIBLE_TOO_LATE_FOR_SALE`，最多做内部紧急提示或转后验跟踪。
5. 如果距离截止或开标为 72-168 小时：只能进入 `PRE_BID_LIMITED_FAST_REVIEW`，适合快速风险扫描，不包装成完整投前预测。
6. 距离截止或开标 168 小时以上：才进入标准 `PRE_BID_STANDARD_PREDICTION_READY`。
7. 只有仍处在 `02/03/04`、截止/开标未过且时间窗口满足上述条件时，才进入对应级别的投前预测。

投前预测抓取失败处理：如果官网页面或附件人工可见，但系统抓取/下载失败，必须进入修复队列；但失败原因要分清楚是适配器代码问题、详情页传输阻断、附件挑战、TLS/WAF/代理问题、登录/CA 限制，还是平台同步延迟/无公开端点，不能把所有失败都写成代码 bug。

投前 `AnalysisStrategyPlan v1` 处理策略：

| 流程 | 处理要求 | 解析策略 |
|---|---|---|
| 01 招标计划 | 机会观察，不卖正式预测 | `METADATA_ONLY` |
| 02 招标文件公示 | 可提前预测，重点看资格、评分、技术、合同 | `SECTION_PARSE` |
| 03 招标公告/关联公告 | 投前预测主输入 | `SECTION_PARSE` |
| 04 澄清答疑 | 投前预测主输入，判断规则变化 | `SECTION_PARSE` |
| 05 开标信息 | 一旦存在即阻断投前预测 | 转后验核验，不走投前 |
| 06-12 | 不属于投前预测主输入 | 若存在则转后验 |

输出边界：

- 可以输出“控标风险线索、疑似定制标、限制竞争线索、废标风险、建议澄清/质疑/谨慎投”。
- 不能输出“已内定、违法成立、控标成立、必然废标”。
- 没有候选公示时，不能做候选人核查、真实竞争者识别、陪标组合判断或中标人受益链判断。

## 3. 抓取与分析策略

### 3.1 默认优先级

| 优先级 | 策略 | 说明 |
|---|---|---|
| P0 | 候选公示后项目全流程回溯 | 核心商业主线，优先用于证据包和销售承接 |
| P1 | 已开标但未出结果项目 | 可做开标/竞争者初判，等待候选公示补链 |
| P2 | 新招标公告投前预测 | 辅助产品线，用于是否值得投和控标预测 |
| P3 | 公告前机会 | 只做 watchlist，不直接形成证据包 |

### 3.1.1 `AnalysisStrategyPlan v1` 固定顺序

每个项目在下载和解析前必须先生成分析策略计划，固定顺序如下：

1. 先判断产品线：`POST_CANDIDATE_EVIDENCE_PACK`、`POST_OPENING_EVIDENCE_TRACK`、`PRE_BID_PREDICTION`、`WATCHLIST_ONLY`。
2. 再判断流程进度：是否存在 05、07、09/10；是否已过投标截止或开标时间。
3. 再决定文件动作：`SKIP`、`METADATA_ONLY`、`LIST_ONLY`、`DOWNLOAD_REQUIRED`、`PARSE_REQUIRED`。
4. 再决定解析深度：`NONE`、`TEXT_PROBE`、`SECTION_PARSE`、`DEEP_PARSE`。
5. 再决定规则/AI：规则先跑；只有规则冲突、高风险、多文件相似、候选响应冲突或解释空间较大时才触发大模型。

输出文件建议为 `analysis-plan.json`，每个 `project_id + flow_no + file_id` 至少说明：

- 所属产品线。
- 是否下载。
- 是否解析。
- 解析深度。
- 是否允许大模型。
- 用于哪些指数：控标、围标、串标、陪标、废标、真实竞争者、买家线索。
- 跳过或降级原因。

### 3.1.2 广州 01-12 流程适配器边界

广州 01-12 流程扫描是“适配器和接口形态验证”，不是生产爬取时固定搜索人类给出的几个网址。生产爬取应从候选公示/结果入口或投前入口进入，再通过广州列表 API、流程码、relationInfo、项目编号、项目名和阶段回溯发现同项目材料。

人类提供的 01-12 样例 URL 只能用于：

- 验证流程码、页面结构、附件入口、脚本下载端点、挑战/登录/CA 状态。
- 补齐低频流程的适配器测试覆盖。
- 生成人工核验表。

不得用于：

- 作为默认生产爬取目标。
- 冒充项目全流程回溯成功。
- 替代列表/API 发现逻辑。
- 作为真实样本数量或下载完整率统计。

### 3.2 三省独立适配

当前专业友好源优先级：

- 广东：只用广州交易集团/广州公共资源交易中心建设工程源。
- 浙江：独立适配浙江公共资源源。
- 四川：独立适配四川公共资源源。

其他省市先进入 `OBSERVATION_NEEDS_SOURCE_ANALYSIS`，需要确认省源还是城市源更完整，再做适配。

### 3.3 项目归档规则

所有真实抓取必须以项目为中心，而不是以样本数量为中心。

每个项目必须有：

- `project_id`。
- `project_name`。
- `source_urls`。
- `verification_urls.all_urls`。
- `file_inventory`。
- `detail_file_count`。
- `attachment_file_count`。
- `replayable_file_count`。
- `parsed_file_count`。
- `download_completeness_state`。
- `parse_completeness_state`。
- `failure_reasons`。

禁止：

- 用目标桶数量冒充真实项目样本数量。
- 用附件链接数量冒充可回放文件数量。
- 用详情页正文冒充完整招标文件。
- 用候选公示或中标结果冒充招标文件正文。
- 用非广州交易集团来源补广东工程建设样本。

## 5. 验收规则

候选公示后证据包主线的最小验收：

- 从近期 `07 中标候选人公示` 进入；评标结果、开标记录、中标结果是回溯/支撑阶段。
- 能回溯同一项目招标公告和招标文件。
- 每个项目有独立目录和 `project-file-audit.json`。
- 每个项目有候选人、竞争者、来源网址和附件清单。
- 每个附件必须可 replay 后才进入解析。
- 能区分投前预测线索和后验证据线索。
- 能输出控标、围标、串标、陪标、废标风险线索，但不输出法律定性。

投前预测线的最小验收：

- 从招标公告/招标文件进入。
- 招标文件、答疑、澄清、补遗可下载和解析。
- 能切出资格条件、评分办法、技术参数、废标条款。
- 能输出系统自动风险判断和证据缺口。
- 不能生成候选人核查或真实竞争者结论。

## 6. 后续代理执行规则

后续代理接到招投标采集、控标、围标、串标、陪标、证据包、销售线索相关任务时，必须先判断任务属于哪条产品线：

- 如果用户说“候选公示、证据包、候选人、真实竞争者、销售”，默认从近期 `07 中标候选人公示` 进入候选公示后证据包主线；评标结果、开标记录、中标结果、合同和异常是回溯/支撑阶段。
- 如果用户说“刚发公告、招标文件、是否值得投、投前预测、澄清质疑”，默认只从 `02 招标文件公示`、`03 招标公告/关联公告`、`04 澄清答疑` 进入投前预测线。
- 如果不清楚，优先选择候选公示后证据包主线，因为这是当前商业主线。

任何开发、测试、报告、脚本命名和样本统计都必须保持这个边界。
