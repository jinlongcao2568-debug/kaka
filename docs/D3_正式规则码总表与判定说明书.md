## [D3-R-001] 0. 文档任务与裁决范围

本文档的任务不是罗列“系统有哪些规则”，而是把  建设工程域中的正式规则层收敛为一套可引用、可落库、可测试、可回归、可审计的统一判定口径。

本文档必须回答以下问题：

1. 什么叫正式结果语义；
2. 规则命中与证据可见性如何共同裁决；
3. 哪些规则可以升级为 `AUTO_HIT`，哪些只能停留在 `CLUE` / `OBSERVATION` / `review_request`；
4. A / B / C / D 四层公开边界如何作用于规则层；
5. 规则与 `project_fact`、`legal_action_recommendation`、`saleable_opportunity` 的消费关系是什么；
6. 什么情况下必须阻断、降级、回退重算或进入人工复核；
7. 规则目录、gate 策略和 D11 回归样本应如何一一承接。

本文档不直接展开以下内容：

- 具体抓取实现、解析脚本、模型提示词或 Worker 调度细节；
- 页面组件如何渲染规则结果；
- 接口字段如何序列化输出；
- 交付模板如何裁剪字段；
- 渠道触达、订单交付与外发审批的执行细节。

上述事项分别由 D4、D5、D6、D7、D9、D10、D11 承接，但不得改写本文对结果语义、双闸门与规则优先序的定义。

## [D3-R-002] 1. 生效规则与对齐原则

### [D3-R-003] 1.1 生效规则

- 本文是 L0 在“规则、结果语义与双闸门”维度的正式展开文档。
- 本文只允许细化总纲与正式 D13，不允许突破 L0 与正式 D13 的边界。
- 若本文与 L0 冲突，以 L0 为准。
- 若本文与正式 D13 冲突，以 L0 为准，并由 D13 对公开边界口径进行上收修订，不允许在规则层私自另立边界。
- 若页面、接口、导出、销售话术、脚本、样例或局部实现与本文冲突，视为实现违规，而非规则扩展。

### [D3-R-004] 1.2 规则层对齐原则

所有正式规则都必须同时对齐以下七个维度：

1. **阶段归属**：规则位于阶段 5，但其前提对象必须来自阶段 2-4 的正式对象，消费方必须指向阶段 6 及其后对象；
2. **对象归属**：规则必须说明消费哪些正式对象，禁止消费哪些非正式对象；
3. **边界归属**：规则必须标注处于 A / B / C / D 哪一层能力边界；
4. **结果归属**：规则必须说明默认结果语义与可升级上限；
5. **门禁归属**：规则必须声明自身受哪些 gate、版本状态、时钟状态、coverage / delivery 状态约束；
6. **证据归属**：规则必须声明最低证据等级、最低可见性要求与不可接受证据类型；
7. **测试归属**：规则必须能映射到 D11 中的金标样本、阻断样本、降级样本与回归清单。

### [D3-R-005] 1.3 规则层不得改写的上位事项

本文不得改写以下上位事项：

- `AUTO_HIT / CLUE / OBSERVATION / review_request` 的正式语义；
- `project_fact` 作为唯一统一事实中枢的地位；
- `rule_gate_decision` 与 `evidence_gate_decision` 缺一不可的总规则；
- `saleable_opportunity` 不得直接消费单条规则或原始公告的单向消费原则；
- 自然人高限制信息、内部补证、受控例外的治理边界。
- 招投标规则必须区分产品线和阶段证据：投前预测只消费 02/03/04 且未过截止/开标的材料；出现 05 开标信息后，投前预测规则必须阻断并转后验核验；候选后证据包规则可以消费 03-12 全流程材料，但仍需区分招标正文、后验公告、投标文件公开和合同信息的证据权重。
- 控标、围标、串标、陪标规则必须先经 `AnalysisStrategyPlan v1` 决定解析深度和 AI 触发条件；不得因为附件存在就默认全量深解析或直接升级正式结论。

## [D3-R-006] 2. 正式结果语义

### [D3-R-007] 2.1 四类正式结果

| 结果类型 | 正式含义 | 允许使用条件 | 明确禁止 |
|---|---|---|---|
| `CLUE` | 具有较强公开依据、可形成主价值点，但仍需人工复核、补证或场景限定的强线索 | 规则前提大体成立，但至少存在一个需要保留风险语义、人工确认或场景限定的条件 | 不得伪装成“已坐实”或“当然无效” |
| `OBSERVATION` | 披露不足、链路不完整、信号偏弱或解释空间较大的观察结论 | 允许作为补充说明、弱信号、观察项或回查依据 | 不得单独作为客户版首页主结论、正式外发主结论或销售主承诺 |
| `review_request` | 需要内部材料、制度确认、外部说明或专项人工核验才能闭环的正式复核请求 | 允许进入复核、补证、治理或客户补证链 | 不得直接转写为 `AUTO_HIT`，不得被页面包装为已成立事实 |

### [D3-R-008] 2.2 结果语义补充裁决

- `AUTO_HIT` 只表示正式命中，不表示终局违法定性，不表示当然废标，不表示自动替代监管、评标或司法裁判。
- `CLUE` 允许进入项目主摘要、经营卖点与正式报告摘要，但必须保留条件语义、风险语义与人工复核空间。
- `OBSERVATION` 只允许作为补充解释或观察信号，不得承担正式对外主结论职能。
- `review_request` 是正式对象，不是失败兜底文案；它必须明确说明缺口属于证据缺口、制度缺口、归一缺口、主体对齐缺口或公开边界缺口中的哪一类。
- 同一规则在不同场景下允许降级，但不允许越过上位边界提升结果上限。

### [D3-R-009] 2.3 结果显示与结果裁决的区分

- 规则目录中的“默认结果”只表示该规则在理想前提下的默认表达方向，不等于最终外显结果。
- 最终结果必须由以下因素共同裁决：规则前提、证据等级、公开边界、项目归一、版本 / 时钟状态、治理状态与人工复核结果。
- 页面显示、接口响应、导出模板与销售表述只允许消费最终裁决后的结果，不得消费规则目录中的默认结果去反推主结论。

## [D3-R-010] 3. 公开边界层级与规则上限

### [D3-R-011] 3.1 四层公开边界

| 层级 | 正式含义 | 默认允许输出 | 一般禁止输出 |
|---|---|---|---|
| `A_PUBLIC_CORE` | 全国稳定公开可得主能力 | `AUTO_HIT` / `CLUE` / 正式事实对象 | 终局违法定性 |
| `B_PUBLIC_ENHANCED` | 建设工程域公开增强能力 | `CLUE` / `review_request` / 条件性事实对象 | 全国默认稳定能力承诺 |
| `C_PUBLIC_CONDITIONAL` | 条件性公开线索能力 | `CLUE` / `OBSERVATION` / `review_request` | `AUTO_HIT` 常态化输出 |

### [D3-R-012] 3.2 边界对规则结果的硬约束

- A 层规则允许在双闸门通过时升级为 `AUTO_HIT`，但仍不得越界写成终局违法裁决。
- B 层规则默认只能形成 `CLUE`、`review_request` 或条件性事实对象；只有在 coverage、平台成熟度、公开颗粒度、证据等级同时满足时，才允许形成条件性事实对象，不得写成全国统一最小能力。
- C 层规则默认只能形成 `CLUE`、`OBSERVATION` 或 `review_request`；不得将 C 层线索常态化升级为 `AUTO_HIT`。


凡出现以下任一情况，必须触发规则层治理阻断：

- 规则命中所依赖的关键证据只存在截图、OCR 文本、模型摘要或不可调阅转述，而不存在可回链原始载体；

## [D3-R-014] 4. 规则目录设计规范

### [D3-R-015] 4.1 规则码命名规范

| 项 | 规则 |
|---|---|
| 规则码格式 | `TOPIC-XXX`，例如 `PROC-001` |
| 主题域 | `PROC`、`QUAL`、`CREDIT`、`PM`、`REL`、`PRICE`、`SCORE`、`DOC`、`GOV` |
| 编号原则 | 同主题下只追加新号，不得重排旧号 |
| 默认结果 | 允许声明默认结果，但最终结果必须服从双闸门、边界、治理与版本状态 |
| 规则状态 | 只允许 `DRAFT / EFFECTIVE / SUPERSEDED / DISABLED` |
| 规则来源 | 必须标注上位裁决、对象前提、最低证据等级与边界说明 |

### [D3-R-016] 4.2 每条正式规则的最小描述模板

每一条正式规则，至少必须说明：

- 规则码；
- 规则名称；
- 主题域；
- 所属阶段；
- 对应能力层级（A / B / C / D）；
- 消费对象；
- 触发前提；
- 默认结果语义；
- 最低证据等级；
- `rule_gate_decision` 条件；
- `evidence_gate_decision` 条件；
- 升级条件；
- 降级条件；
- 明确阻断条件；
- 明确禁止事项；
- 输出对象；
- D11 测试与回归要求。

### [D3-R-017] 4.3 规则与对象的单向消费关系

- `rule_hit` 只能消费阶段 2-4 的正式对象，不得消费页面拼装对象、销售备注、口头说明或临时脚本输出。
- `project_fact` 只能消费经裁决后的 `rule_hit`、`rule_gate_decision`、`evidence_gate_decision`、人工复核结果与治理状态，不得反向被单条规则推翻。
- `legal_action_recommendation` 只能建立在 `project_fact`、时钟链、证据等级与双闸门之上，不得由单条规则临时拼装。
- `saleable_opportunity` 不得直接消费任何单条规则，不得把 `CLUE` 或 `AUTO_HIT` 直接等同于正式机会对象。

| 规则码 | 名称 | 阶段 | 触发条件（合规） | 默认结果 | 影响对象 |
|---|---|---|---|---|---|
| `OUT-001` | 触达合规阻断 | 8 | `contact_legal_basis=BLOCKED` / `opt_out_state` 阻断 / `quiet_hours_policy_state=BLOCK` / `frequency_policy_state=BLOCK` / `channel_policy_status` 阻断 | `BLOCK` | `contact_target`、`outreach_plan` |
| `OUT-004` | 触达合规待审 | 8 | `contact_legal_basis=REVIEW_REQUIRED` / `opt_out_state=PENDING_CONFIRMATION` / `quiet_hours_policy_state=REVIEW` / `frequency_policy_state=REVIEW` / 高限制 `channel_family` | `review_request` | `contact_target`、`outreach_plan` |
| `OUT-002` | 触达计划审批与运行模式约束 | 8 | `run_mode in [APPROVAL_RUN,REAL_RUN]` 且 `approval_state!=APPROVED` | `review_request` | `outreach_plan` |
| `OUT-003` | 触达记录写入阻断 | 8 | `plan_status` 未审批或 `run_mode=DRY_RUN` | `BLOCK` | `touch_record` |

## [D3-R-018] 5. 正式证据等级体系

### [D3-R-019] 5.1 证据等级用途

证据等级不回答“规则是否命中”，只回答“该规则结果是否达到目标场景的最低可见性与可外发阈值”。

### [D3-R-020] 5.2 首版正式证据等级

| 证据等级 | 正式含义 | 典型用途 | 不允许用途 |
|---|---|---|---|
| `E1_INTERNAL_ONLY` | 仅可内部参考，证据尚不足以进入正式客户可见或正式建议 | 内部排查、复核排队、内部解释 | 客户版摘要、外发建议、正式主结论 |
| `E2_REVIEW_READY` | 足以支持正式复核请求或内部复核处理，但不足以稳定外显 | `review_request`、内部复核、补证清单 | `AUTO_HIT`、正式客户主结论 |
| `E3_CLIENT_VISIBLE` | 足以支持客户可见摘要或正式报告中的受限结论 | `CLUE`、客户版摘要、正式报告中的受限主价值点 | 正式外发主证、动作建议主证 |
| `E4_EXTERNAL_ACTION_READY` | 足以支撑正式外发动作辅助或对外正式建议 | `AUTO_HIT`、外发异议辅助、正式动作建议核心证据 | 终局违法裁决 |

### [D3-R-021] 5.3 不可接受证据类型

以下证据不得单独支撑 `E3_CLIENT_VISIBLE` 或 `E4_EXTERNAL_ACTION_READY`：

- 仅截图、无原始载体；
- 仅 OCR 文本、无原始页面或原始附件；
- 仅模型摘要、无可调阅引用；
- 仅单一弱匹配主体、无字段级来源切片；

## [D3-R-022] 6. 双闸门体系

### [D3-R-023] 6.1 规则命中闸门 `rule_gate_decision`

`rule_gate_decision` 只回答：规则前提是否成立、对象依赖是否满足、版本 / 时钟 / 归一状态是否允许、边界状态是否允许升级。

| 状态 | 正式含义 | 允许动作 |
|---|---|---|
| `PASS` | 规则前提、对象依赖、边界状态、版本状态与时钟状态均满足 | 允许进入升级判定 |
| `REVIEW` | 规则前提大体成立，但存在制度不确定性、对齐缺口、版本待裁决或需人工确认 | 只允许 `CLUE` / `OBSERVATION` / `review_request` |

### [D3-R-024] 6.2 证据可见性闸门 `evidence_gate_decision`

`evidence_gate_decision` 只回答：证据是否达到目标场景所需的最低可见性、固定能力、调阅能力与外发阈值。

| 状态 | 正式含义 | 允许动作 |
|---|---|---|
| `PASS` | 证据可回链、固定可核、达到目标场景最低证据等级 | 允许进入升级判定 |
| `REVIEW` | 证据足以内部使用或推动复核，但不足以客户可见或正式动作建议 | 只允许内部说明、复核、补证、`review_request` |
| `BLOCK` | 缺原始载体、缺固定件、缺必要回链、仅有截图 / OCR / 模型摘要，或关键主体对齐失败 | 禁止升级与正式外发 |

### [D3-R-025] 6.3 双闸门共同裁决规则

- 只有 `rule_gate_status=PASS` 且 `evidence_gate_status=PASS`，并且公开边界、项目归一、版本 / 时钟、coverage / delivery 等高优先级门禁未阻断时，结果才允许升级为 `AUTO_HIT`。
- 任一 gate 为 `REVIEW` 时，结果只允许停留在 `CLUE`、`OBSERVATION` 或 `review_request`。
- 任一 gate 为 `BLOCK` 时，最终处置只能是阻断、回退重算、进入复核或进入治理隔离链，不得借由人工覆盖直接抬升为通过。
- 人工复核只能补充裁决，不得推翻更高优先级硬门禁。

## [D3-R-026] 7. 全局优先序

所有规则裁决必须服从以下全局优先序；低优先级不得推翻高优先级：

| 优先序 | 决策层 | 正式说明 |
|---|---|---|
| 2 | 项目归一与版本 / 时钟完整性 | `project_root`、`notice_version_chain`、`clock_chain_profile` 未完成正式裁决时，不得升级 |
| 3 | 双闸门 | `rule_gate_decision` 与 `evidence_gate_decision` 缺一不可 |
| 4 | coverage / delivery / release 治理门禁 | 决定是否允许进入正式机会、正式导出、正式外发或正式触达 |
| 5 | `sale_gate_status` 与业务对象化 | 只在前四层允许后才决定推进方式 |
| 6 | 人工覆盖与受控例外 | 只能降级、限制、暂停、受限放行，不得推翻高优先级门禁 |
