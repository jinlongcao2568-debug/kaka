## [D6-R-001] 0. 文档任务与裁决范围

本文档解决的不是“某个页面这次先不显示什么字段”这样的临时 UI 取舍，而是冻结以下正式问题：

1. 哪些字段默认属于正式客户可见字段；
2. 哪些字段只允许内部研判、人工复核、治理后台或隔离链消费；
3. 哪些字段必须脱敏；
4. 哪些字段必须走审批链、双角色链或 SoD；
5. 哪些字段绝对不得进入客户版、外发包、对外 API 或销售话术；
6. 页面、接口、导出、销售、交付若对同一字段给出不一致可见性时，以什么为准；
7. 机器资产应如何承接字段策略，测试与发布如何校验字段越界。

本文档不是对象级交付矩阵，也不是公开边界能力清单：

- **D13** 负责“能力能不能成立”；
- **D7** 负责“对象在什么交付形态下能不能出去”；
- **D6** 负责“字段在既定对象与既定交付形态下能不能出现、以什么强度出现、是否需脱敏、是否需审批”。

三者缺一不可，且以更严格者为准。

## [D6-R-002] 1. 生效规则与优先级

### [D6-R-003] 1.1 生效规则

- 本文是 L0 在字段治理、客户可见范围、审批链与脱敏规则上的正式展开文档。
- 本文只允许细化 L0，不允许突破 L0。
- 若本文与 L0 冲突，以 L0 为准。
- 若字段策略与 D7 对象级交付矩阵冲突，以**更严格者**为准。
- 若页面、接口、导出、销售物料、交付模板、脚本或临时操作与本文冲突，视为实现违规，而非业务灵活表达。

### [D6-R-004] 1.2 优先级顺序

字段是否允许出现、如何出现、向谁出现，统一按以下顺序裁决：

1. **L0 硬边界**：公开边界、D 层禁入、单一事实中枢、正式对象消费顺序；
2. **D13 能力边界**：能力本身是否允许进入正式公开主链；
3. **D7 对象级交付矩阵**：对象在目标交付形态下是否允许出现；
4. **D6 字段策略**：字段分类、默认可见层、脱敏、审批链、SoD；
5. **release gates**：当前 release level、审批状态、证据等级、外发条件；
6. **角色权限矩阵**：当前用户、系统、模板或导出动作是否具备作用域；
7. **页面 / API / 导出实现**：只能执行前六层已允许的内容，不得自发明例外。

### [D6-R-005] 1.3 字段治理的总原则

- 先判断“对象能不能出现”，再判断“字段能不能出现”。
- 字段策略不得绕开对象级交付矩阵。
- 字段允许出现在内部，不等于允许出现在客户版。
- 字段允许出现在客户版，不等于允许出现在外发异议辅助包。
- 自然人高限制字段默认**不自动放行**。
- 任何来源于 D 层、不可审计来源、不可回链来源的字段，一律不得作为正式客户可见字段或正式外发字段。

## [D6-R-006] 2. 字段治理硬约束

### [D6-R-007] 2.1 单一字段口径

- 同一正式字段在页面、接口、导出、销售物料、交付物、治理后台中的名称、分类、脱敏强度、审批要求必须保持一致。
- 不允许页面层用“隐藏列”制造一套字段口径，接口层再用“内部字段”制造第二套，导出层再用模板制造第三套。
- 字段是否可见、如何显示、是否允许搜索 / 复制 / 导出，必须来自统一的字段策略字典。

### [D6-R-008] 2.2 公开边界硬约束

- 任何由 `D_NON_PUBLIC_OR_FORENSIC` 输入形成的字段，不得进入正式客户版、正式外发包或对外 API。
- 任何依赖截图、OCR 文本、模型摘要且没有原始可回链载体支撑的字段，不得进入正式外发主证字段集。
- 任何来自 `private_supplement_record`、`controlled_exception_record` 正文的字段，未完成相应 release 与审批前，一律不得进入客户可见层。

### [D6-R-009] 2.3 单向消费原则

- 客户可见字段只能来源于允许进入正式公开主链或允许进入正式受控交付链的正式对象。
- 销售、页面、导出不得反向拼装 `field_lineage_record`、`rule_gate_decision`、`review_queue_profile`、私有补证正文来生成客户可见字段。
- `project_fact`、`legal_action_recommendation`、`report_record` 是客户可见字段的主消费面，但仍受字段策略与 release gate 约束。

### [D6-R-010] 2.4 审批与 SoD 原则

- 高限制字段的申请人、批准人、释放人不得落在同一角色链。
- 任何影响客户可见范围、自然人高限制字段、外发主证字段、字段脱敏强度的变更，都必须留审计记录。
- 任何字段放行若需要例外，只能进入受控例外链，不得通过脚本、数据库、页面开关或人工私下沟通静默放行。

## [D6-R-011] 3. 正式字段分类体系

正式字段必须归入以下分类之一，不允许“暂不分类、后补治理”。

| 字段分类 | 正式含义 | 默认可见层 | 默认要求 | 典型字段族 |
|---|---|---|---|---|
| `PUBLIC_FACT` | 已进入正式统一事实主链、可稳定承接客户交付的事实字段 | 客户交付版 | 可原值输出，但仍受 release gate 约束 | `project_fact` 的正式事实摘要、阻断原因摘要、正式状态字段 |
| `CLIENT_SUMMARY` | 可对客户或授权外部对象展示的正式摘要字段 | 客户交付版 / 受控对外 API | 允许摘要投影，不得暴露内部推理细节 | `legal_action_recommendation` 摘要、`report_record` 正式摘要 |
| `INTERNAL_REASONING` | 内部研判、追溯、双闸门、解释与复核内部字段 | 内部工作台 | 默认不得客户可见、不得对外 API 透传 | `field_lineage_record`、`rule_gate_decision`、`review_queue_profile` |
| `GOVERNANCE_RESTRICTED` | 治理配置、coverage、delivery、审批链、例外与审计字段 | 治理后台 | 默认只允许治理角色消费 | `coverage_registry`、`release_gates`、审批状态 |
| `NATURAL_PERSON_HIGH_RESTRICTION` | 自然人身份、联系方式、精确角色归属、可单点触达字段 | 受控审批后局部可见 | 默认脱敏 + 审批 + 审计 | 联系方式、精确职位、主体精确定位 |
| `PRIVATE_SUPPLEMENT_ISOLATED` | 私有补证、隔离正文、受控例外正文、内部材料引用字段 | 默认隔离 | 未 release 前不得跨出隔离链 | 私有补证正文、例外正文、内部附件路径 |
| `SYSTEM_CONTROL` | 供系统校验、发布、回归、追踪使用的内部控制字段 | 内部系统 / 治理后台 | 不得对外业务展示 | hash、retention、archive、policy tag、audit ref |

### [D6-R-012] 3.1 分类使用规则

- 同一字段只能有一个主分类，不允许多主分类并存。
- 如确需在不同场景中以不同强度显示，应通过“**同字段不同交付形态策略**”表达，不得复制为多个含义相近字段。
- 任一字段若未完成分类，不得进入正式 API、正式页面、正式导出、正式测试与正式脚本。

## [D6-R-013] 4. 正式可见层级与交付视图

| 可见层级 | 正式含义 | 允许消费面 |
|---|---|---|
| `INTERNAL_ONLY` | 仅内部研判、复核、治理、调试与审计可见 | 内部工作台、治理后台、内部校验脚本 |
| `REVIEWER_VISIBLE` | 仅核查分析员、人工复核员或治理审批链可见 | 证据工作台、复核与报告工作台 |
| `CLIENT_VISIBLE` | 可进入正式客户版报告、客户只读投影、受控对外 API | 客户版报告、客户授权 API |
| `EXTERNAL_ACTION_READY` | 可进入正式外发异议辅助包或正式动作准备材料 | 外发辅助包、外发审批态模板 |
| `DEMO_ONLY` | 仅演示版、预览版、售前内部展示可用 | 演示版页面、售前预览 |

### [D6-R-014] 4.1 可见层级规则

- `DEMO_ONLY` 不得冒充 `CLIENT_VISIBLE`。
- `CLIENT_VISIBLE` 不自动等于 `EXTERNAL_ACTION_READY`。
- `REVIEWER_VISIBLE` 字段不得因页面层展示需求升级为客户可见。
- 高限制字段即使在 `CLIENT_VISIBLE` 模板中，也应按更严格策略脱敏或摘要化。

## [D6-R-015] 5. 正式脱敏规则

| 脱敏规则 | 正式含义 | 允许场景 | 禁止场景 |
|---|---|---|---|
| `NONE` | 原值输出 | 正式事实主链中低风险、已放行字段 | 不适用于自然人高限制字段 |
| `SUMMARY_ONLY` | 仅输出摘要、标签、结论性短语或有限说明 | 客户版报告、对外 API 摘要 | 不得借摘要字段反向恢复内部细节 |
| `NAME_MASK` | 姓名局部遮罩 | 演示版、受控客户版、外部摘要 | 不得代替审批释放完整身份 |
| `PHONE_MASK` | 手机 / 电话局部遮罩 | 联系对象预览、审批前列表 | 不得用于正式触达执行面 |
| `EMAIL_MASK` | 邮箱局部遮罩 | 预览、审批前确认 | 不得用于自动触达 |
| `ORG_ROLE_SUMMARY` | 只显示组织与角色摘要，不显示精确自然人识别信息 | actor 对象客户摘要、售前展示 | 不得替代联系方式释放审批 |
| `HASH_ONLY` | 只保留哈希或不可逆标识 | 审计、去重、系统内引用 | 不得作为客户版字段 |
| `METADATA_ONLY` | 仅允许状态、时间、审批链、来源元数据 | `private_supplement_record`、`controlled_exception_record` | 不得输出正文或原文内容 |
| `FULL_BLOCK` | 一律禁止输出 | D 层、高限制正文、未授权身份串 | 无 |

### [D6-R-016] 5.1 脱敏规则使用纪律

- 脱敏规则必须在字段策略字典中逐字段声明，不能由页面组件临时决定。
- 同一字段在不同交付形态中可采用不同脱敏强度，但必须有明示策略条目。
- 任何高限制字段在未完成审批前，默认应采用 `FULL_BLOCK` 或 `METADATA_ONLY`。

## [D6-R-017] 6. 审批链与 SoD 模型

### [D6-R-018] 6.1 最小审批链目录

| 审批链 ID | 适用动作 | 最低要求 | SoD 要求 |
|---|---|---|---|
| `internal_review_release` | 内部研判简报、内部摘要释放 | 单审批 + 审计 | 创建人与批准人可同部门，但须留痕 |
| `client_report_release` | 客户版报告、客户版 API 投影 | 人工复核 + 交付治理双角色链 | 申请人与最终批准人不得同一人 |
| `external_action_release` | 外发异议辅助包 | 主证要求 + 双闸门 + 双角色链 | 创建者、批准者、放行者不得同一人 |
| `high_restriction_field_view` | 查看自然人高限制字段 | 专项申请 + 审批 + 审计 | 申请人与批准人不得同一角色链 |
| `high_restriction_contact_release` | 释放联系对象或触达准备字段 | lawful basis + 频控 + quiet hours + 审批 | 强制 SoD |
| `coverage_state_change` | coverage 状态调整 | 治理负责人审批 | 变更人不得自批 |
| `delivery_matrix_change` | 对象级交付矩阵调整 | 治理负责人审批 + 回归 | 变更人不得自批 |
| `private_supplement_release` | 私有补证进入可影响阶段 5/6 | 范围声明 + 目标场景阈值 + 审批 | 创建者与释放者分离 |
| `boundary_exception` | 字段边界例外 | 受控、短期、可追溯 | 强制 SoD |

### [D6-R-019] 6.2 审批链规则

- 审批链是正式机器资产，不是流程建议。
- 任何高限制字段释放都必须能回指审批链 ID。
- 任何字段若声明需要审批链但当前审批链目录中没有对应条目，则该字段不得进入正式实现。

## [D6-R-026] 8. 客户可见白名单与绝对禁出项

### [D6-R-027] 8.1 默认客户可见白名单

客户版、客户只读投影、受控对外 API 默认可见的字段，限于以下字段族的**白名单摘要**：

- `project_fact` 的正式事实摘要、阻断原因摘要、`sale_gate_status`、`coverage_sellable_state`、`manual_override_status` 摘要；
- `legal_action_recommendation` 的窗口状态、适用制度范围、主管机关范围摘要、建议下一步；
- `report_record` 的正式摘要、模板状态、release level 摘要；
- 经 D7 对象级交付矩阵与本文字段策略共同批准的受控摘要字段。

### [D6-R-028] 8.2 默认绝对禁出项

以下字段默认绝对禁出，除非上位文档未来明确修订并完成治理审批：

- `field_lineage_record` 原始切片、原始来源路径、解析规则、归一规则细节；
- `rule_gate_decision`、`evidence_gate_decision` 的内部完整阻断明细；
- `review_queue_profile` 的优先级算法、排队策略、复核 bucket；
- `private_supplement_record` 与 `controlled_exception_record` 正文；
- 自然人完整联系方式、完整身份串、未经审批的精确识别字段；
- 任何由 D 层来源生成的字段；
- 任何仅依赖截图、OCR、模型摘要、不可审计中间件文本生成的外发主证字段。

## [D6-R-029] 9. 页面、接口、导出与销售一致性规则

### [D6-R-030] 9.1 页面

- 页面组件不得直接决定字段是否隐藏、是否可复制、是否可导出，必须消费字段策略字典与对象级交付矩阵。
- 页面不得把 `INTERNAL_REASONING` 字段塞进 tooltip、悬浮层、调试信息或导出预览中，绕开正式治理。
- 页面中的“客户版视图”“演示版视图”“内部复核视图”必须绑定不同的字段策略与 release level。

### [D6-R-031] 9.2 API

- API 不得返回字段策略未声明的客户可见字段。
- API 不得把 `INTERNAL_REASONING`、`GOVERNANCE_RESTRICTED`、`PRIVATE_SUPPLEMENT_ISOLATED` 字段混入客户版响应封套。
- API 的 `included`、`warnings`、`policy_tags` 也受本文约束，不是自由扩展豁免区。

### [D6-R-032] 9.3 导出

- 导出模板只能消费白名单字段族。
- 导出模板不得绕开字段策略直接导出页面当前表格或对象原始 JSON。
- 外发异议辅助包中的字段必须同时满足：对象级交付矩阵允许、字段策略允许、目标 release gate 通过。

### [D6-R-033] 9.4 销售与演示

- 销售材料与演示版不得用脱敏前字段替代正式白名单摘要。
- 售前演示若需展示高限制字段，只允许使用演示专用脱敏样例，且必须与真实正式数据隔离。
- 销售口头说明不得通过“口头补充”泄露默认禁出字段。

## [D6-R-039] 11. 测试与验收要求

### [D6-R-040] 11.1 最小必测项

D6 生效后，至少必须有以下测试面：

1. **字段分类完整性测试**：所有正式字段均有分类；
2. **脱敏规则完整性测试**：所有高限制字段均有脱敏策略；
3. **审批链映射测试**：所有需要审批的字段都能映射到审批链；
4. **客户白名单测试**：客户版模板、客户 API、客户视图不得出现默认禁出字段；
5. **自然人高限制字段阻断测试**：无审批时不得进入触达执行面；
6. **D 层字段渗透阻断测试**：D 层来源不得生成客户可见字段；
7. **导出一致性测试**：页面显示、API 返回、导出模板字段集一致；
8. **SoD 测试**：申请人与批准人不得同人放行高限制字段。

### [D6-R-041] 11.2 一票失败条件

以下任一成立，视为 D6 验收失败：

- 客户版、外发包或对外 API 出现默认禁出字段；
- 自然人高限制字段在无审批链情况下进入执行面；
- 页面 / API / 导出对同一字段出现三套不同治理口径；
- 高限制字段脱敏前值进入日志、前端调试面板、导出缓存或错误返回；
- `field_policy_dictionary`、`approval_chain_catalog` 与实际实现不一致；
- 字段策略变更未触发 D11 与治理回归。

## [D6-R-042] 12. 变更治理

### [D6-R-043] 12.1 允许直接修改的变更

- 说明性文字修订；
- 例句优化；
- 不影响字段分类、脱敏、审批链、可见层级的编辑性调整。

### [D6-R-044] 12.2 必须同步修改机器资产与回归的变更

- 新增、删除、重命名客户可见字段；
- 调整字段分类、脱敏、可见层级；
- 调整审批链；
- 调整默认绝对禁出字段；
- 调整与 D7、D11 的联动规则。

### [D6-R-045] 12.3 必须先回收 L0 / D13 / D7 的变更

- 扩大 D 层字段可见范围；
- 将原本禁止的内部推理字段升级为客户可见字段；
- 调整对象级交付形态定义；
- 调整自然人高限制字段的总边界；
- 调整对外表达红线。

## [D6-R-046] 13. 完成定义（Definition of Done）

D6 只有同时满足以下条件，才算正式完成：

1. 所有正式字段均完成分类；
2. 所有高限制字段均完成脱敏与审批声明；
3. `field_policy_dictionary` 与 `approval_chain_catalog` 已落库；
4. 与 D7 对象级交付矩阵已对齐；
5. 与 D11 的字段回归测试已补齐；
6. 页面、接口、导出、销售与治理消费面已按统一字段口径运行；
7. 无默认禁出字段泄露路径；
8. 无未审计的字段放行路径；
9. 无 SoD 违规；
10. 无第二套字段治理口径。

## [D6-R-047] 14. 禁止事项总表

AI / Codex、研发、前端、后端、测试、交付、治理一律不得：

- 把字段策略写成页面开关；
- 把默认禁出字段塞进 tooltip、日志、错误返回、缓存导出或调试面板；
- 把 `INTERNAL_REASONING` 字段包装成客户版“解释说明”；
- 把高限制字段因“客户急需”“销售方便”“演示顺手”而跳过审批；
- 以导出模板或 CRM 集成为理由绕开字段治理；
- 把 D 层来源字段静默映射成客户可见摘要；
- 用口头说明、备注、临时脚本取代正式审批链与审计链；
- 在未同步更新机器资产和回归的情况下修改字段可见范围。
