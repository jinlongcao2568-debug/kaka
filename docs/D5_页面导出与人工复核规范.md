## [D5-R-001] 0. 文档任务与裁决范围

本文档不是页面原型备注，不是前端任务拆解表，也不是“先把页面做出来再回填语义”的设计草稿。

本文档的正式任务是：

1. 冻结  正式页面与工作台目录；
2. 冻结页面顶层主判断的消费来源，防止页面从 `rule_hit`、原始公告、CRM 备注、截图摘要或前端临时逻辑重算主结论；
3. 冻结证据工作台、公开链工作台、复核与报告工作台的解释层与追溯层职责边界；
4. 冻结按钮流、导出模板、审批前置条件与阻断提示规则；
5. 冻结人工复核的正式入口、允许动作、升级 / 降级方式与审计要求；
6. 冻结 D8 / D9 / D10 尚未正式生效前的保留态页面、保留态按钮与阻断口径；
7. 冻结与 D6 / D7 / D11 的承接关系，确保字段策略、交付矩阵、release gate 和测试回归能够覆盖页面与导出行为；
8. 冻结 `contracts/ui/*` 的机器承接纪律，确保 UI 目录、按钮流、模板与保留态保持单一口径。

本文档必须回答以下问题：

- 哪些页面属于正式页面，哪些页面只能是保留态；
- 每个正式页面的主消费对象是什么，什么对象只能作为解释层展示；
- 哪些按钮可以执行正式动作，哪些按钮只能申请、排队、阻断或提示后续批次放行；
- 导出模板有哪些正式分层，导出时必须校验什么，什么情况下必须阻断；
- 人工复核可以修改什么，不能修改什么，如何与 `project_fact`、双闸门、审批链和 release gate 协同；
- 页面与导出如何承接正式边界、字段策略、外发治理和测试验收。

本文档不直接展开以下内容：

- 对象定义、字段字典和枚举正文，由 D2 承接；
- 规则码、证据等级、双闸门与优先序正文，由 D3 承接；
- API 路径、请求 / 响应封套、错误码与权限矩阵正文，由 D4 承接；
- 字段分类、脱敏、审批链和线索包外发字段正文，由 D6 承接；
- 对象级交付矩阵、外发阻断和 release gate 正文，由 D7 承接；
- 商业对象、触达对象、订单交付对象的放行条件，由 D8 / D9 / D10 承接；
- 测试验收、金标回归和发布前检查正文，由 D11 承接。

## [D5-R-002] 1. 生效规则与优先级

### [D5-R-003] 1.1 生效规则

- 页面、导出与人工复核只能细化 L0、D2、D3、D4、D13 已冻结的正式语义，不得反向定义第二套主判断。
- 页面层、导出层、按钮层和前端状态层都不拥有独立的领域定义权；如与上位对象、结果语义、能力边界或治理门禁冲突，以上位文档为准。
- 保留态页面、保留态按钮、演示态模板和受限子视图都必须显式标注 `surface_state`，不得伪装成正式已放行功能。
- 页面展示、导出模板、按钮启用状态与复核放行动作，必须能回指机器承接目录中的正式条目，不得只存在于组件代码或页面注释中。

### [D5-R-004] 1.2 优先级

页面与导出的消费优先级固定为：

1. `project_fact`、`legal_action_recommendation`；
2. `rule_gate_decision`、`evidence_gate_decision`、`coverage_sellable_state`、`delivery_risk_state` 等正式 gate / governance 对象；
3. `report_record`、`review_queue_profile`、`public_attack_surface`、`evidence_grade_profile` 等解释层对象；
4. `public_chain`、`clock_chain_profile`、`notice_version_chain`、`field_lineage_record` 等追溯层对象；
5. `private_supplement_record`、`controlled_exception_record` 元数据与其他治理隔离对象。

任何页面、模板或按钮都不得从低优先级对象反向生成高优先级主结论。

### [D5-R-005] 1.3 单向消费原则

- 顶层项目判断页面、经营视图、销售视图、线索包外发摘要一律优先消费 `project_fact`。
- 证据工作台、公开链工作台与追溯页可以展示低优先级对象，但只能作为解释层、追溯层或补证层，不得把解释层升级为线索包主结论。
- 页面不得通过抓取层数据、接口组合层、临时缓存、前端 reducer、本地派生字段或按钮点击逻辑形成第二套主判断。
- 导出不得把页面当前渲染结果、隐藏区块、折叠区块或内部 JSON 直接当作正式交付物。

### [D5-R-008] 2.2 页面状态的正式含义

| `surface_state` | 正式含义 | 页面行为 |
|---|---|---|
| `ACTIVE_BATCH2` | 已进入当前正式页面范围，可按本文与 D4 正式消费对象实现 | 允许真实接口调用与正式按钮流 |
| `ACTIVE_BATCH3` | 需要治理资产同步生效后才能放行 | 可实现正式页面，但所有治理写操作必须联动审批与审计 |
| `RESERVED_UNTIL_D8` | 路径、名称与主区块冻结，但商业对象规范未生效 | 只允许占位、说明、阻断或预览态，不得形成真实执行面 |
| `RESERVED_UNTIL_D9` | 路径、名称与主区块冻结，但触达规范未生效 | 只允许占位、说明、阻断或预览态，不得形成真实执行面 |
| `RESERVED_UNTIL_D10` | 路径、名称与主区块冻结，但订单交付规范未生效 | 只允许占位、说明、阻断或预览态，不得形成真实执行面 |
| `INTERNAL_PREVIEW` | 内部预览与草案态 | 只允许内部工作台承接与审批前置，不得对外放行 |
| `DISABLED_BY_POLICY` | 因字段、交付矩阵、release gate 或审批链阻断而不可操作 | 页面可见，但关键动作必须禁用并显示阻断原因 |

### [D5-R-008A] 2.3 页面可见层级补充

- `internal_visible`：内部运营可见，默认允许在内部工作台呈现。
- `sales_visible`：内部销售可见，必须遵守字段策略与触达合规。
- `external_visible`：默认一律为 `否`；外部只允许通过线索包导出与交付结果体现，不存在客户工作台。
### [D5-R-009] 2.4 不允许出现的页面行为

- 不得把保留态页面做成“可点击但返回伪数据”的假执行面；
- 不得因为页面原型已存在就默认允许真实接口落地；
- 不得把治理阻断页面文案写成“功能异常”或“暂不可用”而隐藏真实阻断原因；
- 不得通过前端条件分支偷偷放行未生效页面。

## [D5-R-010] 3. 页面硬约束

### [D5-R-011] 3.1 顶层主判断约束

- 项目事实工作台、经营视图、销售视图、线索包摘要页一律优先消费 `project_fact`。
- 页面不得从 `rule_hit`、原始公告、字段切片、CRM 备注、人工笔记或导出缓存反向拼装主结论。
- 页面不得把 `delivery_risk_state` 解释为项目核查结论，也不得把 `coverage_sellable_state` 解释成项目事实判断。
- 页面不得把 `OBSERVATION`、`review_request` 文案伪装成“已成立结论”或“当然无效 / 当然违法”。

### [D5-R-012] 3.2 解释层与追溯层约束

- 证据工作台与公开链工作台可以展示 `field_lineage_record`、`public_chain`、`clock_chain_profile`、`notice_version_chain`、`evidence`、`rule_hit` 等对象，但只能用于解释、追溯、复核与补证。
- 页面不得把追溯层对象单独抬升为线索包主结论；如需要外发投影，必须先经过 D6 字段策略与 D7 对象级交付矩阵裁决。
- `private_supplement_record`、`controlled_exception_record` 在页面层默认只允许元数据消费，不得展示正文，不得被页面摘要偷带到外发区域。

### [D5-R-013] 3.3 高限制对象约束

- 未完成审批、频控、quiet hours 与 lawful basis 校验前，页面不得展示自然人完整联系方式。
- `contact_target`、`legal_action_actor_profile`、`procurement_decision_actor_profile` 中涉及自然人精确身份或联系方式的字段，默认必须脱敏或阻断。
- 任何高限制字段的展示状态，都必须由 D6 字段策略驱动，不得由组件自行决定。

### [D5-R-014] 3.4 不得越权的 UI 行为

- 页面按钮不得直接改写 `project_fact`、`rule_gate_decision`、`evidence_gate_decision` 或正式对象的核心结论字段；
- 页面不得跳过 D4 已冻结的正式接口，直接调用隐藏端点、脚本或本地逻辑完成真实业务动作；
- 页面不得把内部异常截图、OCR 文本、模型摘要或人工口头备注展示成正式证据主面；
- 页面不得生成第二套“页面状态枚举”替代正式对象状态。

## [D5-R-015] 4. 批次 2 最小工作台与区块结构

### [D5-R-016] 4.1 任务入口

**必有区块：**

- 表单区：地区、时间范围、策略模板、复核车道；
- 状态区：任务队列状态、执行上下文、最近重跑记录、窗口期提示；
- 动作区：创建任务、触发重跑、查看执行日志入口。

**页面约束：**

- 任务入口只能创建 / 调度，不得直接形成业务结论；
- 重跑动作只能触发正式重建流程，不得改写真相层结论；
- 窗口期提示必须来源于正式时钟对象，不得由页面临时计算。

### [D5-R-017] 4.2 项目事实工作台

**必有区块：**

- 顶部摘要：`sale_gate_status`、`coverage_sellable_state`、`delivery_risk_state`、`manual_override_status`；
- 统一事实摘要区：`project_fact` 的主价值点、阻断原因、统一摘要；
- 动作建议区：`legal_action_recommendation` 的窗口状态、适用制度、建议下一步；
- 关联跳转区：进入公开链工作台、证据工作台、复核与报告工作台。

**页面约束：**

- 页面头部状态必须来自正式对象字段，不得由前端聚合逻辑另行生成；
- 事实摘要区只能消费已放行字段；
- 动作建议区不得展示未达到对应证据等级的外发语义。

### [D5-R-018] 4.3 公开链与核验工作台

**必有区块：**

- 公开链区：`public_chain` 的节点完整性、缺失情况、时间线状态；
- 多时钟区：`clock_chain_profile`；
- 多版本区：`notice_version_chain`；
- 攻击面与证据等级区：`public_attack_surface`、`evidence_grade_profile`；
- 追溯跳转区：进入字段追溯、证据详情、规则解释。

**页面约束：**

- 该工作台展示的是解释层与追溯层，不得直接替代项目事实页；
- 多时钟、多版本冲突必须原样显示正式状态，不得被页面自动“替你裁决”；
- 页面可展示“建议进入复核”，不得自动改写为已成立结论。

### [D5-R-019] 4.4 证据工作台

**必有区块：**

- 规则命中区：`rule_hit` 列表、结果语义、边界说明；
- 双闸门区：`rule_gate_decision`、`evidence_gate_decision` 的通过 / 待审 / 阻断说明；
- 证据追溯区：`evidence`、`field_lineage_record`；
- 复核触发区：`review_request` 与缺口清单。

**页面约束：**

- 双闸门的状态必须原样消费正式对象，不得在页面层自行升级 / 降级；
- `field_lineage_record` 只允许在内部页面展示，外发版和演示版不得原样透出；
- 页面不得因为命中条目多就把 `CLUE` 或 `OBSERVATION` 改写为 `AUTO_HIT`。

### [D5-R-020] 4.5 复核与报告工作台

**必有区块：**

- 复核队列区：`review_queue_profile`、`review_lane`、`review_priority_score`；
- 报告区：`report_record` 的简报、证据包、导出模板申请；
- 动作建议区：`legal_action_recommendation`；
- 补证区：`private_supplement_record` 元数据与审批状态；
- 审批区：导出申请、补证释放申请、复核结论提交。

**页面约束：**

- `private_supplement_record` 正文默认不得展示；
- 复核动作必须进入正式复核流，不得只在页面上改显示状态；
- 导出申请必须显式呈现模板、最低 release level、审批状态与阻断原因。

## [D5-R-021] 5. 页面级子视图与预览态规则

### [D5-R-022] 5.1 正式子视图

正式子视图只允许作为正式工作台的组成部分存在，不得独立夺取主判断权。允许的子视图包括：

- `fact_summary_panel`
- `legal_action_panel`
- `public_chain_timeline_panel`
- `version_conflict_panel`
- `gate_status_panel`
- `evidence_trace_panel`
- `review_queue_panel`
- `report_export_panel`

任何新增子视图，必须在 `contracts/ui/workbench_catalog.json` 中登记其：

- 所属页面；
- 主消费对象；
- 可见角色；
- 可见层级；
- 是否允许线索包外发投影。

### [D5-R-023] 5.2 预览态与演示态

- 预览态页面只允许使用正式对象的摘要化投影，禁止直接暴露内部对象全文；
- 演示态必须明确标注“非正式交付物”与“仅供预览”；
- 预览态不得伪装成正式线索包外发页面；
- 保留态页面若需要演示，只允许展示阻断说明、占位说明与脱敏预览。

## [D5-R-024] 6. 按钮流原则

### [D5-R-025] 6.1 按钮类型总表

| 按钮类型 | 允许动作 | 明确禁止 |
|---|---|---|
| 导航按钮 | 跳转到正式工作台、正式子视图、正式追溯页 | 跳转到未声明对象或隐藏原始数据页 |
| 任务按钮 | 创建任务、触发重跑、查看任务状态 | 直接写业务结论或绕开阶段 6 聚合 |
| 复核按钮 | 打开 review queue、提交复核结论、申请补证、申请导出 | 跳过审批直接放行线索包外发导出 |
| 导出按钮 | 申请正式模板导出，并显示 release level、审批与阻断信息 | 直接把页面内容整页导出为正式交付物 |
| 治理按钮 | 进入治理页、发起审批、查看例外与审计 | 绕过治理接口做隐藏写操作 |
| 保留态按钮 | 显示“能力未生效”或“需后续批次放行” | 被误连到真实执行接口 |

### [D5-R-026] 6.2 按钮启用前置条件

正式按钮启用前，必须同时满足：

1. 页面所属 `surface_state` 为允许执行状态；
2. 对应对象已形成正式 ID 与正式状态；
3. 对应接口已在 D4 正式放行；
4. 若涉及导出、治理、高限制字段、触达或外发，必须同步满足 D6 / D7 / D9 / D10 的前置条件；
5. 当前角色具备权限，且审批链状态允许。

### [D5-R-027] 6.3 正式阻断与按钮提示

按钮阻断时，页面必须显示正式阻断原因，不得只显示通用报错。阻断原因至少包括：

- 对应能力尚未正式生效；
- 当前对象未达到目标 release level；
- 当前审批链未完成；
- 当前字段策略不允许；
- 当前交付矩阵不允许；
- 当前对象或页面受保留态规则限制；
- 当前项目处于双闸门阻断或高优先级治理阻断状态。

## [D5-R-033] 8. 页面文案与风险提示

### [D5-R-034] 8.1 正式文案原则

- `AUTO_HIT` 只表示正式命中，不得写成终局违法定性；
- `CLUE` 必须保留“强线索 / 需结合复核或补证”语义；
- `OBSERVATION` 必须保留“观察 / 弱信号 / 颗粒度不足”语义；
- `review_request` 必须保留“请求核查 / 待补证 / 待确认”语义；
- `BLOCK` 只表示正式阻断，不得被解释成“项目无价值”。

### [D5-R-035] 8.2 统一提示文案要求

页面提示文案必须能够说明：

- 阻断来自哪里；
- 是能力未生效、审批未完成、字段未放行、对象未形成还是证据 / gate 未达标；
- 当前允许的下一步是什么；
- 是否需要人工复核、治理审批或等待后续批次放行。

### [D5-R-036] 8.3 禁止文案

- 不得把 B / C 层能力写成全国默认稳定能力；
- 不得把 D 层隔离对象包装成正式结论；
- 不得把“页面可见”偷换为“线索包外发可见”或“外发可用”；
- 不得用营销文案掩盖正式状态、证据等级或阻断条件。

## [D5-R-037] 9. 人工复核规范

### [D5-R-038] 9.1 复核的正式职责

人工复核的正式任务是：

1. 解释 `CLUE`、`OBSERVATION`、`review_request` 背后的边界和缺口；
2. 审查证据完整性、追溯链完整性与导出放行条件；
3. 提交复核结论、补证需求、导出申请或阻断意见；
4. 在不突破上位硬门禁的前提下，对结果进行受控确认、限制放行或退回复核；
5. 把复核动作沉淀到正式队列、正式报告对象和正式审计链。

### [D5-R-039] 9.2 复核可以做的事

- 确认当前对象应停留在 `CLUE`、`OBSERVATION` 或 `review_request`；
- 确认导出是否进入审批；
- 确认是否触发补证；
- 确认是否需要进入更高优先级复核队列；
- 在正式范围内补充说明、边界备注与复核意见；
- 在治理允许的前提下提交放行申请或阻断申请。

### [D5-R-040] 9.3 复核不能做的事

- 不得改写阶段 3 真相层事实；
- 不得推翻高优先级公开边界、双闸门、coverage / delivery 阻断；
- 不得把 D 层材料静默转成正式公开主证；
- 不得把保留态页面或未生效能力强行放行为正式执行面；
- 不得只在页面上改文案而不落正式对象、正式报告或正式审计记录。

### [D5-R-041] 9.4 复核动作总表

| 复核动作 ID | 正式动作 | 作用对象 | 结果 |
|---|---|---|---|
| `review_confirm_signal` | 确认信号成立但不升级语义 | `rule_hit`、`review_request`、`report_record` | 保持当前语义，补充复核意见 |
| `review_request_more_material` | 请求补证 | `review_request`、`private_supplement_record` | 进入补证与审批流 |
| `review_submit_for_export` | 提交导出申请 | `report_record` | 进入审批与 release gate 流 |
| `review_escalate_priority` | 提升复核优先级 | `review_queue_profile` | 调整正式队列优先级 |
| `review_block_release` | 明确阻断导出或放行 | `report_record`、治理对象 | 进入阻断与审计链 |
| `review_return_for_rebuild` | 退回重算或重核验 | `project_fact`、阶段链对象 | 触发正式重算入口 |

### [D5-R-042] 9.5 复核留痕要求

每一次正式复核动作至少必须记录：

- 操作人；
- 作用对象与对象 ID；
- 操作时间；
- 原状态与目标状态；
- 复核理由；
- 是否触发审批、补证、导出或重算；
- 对应审计事件 ID。
