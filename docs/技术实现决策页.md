- capability permission layer 已落地，Stage 8 / Stage 9 已接入 runtime permission layer

## 3. 已正式冻结的技术实现底座
### 3.1 后端底座
- 正式采用：Python 3.11+、FastAPI、Pydantic v2。
- 说明：对象契约强、规则密度高、主链清晰，适合 AI 协作与测试回归。

### 3.2 数据存储
- 正式采用：PostgreSQL 作为正式主库；SQLAlchemy 2 作为 ORM；Alembic 作为迁移方案。
- 明确：SQLite 仅限开发便利用途，不作为正式主库；不采用 MongoDB 作为正式主存储。

### 3.3 队列与缓存
- 正式采用：Redis + Dramatiq。
- 用途：样本处理、批量规则重跑、预览生成、回写任务；不得压在同步 API 请求链路。

### 3.4 对象存储
- 正式采用：MinIO（local / internal_pilot），S3 兼容作为未来外部兼容目标。
- 说明：证据固定件、导出物、LeadPack Preview、附件类对象不直接依赖数据库大字段保存。

### 3.5 本地运行与第一阶段部署
- 正式采用：Docker Compose 作为当前最小正式运行方式。
- 当前支持：local、internal_pilot；external 不作为当前放行目标。

### 3.6 前端策略
- 正式采用：Next.js + TypeScript 作为前端推荐实现栈。
- 说明：前端不是第一批 P0 主链实现阻断项；第一优先级为后端主链与 Preview 结果形成。

## 11. 架构优化 A：统一 policy executor + runtime validator

### 11.1 当前正式目标
- 把 Stage 7/8/9 从“service 内散落 hardcode + 轻校验”收口到统一运行时骨架。
- 当前只冻结统一入口、最小 agent loop、trace 语义与 fallback/review/blocked 分支；不接真实外部系统。

### 11.2 当前正式入口
- `src/shared/policy_executor.py`
- `src/shared/capability_runtime.py`
- `src/shared/context_packet.py`
- `src/shared/state_packet.py`
- `src/shared/runtime_validator.py`

### 11.3 最小 agent loop
当前最小执行链固定为：

1. `load context/state`
2. `resolve capability mode`
3. `load policy`
4. `evaluate`
5. `emit decision`
6. `write trace`
7. `fallback / review / blocked`

补充说明：
- `capability_mode` 当前至少覆盖：`stage7_sales`、`stage8_outreach`、`stage9_delivery`
- 每一步都必须可留痕到 runtime trace
- fallback 不等于放行；fallback 只代表“用保守默认值继续内部链路”

### 11.4 统一 policy executor 当前消费范围
- Stage 7：`window_value_policy_catalog`、`lead_value_scoring_catalog`、`price_normalization_catalog`、`competitor_confidence_catalog`
- Stage 8：`contact_priority_policy_catalog`、`contact_source_policy_catalog`、`contact_compliance_matrix`、`outreach_cadence_catalog`、`retry_policy_catalog`、`touch_stop_condition_catalog`
- Stage 9：`outcome_taxonomy_catalog`、`governance_feedback_policy_catalog`

### 11.5 当前 decision output 要求
- 每次 policy 执行必须输出：
  - `policy_key`
  - `decision_state`
  - `outputs`
  - `reasons`
  - `fallback_used`
  - `catalog_id / policy_id`
- `decision_state` 当前统一收口为：`ALLOW / REVIEW / BLOCK / FALLBACK`

### 11.6 runtime validator 当前正式范围
- 入口：`ContractStore.build_record -> RuntimeValidator.validate_payload`
- 当前优先生效对象：
  - Stage 7：`legal_action_actor_profile`、`procurement_decision_actor_profile`、`buyer_fit`、`challenger_buyer_fit`、`saleable_opportunity`
  - Stage 8：`contact_target`、`outreach_plan`、`touch_record`
  - Stage 9：`order_record`、`payment_record`、`delivery_record`、`opportunity_outcome_event`、`governance_feedback_event`
- 当前最小校验面：
  - required
  - enum
  - type
  - shape
  - bool vs string
  - list vs scalar
  - missing runtime critical field

### 11.7 当前明确不做的事
- 不把 policy executor 直接接到真实 vendor/provider/source API
- 不在本轮引入真实发送、真实支付、真实交付
- 不在本轮改写 contracts/sales 或 release/gate 资产
- 不在本轮把 runtime validator 推广到全仓所有对象

## 12. 架构优化 B：统一 capability_mode matrix

### 12.1 当前正式定位
- `capability_mode` 是统一运行态维度。
- 它与 `A/B/C/D`、`release_layer` 并列，不替代其中任一维度。
- 它用于统一 source/vendor/provider/execution 的“当前怎么运行”口径。

### 12.2 当前正式词表
- `PERMANENTLY_BLOCKED`
- `BUILDABLE_BUT_OFF_BY_DEFAULT`
- `INTERNAL_ONLY`
- `INTERNAL_GOVERNED`
- `APPROVAL_REQUIRED`
- `SHADOW_MODE`
- `DRY_RUN`
- `REAL_RUN_READY`
- `EMERGENCY_OFF`

### 12.3 与现有维度的关系
- `A/B/C/D`：能力边界性质
- `release_layer`：允许消费/交付到哪一层
- `capability_mode`：当前运行态怎么开、默认是否关闭、是否只允许 shadow/dry-run/approval
- `capability_mode` 不得提升 `release_layer`
- `EXTERNAL_CONTROLLED_OPENING` 继续是外部受控开放要求

### 12.4 当前纳入统一 capability_mode 的能力族
- external source
- contact enrichment
- execution vendor
- model provider
- tool provider
- Stage 8 execution
- Stage 9 execution
- delivery/export variants
- risky automation
- emergency off

### 12.5 当前实现约束
- 只收口 machine-readable matrix，不接真实 provider/vendor 代码
- `REAL_RUN_READY` 仅表示运行准备态，不表示允许 external release
- `EMERGENCY_OFF` 必须保留最高优先级停机语义

### 12.6 capability runtime permission layer（本轮收口）
- `capability_mode` 不再只作为静态词表存在，必须通过统一 resolver 进入 Stage 8 / Stage 9 runtime。
- resolver 最小输入固定为：`stage / capability_family / target_id_or_role / requested_action / release_layer / approval_state / runtime inventory / manifest / policy context`。
- resolver 最小输出固定为：`capability_mode / decision / review_required / blocked_reason / trace_fields`，并允许追加 governed metadata。
- `EMERGENCY_OFF` 与 `PERMANENTLY_BLOCKED` 必须拥有最高优先级短路能力，且发生短路时不得继续进入下游 policy executor。
- `REAL_RUN_READY` 只表示运行准备态；若 `release_layer=EXTERNAL_CONTROLLED_OPENING`，则仍必须阻断真实执行。
- Stage 8 本轮至少必须在 source resolution、execution vendor、stage8 risky execution 三类路径前调用 resolver。
- Stage 9 本轮至少必须在 internal writeback / governed execution 前调用 resolver。
- `permission_trace`、`permission_decision_state` 与 additive governance 信息必须进入 runtime trace / inputs / handoff，供测试与脚本消费。

### 12.7 capability canonical source 与 resolver precedence（本轮收口）

- capability runtime 当前只允许以下 canonical source：
  - `contracts/release/runtime_policy_catalog.json`：`capability_mode_vocabulary`、`capability_mode_rules`、`capability_families`、`capability_mode_priority_order`、`runtime_resolver_precedence`
  - `contracts/release/external_unlock_prerequisite_matrix.json`：future unlock tier vocabulary / prerequisite baseline
  - `contracts/release/future_unlock_decision_matrix.json`：future unlock candidate / deny / blocked classification 与推荐序列
- `control/runtime_inventory.yaml`、`control/release_manifest.yaml`、`control/model_release_manifest.yaml`、`control/external_unlock_prerequisite_state.yaml`、`control/future_unlock_decision_state.yaml` 只允许作为 projection / mirror / status source，不得再作为 capability 语义源；family current mode / priority order / protected short-circuit / `EXTERNAL_CONTROLLED_OPENING` controlled_opening_requirement 只允许引用 contracts canonical source。
- runtime resolver 当前正式 precedence 固定为：
  1. `runtime_override`
  2. target policy `current_capability_mode`
  3. target registry/provider `current_capability_mode`
  4. family `current_capability_mode`
- `EMERGENCY_OFF` 与 `PERMANENTLY_BLOCKED` 继续拥有最高优先级短路语义；其 short-circuit 顺序以 `runtime_policy_catalog.json#runtime_resolver_precedence` 为准。
- `REAL_RUN_READY` 仍只表示运行准备态；即使出现，也不得覆盖 `EXTERNAL_CONTROLLED_OPENING`。
