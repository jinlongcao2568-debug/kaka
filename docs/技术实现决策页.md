# AX9S 技术实现决策页（正式）

## 1. 文档目的

本文只保留 AX9S 当前阶段仍有效的**技术实现裁决摘要**：

- 当前实际采用什么技术底座
- 哪些是已经落地的 runtime 现实
- 哪些只是未来可选方向
- `REAL_RUN_READY / EXTERNAL_CONTROLLED_OPENING / EMERGENCY_OFF / PERMANENTLY_BLOCKED` 的关系由谁解释

本文是技术实现依据，不是 phase/readiness 的正式状态源；状态源仍以 `control/repo_status.md`、`control/current_task.yaml`、`control/milestone_status.yaml`、`docs/文档与资产状态板.md` 为准。

## 2. 当前适用范围

- 当前阶段：内部线索运营平台开发放行阶段
- 当前仓库总体 readiness：`READY_FOR_POST-REPAIR_MAINLINE_SELECTION`
- 适用环境：`local`、`internal_pilot`
- `future_external` 只作预留，不构成当前放行

当前恢复的是 guarded development，不是 business fully ready；`internal governed / preview / dry-run / approval-required` 不得被误写成 live-ready。

## 3. 当前技术底座

### 3.1 后端与接口

- 当前正式后端底座：Python 3.11+、FastAPI、Pydantic v2
- 当前正式 app 入口：`src/api/main.py`
- 当前 Stage7-9 已有 repository-backed internal preview / draft surface，并已接入真实 FastAPI transport

### 3.2 存储与迁移

- 正式主库口径：PostgreSQL + SQLAlchemy 2 + Alembic
- 本地 direct-dev / unittest 默认仍走 `json-file` 隔离环境
- SQLite 仅限开发便利，不作为正式主库

### 3.3 队列、缓存与对象存储

- 队列 / 缓存目标底座：Redis + Dramatiq
- 对象存储目标底座：MinIO（`local` / `internal_pilot`）+ S3 兼容预留
- 证据固定件、导出物、LeadPack Preview、附件类对象不应依赖数据库大字段保存

### 3.4 当前前端现实 vs 未来方向

- **当前实际**：仓库当前 operator / customer internal surface 是既有 FastAPI-mounted、server-rendered internal surface
- **未来可选**：`Next.js + TypeScript` 只作为可选未来前端方向，不是当前已采用的运行时现实

校正口径：

- 不得再把 `Next.js + TypeScript` 写成“当前实际已采用的前端正式栈”
- 当前已落地事实应以 `PTL-I100-132 productizes the existing FastAPI server-rendered frontend` 和 `src/api/main.py` 的 transport reality 为准

## 4. 模型 / Provider / Tool 边界

### 4.1 当前冻结的是职责边界，不是品牌

- 当前不冻结：具体模型供应商、具体模型型号、具体工具供应商、具体外部 source / execution vendor
- 当前正式冻结：
  - 模型只允许做辅助抽取、辅助归因、辅助排序、辅助建议
  - 模型不得直接决定正式外发结果、正式触达、正式交付
  - 模型不得替代审批链 / 审计链 / review

### 4.2 当前统一扩展接口语义

- source 侧：`source_vendor_adapter`、`contact_enrichment_adapter`
- execution 侧：`channel_execution_adapter`
- model / tool 侧：`model_provider_adapter`、`tool_provider_adapter`
- trace / audit 侧：`vendor_id`、`vendor_type`、`vendor_role`、`provider_id`、`provider_type`、`source_audit_ref`、`query_trace_id`、`execution_trace_id`、`provider_trace_id`、`vendor_response_ref`、`fallback_vendor_id`、`fallback_provider_id`、`requires_manual_review`

## 5. Capability Runtime 统一口径

### 5.1 当前正式词表

- `PERMANENTLY_BLOCKED`
- `BUILDABLE_BUT_OFF_BY_DEFAULT`
- `INTERNAL_ONLY`
- `INTERNAL_GOVERNED`
- `APPROVAL_REQUIRED`
- `SHADOW_MODE`
- `DRY_RUN`
- `REAL_RUN_READY`
- `EMERGENCY_OFF`

### 5.2 关键解释

- `REAL_RUN_READY` 只表示运行准备态，不表示 release 放行
- `EXTERNAL_CONTROLLED_OPENING` 继续是外部受控开放要求
- `EMERGENCY_OFF` 与 `PERMANENTLY_BLOCKED` 必须保持最高优先级短路语义
- 即使 capability 进入 `REAL_RUN_READY`，只要 `release_layer=EXTERNAL_CONTROLLED_OPENING`，真实执行仍必须阻断

### 5.3 canonical source

capability mode / priority / protected short-circuit / controlled_opening_requirement 的唯一 canonical source 仍是：

- `contracts/release/runtime_policy_catalog.json`
- `contracts/release/external_unlock_prerequisite_matrix.json`
- `contracts/release/future_unlock_decision_matrix.json`

以下只允许作为 projection / mirror / status source，不得再充当 capability 语义源：

- `control/runtime_inventory.yaml`
- `control/release_manifest.yaml`
- `control/model_release_manifest.yaml`
- `control/external_unlock_prerequisite_state.yaml`
- `control/future_unlock_decision_state.yaml`

## 6. 当前实现约束

### 6.1 Stage7-9 runtime

- Stage7 已进入 shared policy runtime chain
- Stage8 / Stage9 已接入 runtime permission layer
- governance runtime guards 已真实消费 field policy / delivery matrix / release gates / usage policy
- semantic validator 已接入 H-01~H-07 关键 consumer runtime 校验

### 6.2 当前明确不做

- 不把 policy executor 直接接到真实 vendor/provider/source API
- 不在本轮引入真实发送、真实支付、真实交付
- 不在本轮把 runtime validator 推广到全仓所有对象
- 不把 external software release、Stage8 real execution、Stage9 real payment / delivery / refund 写成默认放开
