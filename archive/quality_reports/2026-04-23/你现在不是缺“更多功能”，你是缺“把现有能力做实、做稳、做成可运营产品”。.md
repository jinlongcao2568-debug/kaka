我的建议很明确：

**不要什么都一起做。**
你现在最该做的是把产品先做成“内部真正可运营”的版本，而不是追求一次把外部发布、真实触达、真实支付也一起做掉。按报告口径，当前仓库已经是“内部主线基本闭合，但前半链、基础设施、真实样本、正式化还没收口；外部化和 live 执行仍被治理阻断”的状态。 

我给你一个新手最容易执行的判断：

## 先做什么

第一阶段只做 **内部 100% 收口**。
最稳的顺序不是乱补功能，而是先把正文/contract/runtime 对齐，再把 Stage6 做成正式产品台，再补稳定持久化、测试/命令收口、真实样本矩阵。报告里这条主线是最一致、最稳的。 

在这条主线里，真正最值钱的开发方向有 6 个：

**1. 真源生产**
先打通 1 条真实公开源纵切：Stage1 scheduler → Stage2 fetcher → Stage3 parser → Stage4 verifier → Stage5 rule/evidence。现在前半链还能跑，但更像 internal preview/runtime，不是真正的情报生产线。 

**2. 真规则工厂**
把 Stage5 从“单规则/flag 驱动”补成“规则目录批量执行 + evidence slice 引用 + 多规则聚合 + 冲突解释”。这决定你到底是演示系统，还是能稳定产证据的产品。

**3. Stage6 产品化**
现在 Stage6 能算事实，但还不是正式 first-class 产品面。要把 `project_fact / report / review_queue / challenger / legal_action_recommendation` 做成正式持久化和工作台对象，不然产品永远是从 Stage7 才开始“能运营”。 

**4. Stage8/9 闭环补全**
这里不是推倒重写，而是补闭环。Stage8 要补 formal carrier 的专门 persistence/repository/readback；Stage9 要把 projected writeback 继续往受控回写闭环推进。 

**5. 正式基础设施**
把文件型存储换成正式底座：数据库、队列、对象存储、运行治理。报告反复强调当前还是单机/内测态，不适合长期稳定运营。 

**6. 真实样本矩阵**
这对 AI 代理特别重要。不要只让它在 happy path 上自我感觉良好，要补 review、reject、reselect、quiet hours、approval missing、partial payment 等真实样本。 

## 现在不要做什么

这几类事不是永远不做，而是**现在先别做**：

不要先冲 `external software release`。
不要先做 Stage8 live execution。
不要先做 Stage9 live delivery/payment/refund。
不要先补花哨 UI，也不要先扩更多 preview 字段。
报告已经把这些列成后续 unlock 程序，而且顺序是冻结的，不能倒着来。 

## 是不是都要做好

**最终很多都要做，但不是现在全部做。**

你可以这样理解：

* **现在必须做好**：真源、真规则、Stage6 产品化、Stage8/9 闭环、正式基础设施、真实样本、全链入口策略。
* **以后再做好**：leadpack 外发、external export、source/vendor externalized usage、Stage8 live、external release、Stage9 live delivery/payment。 

所以答案不是“都忽略”，也不是“全都一起做”，而是：

**先把内部产品做硬，再做外部解锁。**

## 怎么让 AI 代理勤快，而且不乱做

关键不是让它“更努力”，而是让它**没有偷懒和乱扩 scope 的空间**。

你要这样管它：

**一次只给 1 个任务包。**
不要同时让它做 7 件大事。一次只做一个 packet，例如“真源纵切包”“Stage6 产品化包”“正式基础设施包”。

**每个任务包必须写死 5 件事。**
写清楚目标、允许改哪些目录、不许碰哪些目录、完成标准、必须跑哪些测试。没有这 5 件，AI 很容易一边做一边扩任务。

**要求它先读现状，再动代码。**
顺序固定成：读 control / docs / 相关代码 → 写计划 → 再改代码 → 跑测试 → 回写结果。不要允许它一上来就改。

**结果必须可验收。**
要求它每次都交代：改了哪些文件、加了哪些对象/字段/接口、跑了什么测试、还剩什么阻断项。不要接受“已优化若干逻辑”这种空话。

**禁止自发扩大目标。**
这句最重要：
“只允许完成当前 packet 的闭环，不允许自发扩大目标，不允许提前进入 external/live 方向。”
这样它才会变勤快，因为它不能靠“顺手重构更多东西”来逃避收口。

## 你现在最稳的开发方向

如果按这个报告来，我建议你就走这一条：

先完成当前内部验收包，再新开一个 dedicated packet，做
**Stage1-5 runtime completion + Stage1-6 transport strategy + 正式基础设施替换 + real-sample internal pilot**。
这条路线最像“把产品能力做到位”的正路。 

再往后，下一层才是：
**真源生产 + 真规则工厂 + Stage8 formal carrier persistence + Stage9 真回写闭环**。
这 4 件补完，你的产品会从“内部能跑”变成“内部真能长期干活”。 

最直白地说：

**你现在不是缺“更多功能”，你是缺“把现有能力做实、做稳、做成可运营产品”。**
